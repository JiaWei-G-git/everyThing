# AI 知识库智能体项目需求文档（PRD）

> **版本**：v2.0
> **日期**：2026-04-25
> **状态**：草案（MVP 阶段）
> **作者**：产品负责人
> **基准文档**：《知识库架构与流转流程-2026-04-10》

---

## 1. 项目概述

### 1.1 背景
在日常开发工作中，与 AI（Claude Code、Kimi Code、Cursor、Codex 等）的协作产生了大量高价值信息：Prompt、调试经验、角色定义、工作流等。这些信息目前散落在各 IDE 会话、聊天记录中，无法沉淀为可复用的个人/团队资产。

### 1.2 核心理念
本知识库不是百科式收纳库，而是一个**以 AI 效能闭环为唯一主轴的资产库**。

```
使用 AI → 沉淀知识库 → 提升 AI 技能 → 更高效使用 AI → 赋能团队
```

所有目录、流程、约定都服务于这个正向飞轮。

### 1.3 项目目标
构建一个**基于 Markdown Vault 的个人 AI 知识库智能体**，实现：
- **捕获闭环**：规范化捕获 AI 协作过程，确保原料不流失；
- **资产提炼**：周期性从原料中识别共性模式，提炼为标准资产；
- **文档自动化**：基于会话记录自动生成周报、项目文档；
- **结构自治**：自动维护知识库目录结构，防止僵化；
- **团队赋能**：将个人成熟资产打包为团队共享包。

### 1.4 目标用户
- **主要用户**：前端/后端开发工程师（个人效能提升）；
- **扩展用户**：技术团队（团队知识沉淀与共享）。

## 2. 术语定义

| 术语 | 定义 |
|------|------|
| **Vault** | 基于 Obsidian 的 Markdown 知识库目录，采用双轴驱动模型组织 |
| **原料（Raw Material）** | 未经提炼的原始工作记录，存放于 `00-Inbox` 和 `01-Work` |
| **资产（Asset）** | 经过提炼、标准化、可复用的知识单元，存放于 `10~70` 各分类目录 |
| **会话记录（Session Record）** | 单次 AI 协作的完整记录，包含背景、Prompt、解决思路、优化经验 |
| **MOC** | Map of Content，项目级索引页，聚合该项目下的所有关联笔记 |
| **Skill** | 原子化的操作指令集，可被 AI 工具（Kimi Code / Claude Code）直接调用 |
| **MCP** | Model Context Protocol 服务定义与配置，存放于 `40-MCP-服务/` |

## 第一部分：原料层输入处理

> **核心原则**：原料区杂乱没关系，鼓励快速捕获。不要求格式、不要求完成度。Inbox 只是中转站，外部教程、临时想法先扔 `00-Inbox`，每周归纳一次，空掉它。

---

### F-001 会话记录 Skill（归档 + 路径记忆）

- **优先级**：P0
- **描述**：封装为 Kimi Code / Claude Code Skill，在 AI 协作完成后，自动将当前会话按标准格式归档到 Vault 的指定原料目录。Skill 自身负责归档路径的记忆与复用，无需用户每次指定。

**首次记录流程：**
1. Skill 被触发（如用户说"保存会话"或执行 `/save-session`）；
2. 扫描 Vault 内的原料层目录，提供选项：
   - `00-Inbox-收件箱/`（临时入口）
   - `01-Work-工作记录/<项目A>/会话记录/`
   - `01-Work-工作记录/<项目B>/会话记录/`
   - `01-Work-工作记录/<项目C>/日志/`
   - ……
3. 用户选择目标文件夹；
4. 生成标准化会话记录文件并写入该目录；
5. 在**项目级说明文件**（当前工作目录下的 `AGENTS.md` 或 `CLAUDE.md`）中记录选择，格式：
   ```markdown
   <!-- AI-Vault-Config -->
   - session_archive_path: "01-Work-工作记录/XX项目/会话记录/"
   ```

**二次及以上记录流程：**
1. Skill 被触发；
2. 读取项目级说明文件中的 `session_archive_path` 配置；
3. 如配置存在且路径有效，直接写入该路径，不再询问；
4. 如配置不存在或路径失效，退回首次记录流程（重新询问）。

**输出规范：**
- 文件路径：`{session_archive_path}/会话记录-YYYY-MM-DD-简短主题.md`
- YAML Frontmatter：project, date, topic, source, tags, status
- 正文模板：背景 / Prompt / 解决思路 / 优化经验
- 文件名自动去重，同名追加序号。

**异常处理：**
- 用户未选择路径且项目说明文件无配置 → 默认写入 `00-Inbox-收件箱/未分类/`；
- 配置的归档路径在 Vault 中不存在 → 提示用户是否创建目录，或退回重新选择。

### F-002 多源输入适配

- **优先级**：P1
- **描述**：支持从不同 AI 协作工具导入内容到原料层。
- **适配源**：
  - 手动粘贴（MVP 支持）；
  - Claude Code 会话导出（P1）；
  - Cursor / Kimi Code 聊天记录（P2）。
- **注**：MVP 阶段以手动录入和 Skill 归档为主，全自动捕获需监听 IDE/拦截 API，P2 再探索。

## 第二部分：资产层提炼处理

> **核心原则**：资产区是经过提炼的可复用组件，结构化、带元数据、带使用场景、带效果记录。原料单向流入资产，资产不往回流入原料。每条资产必须记录"从哪条原料提炼出来的"。资产目录初期扁平，数量超过 10 条再考虑子分类。

---

### F-003 资产提炼 Skill（手动触发）

- **优先级**：P0
- **描述**：封装为 Skill，由用户手动触发，读取指定原料并执行资产提炼。用户掌握提炼时机，而非被动等待周期任务。
- **触发方式**：
  - 主动触发：用户执行 `/extract` 或说"提炼这份会话记录"；
  - 被动询问：当 F-001 会话记录 Skill 完成归档后，询问用户"是否提炼本次会话中的资产？"。
- **输入**：用户指定的原料文件路径，或默认读取当前项目近期（如最近 7 天）新增的原料。
- **读取筛选**：
  - 排除已标记 `status: 已提炼` 的原料；
  - 排除空文件或少于 100 字的记录；
  - 优先处理带有 `提炼候选` 标签的记录。

---

### F-004 LLM 资产识别与提取

- **优先级**：P0
- **描述**：调用大模型分析原料内容，识别并提取 6 类可复用资产。
- **资产类别与目标目录**：
  1. **优质 Prompt** → `10-Prompts/`
  2. **角色定义与交互流程** → `20-Agents/`
  3. **原子能力模块（Skill）** → `30-Skills/`
  4. **MCP 服务定义** → `40-MCP/`
  5. **多步骤工作流** → `50-Workflows/`
  6. **工具使用经验与教程** → `60-Tutorials/`
- **输出要求**：
  - 每条资产须标注 `source`（来源原料文件路径）、`extracted_date`、`applicable_scenario`；
  - 输出为草稿格式，不直接写入资产目录，先进入待提炼区。

---

### F-005 人工确认与待提炼区管理

- **优先级**：P0
- **描述**：MVP 阶段以**人工审阅**为核心把关环节。LLM 提取的资产先生成到 `00-Inbox-收件箱/待提炼/`（或独立审核区），由用户逐条审阅后决定是否进入资产区。
- **状态流转**：
  - `draft`（草稿）→ 用户审阅 → `approved`（确认）→ 自动迁移至目标资产目录；
  - `draft` → 用户驳回 → `rejected` → 保留在待提炼区；
  - 已 `approved` 的资产，对应原料更新 `status: 已提炼`（原料本身保留不删除）。
- **待提炼区历史保留**：
  - 用户确认或驳回后，草稿不删除，保留于待提炼区；
  - 追加 `review_date`、`review_result`、`reviewer_note` 等元数据；
  - 方便后续回溯："我之前为什么拒绝这条资产？""这条资产还能补充什么？"
- **远期演进（P2）**：当资产积累到一定数量后，AI 可分析用户的历史提炼习惯（哪些类型的原料经常被提炼、哪些被驳回），实现**高置信度自动提炼**，低置信度仍走人工确认。

---

### F-006 资产去重与版本管理

- **优先级**：P1
- **描述**：在写入资产目录前，检测是否已存在相似资产。
- **去重策略**：基于标题 + 内容摘要的向量相似度，阈值 ≥ 0.85 视为重复，提示合并或更新。
- **版本管理**：同一资产的迭代更新保留历史版本于 `99-Archive-归档/资产版本历史/`。

---

### F-007 自动标签与关联建议

- **优先级**：P1
- **描述**：为提炼出的资产自动打上领域标签，并基于内容相似度推荐关联笔记。
- **标签体系**：
  - **领域层**：frontend, backend, devops, ai-tools, database, testing
  - **动作层**：prompt, debug, refactor, review, deploy, optimize
  - **对象层**：api, ui, database, cicd, documentation
  - **状态层**：draft, verified, official, deprecated, archived
- **关联推荐**：基于向量相似度，在资产文件底部自动追加 `## 关联笔记` 区块，使用 Obsidian 双链语法 `[[文件名]]`。

---

### F-008 周期化资产整理（后续需求）

- **优先级**：P2
- **描述**：当知识库使用频率提高、每周有足够原料产出时，启用周期化自动整理。
- **触发方式**：每周自动扫描 `00-Inbox` 和 `01-Work` 中的新增/更新原料，批量执行 F-003 ~ F-007 流程。
- **前置条件**：用户日常已养成记录习惯，原料产出稳定。

## 第三部分：产出与自治

> **核心原则**：产出层将原料和资产转化为可直接使用的文档（周报、项目概览），自治层确保目录结构不僵化、不失效。每月评估闭环运转状况，识别瓶颈，调整结构。

---

### 3.1 周报与项目文档自动生成

#### F-009 周期化内容聚合

- **优先级**：P0
- **描述**：按项目聚合 `01-Work` 下的会话记录与日志，支持日、周、月三种周期视图。
- **聚合维度（默认）**：
  - 完成事项（Completed）；
  - 进行中事项（In Progress）；
  - 阻塞项与风险（Blockers）；
  - 关键决策与方案（Key Decisions）；
  - 下阶段计划（Next Plan）；
  - 资产沉淀清单。
- **自定义聚合维度**：不同项目可在项目级说明文件（`AGENTS.md` / `CLAUDE.md`）或模板配置中定义额外维度，如"客户需求变更"、"上线风险"等。
- **触发方式**：Skill 手动触发（`/weekly` 或"生成本周周报"）。

#### F-010 结构化文档生成（支持模板自定义）

- **优先级**：P0
- **描述**：基于聚合内容，调用 LLM 生成符合模板的 Markdown 文档。
- **模板体系**：
  - **默认模板**：系统内置通用周报模板，适用于大多数项目；
  - **自定义模板**：存放于 `90-Templates-模板/周报/` 目录下，用户可创建项目专属模板，如：
    ```
    90-Templates-模板/周报/
    ├── 默认周报模板.md
    ├── 敏捷开发-周报模板.md
    ├── 客户项目-周报模板.md
    └── 技术调研-周报模板.md
    ```
- **模板选择规则**：
  1. 优先读取项目级说明文件（`AGENTS.md` / `CLAUDE.md`）中的配置：
     ```markdown
     <!-- AI-Vault-Config -->
     - session_archive_path: "01-Work-工作记录/XX项目/会话记录/"
     - weekly_template: "90-Templates-模板/周报/敏捷开发-周报模板.md"
     ```
  2. 如项目未指定模板，使用默认模板；
  3. 如指定模板不存在，回退到默认模板并提示用户。
- **模板语法**：使用 Jinja2 或类似占位符，如 `{{本周进展}}`、`{{阻塞项}}`、`{{资产沉淀}}` 等。
- **输出路径**：
  - 周报：`01-Work-工作记录/<项目名>/文档/<项目名>-周报-YYYY-MM-DD.md`
  - 项目情况文档：`01-Work-工作记录/<项目名>/文档/项目情况-<项目名>.md`（持续追加，非覆盖）

#### F-011 文档质量自检

- **优先级**：P1
- **描述**：生成后执行基础质量检查。
- **检查项**：
  - 关键字段非空校验（根据模板定义的关键占位符）；
  - 阻塞项是否有对应解决方案；
  - 关联资产链接是否有效（`[[WikiLink]]` 校验）。

---

### 3.2 知识库结构自治

#### F-012 目录健康监控

- **优先级**：P1
- **描述**：系统定期扫描 Vault 目录结构，识别异常状态。
- **监控规则**：
  - 单类资产数量 > 10 条 → 建议并执行子分类拆分（遵循"资产先扁平后分层"原则）；
  - 单项目会话记录 > 30 条 → 建议按月份拆分子目录；
  - 空目录存在 > 7 天 → 自动清理；
  - 失效链接（指向不存在的 `[[WikiLink]]`）→ 标记待修复。

#### F-013 自动子分类建议

- **优先级**：P1
- **描述**：当资产区单类数量超过阈值时，基于现有资产的标签/内容聚类，生成子分类方案。
- **示例**：`10-Prompts-提示词/` 达到 11 条 → 建议拆分为 `10-Prompts-提示词/前端/`、`10-Prompts-提示词/后端/` 等。

#### F-014 失效内容归档

- **优先级**：P1
- **描述**：定期（每月）扫描全库，将长期未更新或标记为 `deprecated` 的内容移至 `99-Archive-归档/`。
- **判定条件**：
  - 最后修改时间 > 90 天且未被任何 MOC 引用；
  - 文件头部 YAML 包含 `status: deprecated`。

#### F-015 MOC 索引自动维护

- **优先级**：P1
- **描述**：每个项目目录自动维护 `_index.md`（MOC 页），聚合该项目下的所有会话记录、日志、文档、周报、关联资产。
- **更新触发**：该项目下新增/删除/移动文件时自动重建。
- **内容要求**：
  - 按文件类型分组（会话记录 / 日志 / 文档 / 资产）；
  - 使用 Obsidian 双链语法 `[[文件名]]`；
  - 顶部包含项目元数据（创建时间、负责人、状态）。

---

### 3.3 团队资产打包分发

#### F-016 资产成熟度评估

- **优先级**：P2
- **描述**：为资产区内容打上成熟度标签，作为打包筛选依据。
- **成熟度等级**：
  - `draft`：草稿，仅个人使用；
  - `verified`：已验证，可团队共享；
  - `official`：官方标准，强制推广。

#### F-017 团队共享包生成

- **优先级**：P2
- **描述**：按团队需求筛选成熟资产，打包并生成上手指南。
- **输出目录**：`70-Sharing-团队共享/<包名>/`
- **包内结构**：
  ```
  70-Sharing-团队共享/后端开发规范包/
  ├── _index.md          # 上手指南
  ├── prompts/           # 精选 Prompts
  ├── agents/            # 角色定义
  ├── skills/            # 可导入 Skill
  ├── workflows/         # 标准工作流
  └── tutorials/         # 配套教程
  ```

#### F-018 导出格式兼容

- **优先级**：P2
- **描述**：支持将共享包导出为多种格式，适配不同团队环境。
- **导出格式**：
  - Markdown 压缩包（通用）；
  - Kimi Code Skill 格式（JSON + Markdown，P2）；
  - Claude Code 自定义指令格式（P3）。

## 第四部分：集成需求

> **核心设计原则**：知识库 Vault 是所有 AI 工具的**单一事实来源**。Skill 在 Vault 中维护一份，多工具共享，版本一致性由 Git 管理。

---

### 4.1 统一 Skill 源（核心架构设计）

- **Skill 仓库**：Vault 的 `30-Skills-技能/` 目录作为所有 AI 工具的共享技能仓库。
- **共享机制**：各 AI 工具不再维护独立的 skill 目录副本，而是通过 **symlink（符号链接）** 或配置映射，直接指向 Vault 内的 skill 目录。
  - **Windows**：
    ```powershell
    mklink /D "C:\Users\<User>\.kimi\skills\我的技能" "D:\我的知识库\30-Skills-技能\我的技能"
    mklink /D "C:\Users\<User>\.claude\skills\我的技能" "D:\我的知识库\30-Skills-技能\我的技能"
    ```
  - **macOS/Linux**：
    ```bash
    ln -s ~/我的知识库/30-Skills-技能/我的技能 ~/.kimi/skills/我的技能
    ln -s ~/我的知识库/30-Skills-技能/我的技能 ~/.claude/skills/我的技能
    ```
- **优势**：
  - 一份 skill，多工具共用；
  - 在 Vault 中编辑 skill，所有工具实时生效；
  - 版本一致性由 Vault 的 Git 管理，无需多份副本分别同步。

### 4.2 与 Kimi Code 集成

- **Skill 共享**：Kimi Code 的 skills 目录通过 symlink 引用 Vault 的 `30-Skills-技能/` 下的子目录；
- **原子能力 Skill**：在 Vault 内维护以下核心 Skill，供 Kimi Code 调用：
  - `保存会话`（F-001）：归档当前会话到原料层；
  - `提炼资产`（F-003）：从指定原料提取可复用资产；
  - `生成周报`（F-009~F-011）：聚合并生成项目周报。
- **工作模式**：
  - 显式触发：用户执行 `/save-session`、`/extract`、`/weekly`；
  - 被动询问：会话结束时询问"是否保存本次会话记录？"。
- **当前限制**：Kimi Code 不具备自主 Agent 循环能力，所有文件操作由 Skill 单次调用完成。

### 4.6 VS Code 插件（P2）

- **定位**：知识库的统一管理界面，降低 CLI 和命令记忆成本。
- **核心功能**：
  - **一键归档**：将当前 VS Code 工作区上下文保存为会话记录到 Vault；
  - **资产提炼面板**：浏览原料文件，一键触发 LLM 提炼；
  - **待审区看板**：可视化 `待提炼/` 中的草稿，支持 approve / reject / 编辑；

  - **目录健康面板**：显示目录异常（空目录、过期文件、失效链接）；
  - **Skill 编辑器**：在 VS Code 中直接编辑 `30-Skills-技能/` 下的 skill，保存后所有工具即时生效。
