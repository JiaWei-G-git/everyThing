# 四 Skill 头脑风暴分析

> 分析日期: 2026-04-25
> 范围: session-recorder / asset-extractor / weekly-generator / pattern-miner
> 目标: 找出当前问题 + 可提升方向

---

## 一、按 Skill 逐条分析

### 1. session-recorder（会话记录保存）

#### 🔴 现有问题

| # | 问题 | 影响 | 严重程度 |
|---|------|------|:-------:|
| 1.1 | **信息提取完全依赖 AI 推断** | 长会话容易遗漏关键信息；不同 AI 提取结果不一致 | 高 |
| 1.2 | **有效提示词判断标准偏主观** | "实质性任务" vs "追问/质疑" 的边界模糊，AI 判断可能不一致 | 中 |
| 1.3 | **模板字段无校验** | 如果 AI 遗漏字段，生成的 Markdown 缺少 YAML 项或正文区块 | 中 |
| 1.4 | **多项目切换时路径记忆冲突** | AGENTS.md 是项目级配置，但用户可能在多个项目共用同一个 Vault | 中 |
| 1.5 | **缺少技术指标记录** | 不记录 Token 消耗、会话时长、AI 工具版本，无法做效能分析 | 低 |
| 1.6 | **occurrence / related_sessions 字段注释化** | 设计文档中有，但模板中注释掉了，功能未实现 | 低 |

#### 🟢 提升方向

1. **结构化提取增强**：在 SKILL.md 中增加"提取 checklist"，要求 AI 必须逐项确认 6 个 YAML 字段 + 4 个正文区块是否都已填充
2. **标签自动推荐**：保存时让 AI 基于内容自动推荐 3-5 个 tags，用户可一键确认或修改
3. **相似会话检测**：保存前扫描 `01-Work/` 下近 30 天记录，如果 topic 相似度 ≥ 0.8，提示"已有 3 条同类记录，建议后续用 pattern-miner 提炼"
4. **项目路径智能推断**：除了 AGENTS.md，再支持从 Git 远程 URL 推断项目名，自动拼接路径

### 2. asset-extractor（资产提炼）

#### 🔴 现有问题

| # | 问题 | 影响 | 严重程度 |
|---|------|------|:-------:|
| 2.1 | **LLM 调用成本高** | 每条原料都要调 LLM，100 条原料 = 100 次调用 | 高 |
| 2.2 | **JSON 输出不稳定** | LLM 经常在 JSON 外加解释文字，导致解析失败 | 高 |
| 2.3 | **单条原料伪通用风险** | 虽然有警告，但缺乏硬性机制阻止从单次会话提炼 Skill | 中 |
| 2.4 | **没有提取效果跟踪** | 提取出的资产后续是否被采用、效果如何，完全无记录 | 中 |
| 2.5 | **空产出无反馈** | 返回 `[]` 时用户不知道是因为原料质量差还是确实没资产 | 低 |

#### 🟢 提升方向

1. **预筛选层**：提取前先用规则过滤——排除 < 200 字、不含任何代码/指令/步骤描述的原料，只对通过筛选的调用 LLM
2. **输出格式容错增强**：除了 JSON，再支持 Markdown 列表格式作为 fallback；或改用"让 LLM 直接输出标准 Markdown 资产草稿"而非 JSON
3. **资产效果闭环**：在 `待提炼/` 的草稿模板中增加 `adopted: false` 字段；当用户迁移到资产区时，asset-extractor 读取并更新为 `adopted: true`，后续可统计"提取命中率"
4. **批量异步模式**：扫描出 N 条原料后，分批（每批 5 条）并行调用 LLM，最后汇总结果

### 3. weekly-generator（周报生成）

#### 🔴 现有问题

| # | 问题 | 影响 | 严重程度 |
|---|------|------|:-------:|
| 3.1 | **时间范围计算歧义** | "最近 7 天"基于文件修改时间还是 YAML date？补录文件会漏或重复 | 高 |
| 3.2 | **内容聚合无去重** | 同一事项在多个会话中重复出现，周报中会重复列出 | 中 |
| 3.3 | **项目情况文档无限追加** | 没有归档/拆分机制，文件会越来越长 | 中 |
| 3.4 | **模板匹配依赖文件名** | `90-Templates/周报/<项目类型>-周报模板.md` 的"项目类型"从哪来？未定义 | 中 |
| 3.5 | **质量检查规则弱** | 6 条检查规则中，"资产链接有效性"需要文件系统遍历，AI 不一定能做到 | 低 |

#### 🟢 提升方向

1. **统一时间基准**：明确规定按 YAML `date` 字段筛选，不按文件修改时间
2. **智能去重**：聚合时调用 LLM 判断两条记录是否描述同一事项，合并为一条并标注"涉及 X 条会话"
3. **项目情况文档拆分**：当文件 > 5000 字时，自动归档为 `项目情况-<项目名>-YYYY-MM.md`，新建当月文档
4. **模板匹配增强**：在 AGENTS.md 中增加 `project_type: 敏捷开发`，weekly-generator 直接读取，不再猜测
5. **对比视图**：生成周报时，自动与上周周报对比，高亮"新增进展""已解决阻塞""持续推进"

---

### 4. pattern-miner（模式挖掘）

#### 🔴 现有问题

| # | 问题 | 影响 | 严重程度 |
|---|------|------|:-------:|
| 4.1 | **聚类规则太粗糙** | 仅靠 topic 关键词 + tags 重叠，容易把"SQL 慢查询"和"SQL 索引优化"误聚为一类 | 高 |
| 4.2 | **硬性门槛 ≥3 次** | 低频高价值问题（如"数据库迁移"）可能几个月才 2 次，被过滤掉 | 中 |
| 4.3 | **每次全量扫描** | 不能增量，Vault 越大扫描越慢 | 中 |
| 4.4 | **标记机制无质量反馈** | `pattern_extracted: true` 只表示"提取过"，不表示"提取得好" | 中 |
| 4.5 | **跨案例抽象质量不可控** | 全靠 LLM 归纳，差异大的记录可能强行统一或放弃 | 中 |

#### 🟢 提升方向

1. **语义聚类增强**：对 topic 做 LLM 语义相似度判断（不只是关键词匹配），相似度 ≥ 0.75 才聚为一类
2. **双阈值机制**：硬性门槛保持 ≥3 次，但增加"软提示"——2 次且涉及关键决策/阻塞项的主题也灰色展示
3. **增量扫描**：记录上次盘点时间戳，下次只扫描新增/修改的记录；已标记 `pattern_extracted: true` 的记录不重复扫描
4. **抽象质量自检**：跨案例抽象后，让 LLM 自检——"如果删除其中任意一条记录，这个 Skill 是否仍然成立？"如果不成立，说明过度归纳
5. **模式演化追踪**：同一主题多次抽象后，保留历史版本在 `99-Archive/模式演化/`，可以看到 Skill 是如何逐步完善的

## 二、系统性问题（跨 Skill）

### 🔴 架构层

| # | 问题 | 说明 |
|---|------|------|
| S1 | **缺少统一配置中心** | AGENTS.md 只存了 `session_archive_path`，其他配置（默认时间范围、LLM 温度、去重阈值）散落在各 SKILL.md 中 |
| S2 | **Skill 之间零联动** | session-recorder 保存后不能自动触发 pattern-miner 增量扫描；weekly-generator 和 pattern-miner 各自独立扫描 `01-Work/`，重复 I/O |
| S3 | **无反馈闭环** | 所有 Skill 的产出（会话记录、提取的草稿、周报）都没有"后续使用效果"的跟踪数据 |
| S4 | **平台绑定过深** | Junction 是 Windows 特有；PowerShell 命令在其他平台不通用；各 AI 工具 Skill 系统差异大 |

### 🟢 系统性提升方向

1. **扩展 AGENTS.md 为统一配置**:
   ```markdown
   <!-- AI-Vault-Config -->
   - session_archive_path: "01-Work-工作记录/XX项目/会话记录"
   - weekly_template: "90-Templates/周报/敏捷开发-周报模板.md"
   - project_type: "敏捷开发"
   - pattern_miner_threshold: 3
   - asset_extractor_batch_size: 5
   ```

2. **Skill 事件轻量联动**：在 session-recorder 保存完成后，如果检测到"本月同类主题已出现 2 次"，主动提示用户"已累计 2 条同类记录，达到 pattern-miner 门槛的 2/3"

3. **产出效果跟踪**：在 `90-Templates/` 下增加 `outcome_tracker.md`，记录每个 Skill 产出的"后续命运"（被确认/被驳回/被修改/被遗忘）

4. **平台抽象层**：在 `bootstrap/07-setup.md` 中增加 Mac/Linux 的符号链接命令，让 AI 根据操作系统自动选择

## 三、优先级排序

### P0（Hackathon 前必须修）
- 1.1 信息提取依赖 AI 推断 → 增加提取 checklist
- 2.2 JSON 输出不稳定 → 增加 Markdown fallback 格式
- 3.1 时间范围计算歧义 → 明确按 YAML date 筛选
- 4.1 聚类规则太粗糙 → topic 增加语义相似度判断

### P1（1 周内）
- 1.3 模板字段无校验 → YAML Frontmatter 必填校验
- 2.1 LLM 调用成本高 → 预筛选层
- 3.2 内容聚合无去重 → 智能去重
- 4.3 每次全量扫描 → 增量扫描

### P2（后续迭代）
- S1 统一配置中心
- S3 反馈闭环
- 1.5 技术指标记录
- 4.5 模式演化追踪
