# 知识库架构与流转流程

> 本文档记录 jw-G-obsidian 的整体设计思路、目录结构、流转规则、功能规划和命名约定,作为后续维护的权威参考。
> 重构日期:2026-04-10 | 规划更新:2026-04-25

---

## 一、核心理念

这个 vault 不是百科式收纳库,而是一个**以 AI 效能闭环为唯一主轴的资产库**。

```
使用 AI → 沉淀知识库 → 提升 AI 技能 → 更高效使用 AI → 赋能团队
```

所有目录、流程、约定都服务于这个正向飞轮。

---

## 二、双轴驱动模型

vault 的全部内容分为两大轴:

```
         原料层(输入)                   资产层(产出)
  ┌─────────────────────┐        ┌──────────────────────┐
  │ 00-Inbox   快速捕获  │  ───▶  │ 10-Prompts  提示词    │
  │ 01-Work    工作实践  │  ───▶  │ 20-Agents   代理      │
  │                     │        │ 30-Skills   技能      │
  │ (你往里扔东西)       │        │ 40-MCP      服务      │
  │                     │        │ 50-Workflows 工作流   │
  │                     │        │ 60-Tutorials 教程     │
  │                     │        │ 70-Sharing  团队共享  │
  └─────────────────────┘        └──────────────────────┘
                                  (AI 帮我从原料提炼出来)
```

- **原料区**:杂乱没关系,鼓励快速捕获。不要求格式、不要求完成度。
- **资产区**:经过提炼的可复用组件,结构化、带元数据、带使用场景、带效果记录。
- **流动方向**:原料单向流入资产,资产不往回流入原料。

## 四、流转规则

```
                       ┌─── 提炼 Prompt  → 10-Prompts
  00-Inbox ──┐         ├─── 提炼 Agent   → 20-Agents
  01-Work  ──┼─ AI ──▶ ├─── 提炼 Skill   → 30-Skills
             │         ├─── 提炼 MCP     → 40-MCP
  (外部教程) ─┘         ├─── 编排 Flow    → 50-Workflows
                       └─── 整理 Tutorial → 60-Tutorials

  成熟资产 ──▶ 70-Sharing(团队分发)
  失效内容 ──▶ 99-Archive
```

### 五条核心规则

1. **Inbox 只是中转站**:外部教程、临时想法先扔 00-Inbox,每周归纳一次,空掉它。
2. **Work 按项目聚合**:每个项目一个子目录,内部分 `会话记录/日志/文档`。避免把多个项目的内容混在一起。
3. **资产来源可追溯**:每条资产笔记应该记录"从哪条原料提炼出来的",方便回溯和验证。
4. **原料不删除**:提炼后原料保留在 Work/Inbox,只打"已提炼"标签。原料是资产的根据地。
5. **资产先扁平后分层**:10/20/30/40/50/60 等资产目录初期扁平,数量超过 10 条再考虑子分类。

### 迭代节奏

| 频率       | 动作                                                                           | 目标          |
| ---------- | ------------------------------------------------------------------------------ | ------------- |
| **日常**   | 使用 AI 完成工作后,5 分钟记录关键 Prompt、问题、优化思路到 00-Inbox 或 01-Work | 原料不流失    |
| **每周**   | 整理 00-Inbox,提炼共性模式,更新或新建资产                                      | Inbox 清零    |
| **每月**   | 评估闭环运转状况,识别瓶颈,调整结构                                             | 避免结构僵化  |
| **不定期** | 成熟资产打包进 70-Sharing,分享给团队                                           | 个人→团队跃迁 |

## 五、功能规划

基于上述架构,规划以下功能模块支撑完整闭环运转。

> **设计原则**: MVP 阶段以 Skill 手动触发为主,周期自动整理作为后续需求保留。人工确认是资产进入资产区的核心把关环节。

### 第一部分:原料层输入处理

#### F-001 会话记录 Skill(归档 + 路径记忆)

- **优先级**:P0
- **描述**:封装为 Kimi Code / Claude Code Skill,在 AI 协作完成后,自动将当前会话按标准格式归档到 Vault 的指定原料目录。Skill 自身负责归档路径的记忆与复用。

**首次记录流程:**
1. Skill 被触发(如用户说"保存会话"或执行 `/save-session`);
2. 扫描 Vault 内的原料层目录,提供选项;
3. 用户选择目标文件夹;
4. 生成标准化会话记录文件并写入该目录;
5. 在项目级说明文件(`AGENTS.md` 或 `CLAUDE.md`)中记录选择。

**二次及以上记录流程:**
1. Skill 被触发;
2. 读取项目级说明文件中的 `session_archive_path` 配置;
3. 如配置存在且路径有效,直接写入该路径,不再询问;
4. 如配置不存在或路径失效,退回首次记录流程。

**输出规范:**
- 文件路径:`{session_archive_path}/会话记录-YYYY-MM-DD-简短主题.md`
- YAML Frontmatter:project, date, topic, source, tags, status
- 正文模板:背景 / Prompt / 解决思路 / 优化经验

---

#### F-002 多源输入适配

- **优先级**:P1
- **描述**:支持从不同 AI 协作工具导入内容到原料层。
- **适配源**:手动粘贴(MVP 支持); Claude Code 会话导出(P1); Cursor / Kimi Code 聊天记录(P2)。
- **注**:MVP 阶段以手动录入和 Skill 归档为主,全自动捕获需监听 IDE/拦截 API,P2 再探索。

---

### 第二部分:资产层提炼处理

#### F-003 资产提炼 Skill(手动触发)

- **优先级**:P0
- **描述**:封装为 Skill,由用户手动触发,读取指定原料并执行资产提炼。用户掌握提炼时机。
- **触发方式**:主动触发(用户执行 `/extract`); 被动询问(当 F-001 完成归档后询问"是否提炼本次会话中的资产?")。
- **读取筛选**:排除已标记 `status: 已提炼` 的原料; 排除空文件或少于 100 字的记录; 优先处理带有 `提炼候选` 标签的记录。

---

#### F-004 LLM 资产识别与提取

- **优先级**:P0
- **描述**:调用大模型分析原料内容,识别并提取 6 类可复用资产。
- **资产类别与目标目录**:
  1. 优质 Prompt → `10-Prompts/`
  2. 角色定义与交互流程 → `20-Agents/`
  3. 原子能力模块(Skill) → `30-Skills/`
  4. MCP 服务定义 → `40-MCP/`
  5. 多步骤工作流 → `50-Workflows-工作流/`
  6. 工具使用经验与教程 → `60-Tutorials-教程/`
- **输出要求**:每条资产须标注 `source`(来源原料文件路径)、`extracted_date`、`applicable_scenario`。

---

#### F-005 人工确认与待提炼区管理

- **优先级**:P0
- **描述**:MVP 阶段以**人工审阅**为核心把关环节。LLM 提取的资产先生成到 `00-Inbox-收件箱/待提炼/`,由用户逐条审阅后决定是否进入资产区。
- **状态流转**:
  - `draft`(草稿) → 用户审阅 → `approved`(确认) → 自动迁移至目标资产目录;
  - `draft` → 用户驳回 → `rejected` → 保留在待提炼区;
  - 已 `approved` 的资产,对应原料更新 `status: 已提炼`(原料本身保留不删除)。
- **待提炼区历史保留**:
  - 用户确认或驳回后,草稿不删除,保留于待提炼区;
  - 追加 `review_date`、`review_result`、`reviewer_note` 等元数据;
  - 方便后续回溯:"我之前为什么拒绝这条资产?"
- **远期演进(P2)**:当资产积累到一定数量后,AI 可分析用户的历史提炼习惯,实现**高置信度自动提炼**,低置信度仍走人工确认。

---

#### F-006 资产去重与版本管理

- **优先级**:P1
- **描述**:在写入资产目录前,检测是否已存在相似资产。
- **去重策略**:基于标题 + 内容摘要的向量相似度,阈值 ≥ 0.85 视为重复,提示合并或更新。
- **版本管理**:同一资产的迭代更新保留历史版本于 `99-Archive-归档/资产版本历史/`。

---

#### F-007 自动标签与关联建议

- **优先级**:P1
- **描述**:为提炼出的资产自动打上领域标签,并基于内容相似度推荐关联笔记。
- **标签体系**:
  - **领域层**:frontend, backend, devops, ai-tools, database, testing
  - **动作层**:prompt, debug, refactor, review, deploy, optimize
  - **对象层**:api, ui, database, cicd, documentation
  - **状态层**:draft, verified, official, deprecated, archived
- **关联推荐**:基于向量相似度,在资产文件底部自动追加 `## 关联笔记` 区块,使用 Obsidian 双链语法 `[[文件名]]`。

---

#### F-008 周期化资产整理(后续需求)

- **优先级**:P2
- **描述**:当知识库使用频率提高、每周有足够原料产出时,启用周期化自动整理。
- **触发方式**:每周自动扫描 `00-Inbox` 和 `01-Work` 中的新增/更新原料,批量执行 F-003 ~ F-007 流程。

### 第三部分:产出与自治

#### 3.1 周报与项目文档自动生成

**F-009 周期化内容聚合**
- **优先级**:P0
- 按项目聚合 `01-Work` 下的会话记录与日志,支持日、周、月三种周期视图。
- 聚合维度:完成事项 / 进行中事项 / 阻塞项与风险 / 关键决策与方案 / 下阶段计划 / 资产沉淀清单。
- **自定义聚合维度**:不同项目可在项目级说明文件中定义额外维度。
- **触发方式**:Skill 手动触发(`/weekly`)。

**F-010 结构化文档生成(支持模板自定义)**
- **优先级**:P0
- **模板体系**:
  - 默认模板:系统内置通用周报模板;
  - 自定义模板:存放于 `90-Templates-模板/周报/` 目录下,用户可创建项目专属模板。
- **模板选择规则**:优先读取项目级说明文件中的 `weekly_template` 配置; 未指定则使用默认模板。
- **模板语法**:Jinja2 占位符,如 `{{本周进展}}`、`{{阻塞项}}` 等。
- **输出路径**:
  - 周报:`01-Work-工作记录/<项目名>/文档/<项目名>-周报-YYYY-MM-DD.md`
  - 项目情况文档:`01-Work-工作记录/<项目名>/文档/项目情况-<项目名>.md`(持续追加)

**F-011 文档质量自检**
- **优先级**:P1
- 关键字段非空校验; 阻塞项是否有对应解决方案; 关联资产链接是否有效(`[[WikiLink]]` 校验)。

---

#### 3.2 知识库结构自治

**F-012 目录健康监控**
- **优先级**:P1
- 单类资产数量 > 10 条 → 建议子分类拆分; 单项目会话记录 > 30 条 → 建议按月份拆分; 空目录存在 > 7 天 → 自动清理; 失效链接 → 标记待修复。

**F-013 自动子分类建议**
- **优先级**:P1
- 当资产区单类数量超过阈值时,基于现有资产的标签/内容聚类,生成子分类方案。

**F-014 失效内容归档**
- **优先级**:P1
- 定期(每月)扫描全库,将长期未更新或标记为 `deprecated` 的内容移至 `99-Archive-归档/`。

**F-015 MOC 索引自动维护**
- **优先级**:P1
- 每个项目目录自动维护 `_index.md`(MOC 页),聚合该项目下的所有会话记录、日志、文档、周报、关联资产。

---

#### 3.3 团队资产打包分发

**F-016 资产成熟度评估**
- **优先级**:P2
- `draft`:草稿,仅个人使用; `verified`:已验证,可团队共享; `official`:官方标准,强制推广。

**F-017 团队共享包生成**
- **优先级**:P2
- 按团队需求筛选成熟资产,打包并生成上手指南。输出至 `70-Sharing-团队共享/<包名>/`。

**F-018 导出格式兼容**
- **优先级**:P2
- Markdown 压缩包(通用); Kimi Code Skill 格式(JSON + Markdown,P2); Claude Code 自定义指令格式(P3)。

## 六、集成需求

> **核心设计原则**:知识库 Vault 是所有 AI 工具的**单一事实来源**。Skill 在 Vault 中维护一份,多工具共享,版本一致性由 Git 管理。

### 统一 Skill 源

- **Skill 仓库**:Vault 的 `30-Skills-技能/` 目录作为所有 AI 工具的共享技能仓库。
- **共享机制**:各 AI 工具不再维护独立的 skill 目录副本,而是通过 **symlink(符号链接)** 直接指向 Vault 内的 skill 目录。
  - **Windows**:
    ```powershell
    mklink /D "C:\Users\<User>\.kimi\skills\我的技能" "D:\我的知识库\30-Skills-技能\我的技能"
    mklink /D "C:\Users\<User>\.claude\skills\我的技能" "D:\我的知识库\30-Skills-技能\我的技能"
    ```
  - **macOS/Linux**:
    ```bash
    ln -s ~/我的知识库/30-Skills-技能/我的技能 ~/.kimi/skills/我的技能
    ln -s ~/我的知识库/30-Skills-技能/我的技能 ~/.claude/skills/我的技能
    ```

### 与 Kimi Code 集成

- Skill 共享:Kimi Code 的 skills 目录通过 symlink 引用 Vault 的 `30-Skills/`;
- 原子能力 Skill:`保存会话`(F-001)、`提炼资产`(F-003)、`生成周报`(F-009~F-011);
- 工作模式:显式触发(`/save-session`、`/extract`、`/weekly`)或被动询问。

### 与 Claude Code 集成

- Skill 共享:`.claude/skills/` 通过 symlink 引用 Vault 的 `30-Skills/`;
- 项目级配置:在 `CLAUDE.md` 中配置 Vault 相关路径和模板;
- 触发器模式:Agent 模式下执行 `/save-to-vault` 触发归档。

### 与 Cursor / Codex CLI 集成(P2)

- Skill 共享:通过 symlink 指向 Vault 的 `30-Skills/`;
- 场景:未来各工具成熟后,统一从 Vault 加载 skill。

### VS Code 插件(P2)

- **定位**:知识库的统一管理界面。
- **核心功能**:一键归档、资产提炼面板、待审区看板、周报生成器、目录健康面板、Skill 编辑器。

## 七、命名约定

### 一级目录:`编号-英文` 双标

- 编号间隔 10(如 10/20/30),方便中间插入新类型
- 英文便于国际化分享和命令行操作
- 中文便于日常识别

示例:`10-Prompts/`

### 二级及文件名:纯中文

避免名字过长,保持可读性。

### 会话记录文件名格式

```
会话记录-YYYY-MM-DD-简短主题.md
```

示例:`会话记录-2026-04-25-接口优化.md`

### 其他文件命名

| 类型 | 命名格式 | 示例 |
|------|---------|------|
| 周报 | `<项目名>-周报-YYYY-MM-DD.md` | `电商后台-周报-2026-04-25.md` |
| Prompt 资产 | `<领域>-<具体能力>.md` | `通用-性能优化.md` |
| Agent 资产 | `<角色名>.md` | `后端专家.md` |
