# AI 技能知识库平台 —— 多平台适配规范

## 一、设计原则

1. **内容为王，格式为壳**：技能的核心价值是 Prompt 逻辑和执行步骤，平台适配只是包装
2. **通用版本优先**：先写好通用版本，再派生各平台版本
3. **最小适配**：只在必要时做平台特定适配，不为了适配而适配
4. **单向兼容**：通用版本可以降级到任何平台，但平台特定版本不能反向兼容

## 二、平台概览

| 平台 | 技能载体 | 调用方式 | 能力范围 | 优先级 | 备注 |
|:---|:---|:---|:---|:---:|:---|
| **Claude Code** | `~/.claude/skills/*.md` | `/skill-name` 斜杠命令 | 最丰富，支持 Skill 系统 | P0 | |
| **Cursor** | `.cursorrules` / `.cursor/rules/*.md` | `@` 引用 / 自动注入 | 丰富，支持 rules 系统 | P0 | |
| **通用 Prompt** | 任意文本 | 复制粘贴到任何 AI 工具 | 最通用，任何工具可用 | P0 | |
| **VS Code + Copilot** | `settings.json` / 内联指令 | 斜杠命令 / 自动触发 | 中等，Prompt 长度受限 | P2 | 待补充适配示例 |
| **Windsurf** | Cascade 规则文件 | 自动应用 | 中等 | P1 | |
| **其他工具** | 纯文本 | 复制粘贴 | 基础 | P2 | |

## 三、通用版本规范

通用版本是技能内容的**标准形式**，所有平台版本都从中派生。

### 3.1 文件格式

- 纯 Markdown（CommonMark 标准）
- 使用 YAML frontmatter 存储元数据
- 无平台特定语法

### 3.2 内容结构

```markdown
---
name: "技能名称"
description: "一句话描述"
type: "skill"
version: "1.0.0"
date: "2026-04-29"
parent-agent: "所属智能体"
standalone: true
tags: ["标签1", "标签2"]
---

# 技能：技能名称

## 适用场景
...

## 输入
...

## 输出
...

## 执行步骤
...

## Prompt 模板
...

## 安装说明
...
```

### 3.3 与平台无关的内容规则

以下内容的编写**不依赖任何平台**：
- 适用场景描述
- 输入/输出定义
- 执行步骤逻辑
- Prompt 的核心逻辑（"做什么"）

以下内容的编写**需要考虑平台差异**：
- Prompt 中的格式指令（如"用 Markdown 表格输出"是通用的，"用 XML 标签包裹"可能不通用）
- 安装说明
- 调用方式

## 四、Claude Code 适配

### 4.1 Skill 文件格式

Claude Code 的 Skill 是 Markdown 文件，放在 `~/.claude/skills/` 或项目 `.claude/skills/` 目录下：

```markdown
---
name: Skill 名称（命令调用名，建议用英文小写+连字符）
description: 一句话描述（显示在 Skill 列表中）
type: skill
---

# Skill 名称

当用户调用 /skill-name 时，执行以下任务：

## 任务描述
...

## 执行步骤
1. ...
2. ...

## Prompt
...
```

### 4.2 与通用版本的区别

| 维度 | 通用版本 | Claude Code Skill 版本 |
|:---|:---|:---|
| frontmatter | 完整的元数据 | 简化的 name + description |
| 内容组织 | 完整的知识型文档 | 面向执行的指令型文档 |
| Prompt 位置 | 放在"Prompt 模板"区块 | 直接融入 Skill 描述 |
| 输入处理 | 用户手动准备输入 | Skill 可以引导用户提供输入 |
| 输出处理 | 用户手动整理输出 | Skill 可以要求 Claude 按格式输出 |

### 4.3 Skill 版本的编写规则

1. **name 字段**：使用英文小写 + 连字符，作为 `/command-name` 的命令名
   ```yaml
   name: requirement-decomposition
   ```

2. **描述风格**：使用祈使句或"当...时"句式，说明 Skill 被调用后做什么
   ```markdown
   当用户需要将一个高层需求拆解为可执行的功能点时，帮助用户完成以下步骤：
   ```

3. **融入执行逻辑**：不需要单独的"Prompt 模板"区块，而是将执行逻辑直接写入 Skill 描述

4. **输出格式指令**：明确告诉 Claude 用什么格式输出，方便用户直接使用

5. **frontmatter 可保留字段**：`type: skill`、`version` 和 `date` 可用于版本管理，与 `name`、`description` 一并放在 frontmatter 中

### 4.4 示例：Claude Code Skill 版本

```markdown
---
name: requirement-decomposition
description: 将高层需求拆解为可执行的功能点
type: skill
version: "1.0.0"
date: "2026-04-29"
---

# 需求拆解

当用户调用此技能时，帮助他们将高层需求拆解为具体的功能点。

## 工作流程

1. **理解背景**：询问用户项目背景（目标用户、关键约束、时间预算），如果用户已提供则跳过
2. **接收需求**：请用户提供需要拆解的原始需求描述
3. **识别主题**：分析需求，识别出 3-7 个主要主题/模块
4. **逐主题拆解**：对每个主题，拆解为具体功能点，每个功能点包含：
   - 用户故事（作为...我可以...以便...）
   - 优先级（P0/P1/P2）
   - 依赖关系
   - 估算工时
5. **检查调整**：检查是否有遗漏、优先级是否合理、依赖是否有循环

## 输出格式

用 Markdown 表格输出：

| 编号 | 功能点 | 用户故事 | 优先级 | 依赖 | 工时 |
|:---|:---|:---|:---:|:---|:---:|
| F001 | ... | ... | P0 | 无 | 3d |

## 注意事项

- P0 功能点不应超过总数的 30%
- 避免循环依赖
- 如果用户提供的原始需求存在明显重复，建议先进行去重处理
```

## 五、Cursor 适配

### 5.1 Rules 文件格式

Cursor 支持两种规则文件：

**方式 1：项目级 `.cursorrules`**
```markdown
# 项目规则

## 需求拆解规则

当用户要求拆解需求时，执行以下步骤：
1. ...
2. ...
```

**方式 2：文件级 `.cursor/rules/*.md`**
```markdown
---
description: 将高层需求拆解为可执行的功能点
glob: "*requirement*"
---

当用户在编辑 Markdown 文件并要求拆解需求时...
```

### 5.2 与通用版本的区别

| 维度 | 通用版本 | Cursor Rules 版本 |
|:---|:---|:---|
| 触发方式 | 用户主动调用 | 自动触发（基于文件类型/内容）或 `@` 引用 |
| 上下文 | 用户提供 | Cursor 自动读取当前文件内容 |
| 持续性 | 单次调用 | 持续生效（项目级 rules） |

### 5.3 Rules 版本的编写规则

1. **使用 Markdown 格式**：与通用版本结构一致
2. **添加 `description` 和 `glob`**（如果使用 `.cursor/rules/*.md`）
   ```yaml
   ---
   description: 需求拆解助手
   glob: "*requirement*"
   ---
   ```
3. **强调自动触发条件**：说明在什么情况下 Cursor 会自动应用此规则
4. **融入当前上下文**：利用 Cursor 能读取当前文件的优势，让规则更贴合实际

### 5.4 示例：Cursor Rules 版本

```markdown
---
description: 需求拆解助手：将高层需求拆解为功能点
glob: "*requirement*"
---

# 需求拆解

当用户在 Cursor 中编辑文档并要求"拆解需求""拆分功能""细化需求"时，执行以下流程。

## 自动触发条件

用户输入包含以下关键词时触发：
- "拆解需求"
- "拆分功能点"
- "细化这个需求"
- "把需求拆一下"

## 执行步骤

1. **读取当前上下文**：查看用户当前编辑的文件内容，提取已有的需求描述
2. **询问补充**：如果信息不足，询问项目背景和约束
3. **执行拆解**：按以下格式输出功能点清单
   - 编号（F001, F002...）
   - 功能点名称
   - 用户故事
   - 优先级（P0/P1/P2）
   - 依赖关系
   - 估算工时
4. **插入文档**：将结果插入到用户当前光标位置，或输出到新文件中

## 输出格式

```markdown
## 功能点清单

| 编号 | 功能点 | 用户故事 | 优先级 | 依赖 | 工时 |
|:---|:---|:---|:---:|:---|:---:|
| F001 | ... | ... | P0 | 无 | 3d |
```

## 约束

- P0 不超过总数 30%
- 工时总和不超过项目约束
- 避免循环依赖
```

## 六、通用 Prompt 适配

### 6.1 格式

通用 Prompt 版本是**最简形式**，适用于任何 AI 工具：

```markdown
# {技能名称}

## 任务
{一句话说明要做什么}

## 输入
{说明需要用户提供什么}

## 执行步骤
{分步骤说明}

## 输出要求
{说明期望的输出格式}

---

# 开始使用

复制以上内容，替换输入部分，发送给 AI。
```

### 6.2 与通用版本的区别

通用 Prompt 版本是通用版本的**精简提取版**：
- 去掉 frontmatter（通用 AI 工具不支持）
- 去掉安装说明（不需要安装）
- 去掉关联内容（不需要导航）
- 保留：任务、输入、步骤、输出要求、Prompt 模板

### 6.3 示例：通用 Prompt 版本

```markdown
# 需求拆解

## 任务
将高层需求拆解为可执行的功能点清单。

## 输入
请提供：
1. 项目背景（目标、用户、约束）
2. 需要拆解的原始需求描述

## 执行步骤
1. 识别需求主题（3-7 个模块）
2. 逐主题拆解成功能点
3. 为每个功能点标注优先级和依赖
4. 检查并调整

## 输出要求
用 Markdown 表格输出：

| 编号 | 功能点 | 用户故事 | 优先级 | 依赖 | 工时 |
|:---|:---|:---|:---:|:---|:---:|

约束：
- P0 不超过 30%
- 避免循环依赖

---

# 开始使用

将以上输入部分替换为你的实际内容，发送给 AI 即可。
```

## 七、平台版本的管理策略

### 7.1 存储位置

**第一阶段（手工维护）**：

所有版本都存储在技能文件的"安装说明"区块内，不单独拆分文件：

```markdown
## 安装说明

### Claude Code 版本
```
...Skill 格式内容...
```
复制以上内容保存为 `.claude/skills/xxx.md`

### Cursor 版本
```
...Rules 格式内容...
```
复制以上内容保存为 `.cursorrules` 或 `.cursor/rules/xxx.md`

### 通用版本
```
...Prompt 格式内容...
```
直接复制使用
```

> **备选方案**：如一个技能文件内的安装说明区块（含多平台版本）超过 50 行，建议在 `skills/` 下创建 `{技能名}/` 子文件夹，内分 `README.md`（通用版）、`claude-code.md`、`cursor.md`、`generic-prompt.md`，避免单个文件过大。

**后续阶段（工具辅助，视实施路线图决策）**：

如果后续开发自动转换工具，可以拆分为：

```
skills/
├── 需求拆解/
│   ├── README.md              # 通用版本（主文件）
│   ├── claude-code.md         # Claude Code Skill 版本
│   ├── cursor.md              # Cursor Rules 版本
│   └── generic-prompt.md      # 通用 Prompt 版本
```

### 7.2 版本同步

当更新技能内容时：

1. 先修改通用版本（核心逻辑）
2. 同步更新各平台版本中的对应内容
3. 如果某平台版本有平台特定的优化，保留该优化部分
4. 更新所有版本的 version 和 date

### 7.3 维护优先级

| 优先级 | 平台 | 维护策略 |
|:---:|:---|:---|
| P0 | Claude Code | 完整适配，提供 Skill 格式 |
| P0 | Cursor | 完整适配，提供 Rules 格式 |
| P0 | 通用 Prompt | 必须提供，保证任何工具可用 |
| P1 | Windsurf | 提供 Cascade 规则文件版本 |
| P2 | VS Code + Copilot | 提供简化的内联指令版本（待补充适配示例） |
| P2 | 其他工具 | 依赖通用 Prompt 版本 |

## 八、平台特定语法对照

| 功能 | 通用 Markdown | Claude Code Skill | Cursor Rules |
|:---|:---|:---|:---|
| 元数据 | YAML frontmatter | YAML frontmatter（简化） | YAML frontmatter（含 glob） |
| 命令名 | 不适用 | `name:` 字段 | 不适用 |
| 触发条件 | 用户手动调用 | `/command-name` | 自动触发（glob）或 `@` |
| 引用其他文件 | 相对链接 | 不适用 | `@` 引用 |
| 代码块 | ```language | ```language | ```language |
| 表格 | Markdown 表格 | Markdown 表格 | Markdown 表格 |

## 九、平台选择决策树

当团队需要决定优先适配哪个平台版本时，参考以下决策逻辑：

- **团队主要用 Claude Code** → 优先 Skill 版
- **团队主要用 Cursor** → 优先 Rules 版
- **混合使用或多工具** → 优先通用 Prompt 版

## 十、转换检查清单

将通用版本转换为平台版本时，检查以下项目：

- [ ] 核心 Prompt 逻辑保持一致
- [ ] 输入/输出格式说明一致
- [ ] 执行步骤一致
- [ ] 平台特定的调用方式正确（`/command` 或 `@file`）
- [ ] 平台特定的元数据正确（frontmatter）
- [ ] 版本号与通用版本一致
- [ ] 无平台不兼容的语法
