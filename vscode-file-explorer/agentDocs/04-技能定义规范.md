# AI 技能知识库平台 —— 技能定义规范

## 一、文件定位

技能定义文件是知识库的**核心资产**，包含：
- 一个具体的 AI 能力单元
- 可直接复制使用的 Prompt 模板
- 清晰的执行步骤和输入/输出定义
- 独立安装说明

技能文件必须是**自包含**的：用户只打开这一个文件，就能理解这个技能是什么、什么时候用、怎么用。

## 二、文件命名

```
{动词}+{名词}.md

示例：
- 需求去重.md
- 需求拆解.md
- 根据需求生成原型.md
- React组件开发.md
- API鉴权设计.md
```

**命名规则**：
- 使用"动词+名词"结构，说明"做什么"
- 避免使用"的"字（如"需求的去重" → "需求去重"）
- 避免使用"技能"后缀（已在 skills/ 目录下）
- 全局唯一，不同集群下也不重复

## 三、文件位置

```
clusters/{集群文件夹}/skills/

示例：
clusters/01-需求工程/skills/需求拆解.md
clusters/02-设计工程/skills/根据需求生成原型.md
```

## 四、Frontmatter 规范

```yaml
---
name: "需求拆解"                        # 技能显示名称
description: "将高层需求分解为可实现的功能点"  # 一句话描述
type: "skill"                          # 固定值：skill
version: "1.0.0"                       # 语义化版本
date: "2026-04-29"                   # 最后更新日期
parent-agent: "需求分析智能体"           # 所属智能体
standalone: true                       # 是否可独立使用：true/false
tags:                                  # 标签（用于检索）
  - "需求"
  - "分析"
  - "拆解"
input: "高层需求描述或用户故事列表"      # 输入描述
output: "功能点清单，含优先级和依赖关系"  # 输出描述
estimated-time: "15-30分钟"            # 预计执行时间（可选）
---
```

**字段说明**：

| 字段 | 必填 | 说明 |
|:---|:---:|:---|
| name | ✅ | 技能名称，与文件名一致 |
| description | ✅ | 一句话描述技能用途 |
| type | ✅ | 固定值 `skill` |
| version | ✅ | 语义化版本号 |
| date | ✅ | 最后更新日期 |
| parent-agent | ✅ | 所属智能体名称 |
| standalone | ✅ | `true` = 可独立使用；`false` = 依赖智能体上下文 |
| tags | ❌ | 标签数组，辅助检索 |
| input | ❌ | 输入内容的描述 |
| output | ❌ | 输出内容的描述 |
| estimated-time | ❌ | 预计完成时间 |

## 五、内容模块规范

技能文件必须包含以下模块，按此顺序排列：

### 模块 1：适用场景

**目的**：让用户判断当前任务是否适合使用这个技能。

**内容要求**：
- 2-5 个具体的适用场景（用"当...时"句式）
- 1-2 个不适用场景（帮助用户排除误用）

**示例**：

```markdown
## 适用场景

✅ **适合使用**：
- 当收到来自客户的一整段需求描述，需要拆成可执行的功能点时
- 当用户故事（User Story）颗粒度过大，需要细化时
- 当需要评估每个功能点的实现优先级和依赖关系时

❌ **不适合使用**：
- 原始需求还存在大量重复和矛盾时（先用"需求去重"技能）
- 已经有明确的功能点清单，只需要排列优先级时
```

### 模块 2：输入

**目的**：明确告诉用户需要准备什么材料。

**内容要求**：
- 输入内容的类型和格式
- 输入示例（如果格式复杂）
- 必须包含的信息项
- 可选信息项

**示例**：

```markdown
## 输入

### 必需输入
- 原始需求描述（文本形式，可以是客户邮件、会议纪要、口头转述）
- 项目背景信息（项目目标、目标用户、关键约束）

### 可选输入
- 已有功能清单（避免重复拆解）
- 技术栈信息（影响拆分粒度）

### 输入示例

```
项目：在线书店系统
背景：为中小书店提供线上销售平台
约束：3个月内上线，预算50万

原始需求：
1. 用户可以浏览图书分类和搜索图书
2. 用户可以下单购买，支持多种支付方式
3. 商家可以管理库存和订单
4. 系统需要支持促销活动
```
```

### 模块 3：输出

**目的**：明确告诉用户执行后会得到什么。

**内容要求**：
- 输出内容的类型和格式
- 输出示例
- 输出质量标准和验收要点

**示例**：

```markdown
## 输出

### 输出格式
功能点清单（Markdown 表格），包含：
- 功能点编号
- 功能点名称
- 用户故事描述
- 优先级（P0/P1/P2）
- 依赖关系
- 估算工时

### 输出示例

| 编号 | 功能点 | 用户故事 | 优先级 | 依赖 | 工时 |
|:---|:---|:---|:---:|:---|:---:|
| F001 | 图书浏览 | 作为用户，我可以按分类浏览图书 | P0 | 无 | 3d |
| F002 | 图书搜索 | 作为用户，我可以通过关键词搜索图书 | P0 | F001 | 2d |
| ... | ... | ... | ... | ... | ... |

### 验收要点
- [ ] 每个功能点有明确的用户视角描述
- [ ] 优先级排列合理，P0 数量不超过总数的 30%
- [ ] 依赖关系无循环依赖
```

### 模块 4：执行步骤

**目的**：提供可复现的执行流程。

**内容要求**：
- 分步骤说明，每步有明确的动作和预期产出
- 步骤编号连续
- 关键步骤有注意事项或提示
- 如果步骤有分支（if/then），明确标注

**示例**：

```markdown
## 执行步骤

### 步骤 1：理解项目背景
阅读输入中的项目背景信息，确认：
- 项目目标是什么
- 目标用户是谁
- 关键约束有哪些

### 步骤 2：识别需求主题
从原始需求中提取主要主题/模块：
- 把相似的需求归为一类
- 每个主题给出一个简洁的命名

> **提示**：主题数量建议在 3-7 个之间，过多说明需求范围太大，过少说明拆分不够。

### 步骤 3：逐主题拆解功能点
对每个主题，拆解为具体的功能点：
- 每个功能点遵循"作为...我可以...以便..."的用户故事格式
- 评估每个功能点的优先级
- 标注功能点之间的依赖关系

### 步骤 4：检查与调整
- 检查是否有遗漏的需求
- 检查优先级是否合理
- 检查依赖关系是否有循环
- 根据工时估算调整 scope（如总工时超出约束，需要降级部分 P0 为 P1）
```

### 模块 5：Prompt 模板

**目的**：提供可直接复制到 AI 工具中使用的完整 Prompt。

**内容要求**：
- 一个完整的、可直接使用的 Prompt 模板
- 使用 `{占位符}` 标记需要用户替换的内容
- 如果 Prompt 较长，标注各部分的作用
- 提供使用示例（占位符替换后的完整 Prompt）

**示例**：

```markdown
## Prompt 模板

### 模板（复制此版本，替换占位符）

```
你是一名需求分析师，擅长将高层需求拆解为可执行的功能点。

{粘贴项目背景信息}

## 原始需求
{粘贴原始需求列表}

## 任务

1. 每个功能点用用户故事格式描述（作为...我可以...以便...）
2. 标注每个功能点的优先级（P0=必须做，P1=应该做，P2=可以做）
3. 标注功能点之间的依赖关系
4. 给出每个功能点的估算工时

## 输出格式
用 Markdown 表格输出，列：编号 | 功能点 | 用户故事 | 优先级 | 依赖 | 工时

## 六、技能粒度控制

### 6.1 粒度判断标准

| 维度 | 太细（不好） | 适中（好） | 太粗（不好） |
|:---|:---|:---|:---|
| **时间** | < 5 分钟能完成 | 15-60 分钟的工作量 | > 2 小时的工作量 |
| **输出** | 只是一个简单句式 | 一份完整的可交付物 | 需要拆成多个独立交付物 |
| **复用** | 几乎不会单独使用 | 有明确的独立使用场景 | 包含太多不相关的子能力 |
| **Prompt** | ≤ 10 行（偏细，需评估是否值得独立成技能） | 10-60 行 | ≥ 80 行（偏粗，考虑拆分） |

### 6.2 合并与拆分原则

**应该合并的情况**：
- 两个技能几乎总是在同一个场景中使用
- 两个技能的 Prompt 高度重叠
- 用户反馈"这两个技能分不清该用哪个"

**应该拆分的情况**：
- 一个技能包含多个完全不相关的任务
- 用户只想执行其中一部分，但不得不跑完全部
- 不同子任务需要不同的输入和输出格式

## 七、禁止事项

1. **禁止技能文件依赖其他技能文件的内容**：Prompt 模板中不得要求用户"先执行 XX 技能"或包含对 XX 技能文件内容的依赖；允许在文本描述中提及其他技能名称作为用户指引。技能必须是自包含的，不能让用户打开 A 技能后又被要求去看 B 技能
2. **禁止在技能文件中放通用知识**：如"什么是 RESTful API"这种基础知识，应放在独立的参考资料中
3. **禁止技能文件超过 200 行**：超限说明技能太复杂，需要拆分
4. **禁止使用过于抽象的 Prompt**：Prompt 必须包含具体的输出格式、示例和约束条件
5. **禁止只写"让 AI 帮你做 X"而不给 Prompt**：必须提供可直接复制的 Prompt 模板

## 八、示例完整文件

```markdown
---
name: "需求去重"
description: "识别并合并重复或相似的需求描述"
type: "skill"
version: "1.0.0"
date: "2026-04-29"
parent-agent: "需求采集智能体"
standalone: true
tags: ["需求", "去重", "合并"]
input: "原始需求列表（每条有唯一编号）"
output: "去重后的需求列表 + 合并说明"
estimated-time: "10-20分钟"

# 技能：需求去重

[← 返回需求采集智能体](../需求采集智能体.md)
[← 返回需求工程集群索引](../INDEX.md)
[← 返回总索引](../../../INDEX.md)

## 适用场景

✅ **适合使用**：
- 当需求来自多个来源（客户口述、邮件、会议纪要），发现重复或相似描述时
- 当不同干系人用不同措辞描述了同一个需求时
- 整理需求清单前，需要先清理冗余时

❌ **不适合使用**：
- 需求数量很少（< 5 条），肉眼即可判断时
- 需求之间存在矛盾需要裁决时（先用其他需求分析技能处理）

## 输入

### 必需输入
- 原始需求列表，每条需求包含：
  - 唯一编号（如 R001, R002）
  - 需求来源（谁提的）
  - 需求描述
  - 优先级（P0/P1/P2，如有）

### 输入示例

```
R001 [客户A-口述] 用户可以通过关键词搜索图书
R002 [客户B-邮件] 系统需要提供搜索功能，让用户找书
R003 [客户A-口述] 用户可以按分类浏览图书
R004 [会议纪要] 用户能够通过书名、作者、ISBN 搜索图书
```

## 输出

### 输出格式

1. **去重后的需求清单**（表格）
2. **合并说明**（哪些编号合并到了哪一条）
3. **冲突标记**（如果有表述相似但要求矛盾的情况）

### 输出示例

```
## 去重后的需求清单

| 新编号 | 合并来源 | 统一描述 | 优先级 |
|:---:|:---|:---|:---:|
| D001 | R001, R002, R004 | 用户可通过关键词（书名/作者/ISBN）搜索图书 | P0 |
| D002 | R003 | 用户可按分类浏览图书 | P0 |

## 合并说明
- R001、R002、R004 描述的是同一需求，合并为 D001
- R004 补充了具体的搜索维度（书名/作者/ISBN），采纳为统一描述

## 冲突标记
无
```

## 执行步骤

### 步骤 1：提取核心语义
对每条需求，提取：核心动词 + 核心对象 + 关键条件

### 步骤 2：相似度分组
将核心语义相同或高度相似的需求归为一组

> **判断标准**：如果两个需求实现后会得到同一个功能，就是重复的

### 步骤 3：选择最优表述
每组中选择一个最清晰、最完整的表述作为统一描述

### 步骤 4：输出结果
按输出格式整理结果

## Prompt 模板

### 模板

```
你是一名需求分析师，擅长识别和合并重复需求。

## 原始需求列表
{粘贴带编号的需求列表}

## 任务
请对以上需求进行去重和合并：
1. 识别描述相同或相似的需求
2. 每组重复需求合并为一条，选择最清晰完整的表述
3. 标注每条合并需求的原始来源编号
4. 如果存在表述相似但要求矛盾的需求，标记为冲突

## 输出格式
1. 去重后的需求清单（表格：新编号 | 合并来源 | 统一描述 | 优先级）
2. 合并说明
3. 冲突标记（如有）
