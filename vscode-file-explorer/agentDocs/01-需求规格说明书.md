# AI 技能知识库平台 —— 需求规格说明书

## 一、功能需求

### FR-001 角色导向导航

**优先级**：P0

**描述**：系统必须提供按角色分类的导航入口，让用户根据自身角色快速找到相关智能体集群。

**验收标准**：
- [ ] 总索引页包含"按角色浏览"区域
- [ ] 覆盖至少 6 个角色：产品经理、前端开发、后端开发、测试工程师、运维工程师、技术负责人
- [ ] 每个角色链接到对应的智能体集群索引
- [ ] 角色定义包含"这个角色日常做什么"的简要说明

### FR-002 场景导向导航

**优先级**：P0

**描述**：系统必须提供按工作场景分类的导航入口，让用户根据当前任务快速找到所需技能。

**验收标准**：
- [ ] 总索引页包含"按场景查找"区域
- [ ] 覆盖至少 10 个高频场景，如"写需求文档""设计 API""审查代码""写测试用例"
- [ ] 每个场景链接到具体技能文件
- [ ] 场景描述用用户原话（如"我想把需求转成原型"）

### FR-003 智能体定义管理

**优先级**：P0

**描述**：每个智能体必须有标准化的定义文件，包含角色定位、能力清单、安装说明。

**验收标准**：
- [ ] 每个智能体一个独立的 Markdown 文件
- [ ] 包含标准化的 frontmatter 元数据（name, type, cluster, roles）
- [ ] 包含角色定位说明（一句话 + 一段话）
- [ ] 包含技能清单表格（技能名、说明、文件链接）
- [ ] 包含多平台安装说明（Claude Code / Cursor / 通用）
- [ ] 包含使用示例
- [ ] 详见 [03-智能体定义规范](03-智能体定义规范.md) 和 [04-技能定义规范](04-技能定义规范.md) 的完整模块定义

### FR-004 技能定义管理

**优先级**：P0

**描述**：每个技能必须有标准化的定义文件，包含适用场景、执行步骤、Prompt 模板、独立安装说明。

**验收标准**：
- [ ] 每个技能一个独立的 Markdown 文件
- [ ] 包含标准化的 frontmatter 元数据（name, type, parent-agent, standalone）
- [ ] 包含适用场景说明
- [ ] 包含明确的输入/输出定义
- [ ] 包含可执行的步骤或流程
- [ ] 包含可直接复制使用的 Prompt 模板
- [ ] 标记是否可独立使用（standalone: true/false）
- [ ] 包含独立安装到不同平台的说明
- [ ] 详见 [03-智能体定义规范](03-智能体定义规范.md) 和 [04-技能定义规范](04-技能定义规范.md) 的完整模块定义

### FR-005 集群索引管理

**优先级**：P0

**描述**：每个智能体集群必须有索引文件，汇总本集群的所有智能体和技能。

**验收标准**：
- [ ] 每个集群一个 INDEX.md 文件
- [ ] 包含本集群所有智能体列表（含定位说明）
- [ ] 包含本集群所有技能列表（按常用度或字母排序）
- [ ] 包含技能与智能体的归属关系
- [ ] 链接到上级总索引

### FR-006 技能按需取用

**优先级**：P0

**描述**：用户必须能够只取用一个技能，而不必安装整个智能体。

**验收标准**：
- [ ] 每个技能文件是自包含的（不依赖智能体文件也能理解完整上下文）
- [ ] 技能文件包含独立的 Prompt 模板
- [ ] 用户可以直接复制单个技能文件到自己的 AI 工具使用
- [ ] 技能文件明确标注所属智能体（用于追溯）

### FR-007 智能体整体安装

**优先级**：P1

**描述**：用户必须能够安装完整的智能体，一次性获得该角色的全部能力。

**验收标准**：
- [ ] 智能体文件提供"整体安装"说明
- [ ] Claude Code 用户知道应该复制哪些文件到哪个目录
- [ ] Cursor 用户知道如何引用整个智能体定义
- [ ] 安装后 AI 工具能理解该智能体的全部能力范围

### FR-008 多平台格式支持

**优先级**：P1

**描述**：技能内容必须支持至少三种平台格式：Claude Code Skill、Cursor Rules、通用 Prompt。后续视维护负担扩展至 Windsurf、VS Code + Copilot 等平台。

**验收标准**：
- [ ] 每个技能包含通用版本（纯 Markdown，不依赖特定工具）
- [ ] 技能文件内提供 Claude Code 安装方式说明
- [ ] 技能文件内提供 Cursor 安装方式说明
- [ ] 通用版本格式适用于 ChatGPT、Claude Web、其他 AI 工具

### FR-009 跨集群技能复用

**优先级**：P2

**描述**：如果一个技能被多个智能体需要，应该只存一份，通过链接引用。

**验收标准**：
- [ ] 相同技能不在多个地方重复存储
- [ ] 技能文件位置在集群的 skills/ 目录下
- [ ] 其他智能体通过相对路径链接引用
- [ ] 修改一处，所有引用处同步更新
- [ ] 跨集群复用的技能放在使用频率最高的集群 skills/ 目录下，其他集群通过相对路径链接引用

### FR-010 版本与更新标记

**优先级**：P2

**描述**：每个文件必须包含版本信息和最后更新时间。

**验收标准**：
- [ ] 所有文件 frontmatter 包含 version 字段
- [ ] 所有文件 frontmatter 包含 `date` 字段（用于记录版本日期）
- [ ] 重大更新有变更说明
- [ ] 用户能判断当前使用的技能是否是最新版

### FR-011 搜索与发现增强

**优先级**：P2

**描述**：除了索引导航，还应提供标签、关键词等辅助发现机制。

**验收标准**：
- [ ] 技能文件 frontmatter 包含 tags 字段
- [ ] 支持按标签过滤（通过索引页或文档工具实现）
- [ ] 技能描述中包含可能被搜索到的关键词

### FR-012 使用反馈机制

**优先级**：P2

**描述**：每个技能文件底部应预留反馈区域，收集使用体验。

**验收标准**：
- [ ] 技能文件底部包含"反馈入口"链接（指向 Git Issue 模板或独立 feedback.md），不直接在技能文件内保留反馈表格，避免多人编辑冲突

## 二、非功能需求

### NFR-001 可读性

- 所有文档必须使用清晰的中文撰写
- 避免使用晦涩术语，必要的术语必须有定义
- 文件结构层次分明，使用 Markdown 标题层级规范

### NFR-002 可维护性

- 目录结构直观，新成员能在 5 分钟内理解组织方式
- 技能文件独立，修改一个技能不影响其他技能
- 使用 Git 管理版本，变更可追溯

### NFR-003 可扩展性

- 新增集群只需创建新的集群文件夹和索引
- 新增智能体只需在对应集群下新增文件
- 新增技能只需在对应集群 skills/ 目录下新增文件
- 不修改现有文件结构即可扩展

### NFR-004 一致性

- 所有文件遵循统一的 frontmatter 格式
- 所有智能体文件遵循统一的内容模块
- 所有技能文件遵循统一的内容模块
- 索引页面的表格格式统一

### NFR-005 平台无关性

- 技能的核心内容（适用场景、执行步骤、Prompt 逻辑）不依赖任何特定 AI 工具
- 平台特定的安装说明作为适配层，允许与核心内容共存于同一文件

### NFR-006 可移植性

- 知识库可以整体复制到其他 Git 仓库或文档系统
- 不依赖特定软件或服务运行
- 纯 Markdown + 文件夹结构，任何文本编辑器都能维护

## 三、用户故事

### US-001 产品经理找需求技能

> 作为一名产品经理，我想在知识库中快速找到需求分析相关的技能，以便我能用 AI 辅助写出高质量的需求文档。

**场景**：
1. 我打开知识库总索引
2. 我看到"按角色浏览"区域，点击【产品经理】
3. 跳转到需求工程集群索引
4. 我看到"需求分析智能体"包含 5 个技能
5. 我今天要写 SRS，打开"SRS文档生成"技能
6. 复制 Prompt 模板到 Claude Code，开始使用

**验收**：整个过程在 2 分钟内完成。

### US-002 后端开发只取用一个技能

> 作为一名后端开发，我今天只需要做 API 设计，不需要完整的 API 设计智能体。

**场景**：
1. 我通过场景索引找到"API 设计"技能
2. 打开技能文件，看到它是独立的（standalone: true）
3. 直接复制 Prompt 模板到 Cursor
4. 输入我的具体需求，得到 API 设计方案

**验收**：不需要安装整个 API 设计智能体，只用一个技能就解决了问题。

### US-003 新成员了解团队 AI 工作流

> 作为一名新入职的前端开发，我想了解团队怎么用 AI 辅助前端开发。

**场景**：
1. 我打开知识库总索引
2. 点击【前端开发工程师】
3. 看到前端开发相关的智能体和技能列表
4. 阅读技能说明，了解团队的前端开发规范
5. 安装"前端组件开发"智能体到 Claude Code

**验收**：新成员在 30 分钟内了解团队的 AI 协作方式。

### US-004 技能维护者更新内容

> 作为一名技能维护者，我发现"需求去重"技能的 Prompt 需要优化。

**场景**：
1. 我找到 `clusters/01-需求工程/skills/需求去重.md`
2. 修改 Prompt 模板
3. 更新 frontmatter 中的 version 和 date
4. 在文件底部的"变更说明"区块记录本次修改内容（使用反馈本身通过 Git Issue 提交，不在文件内保留反馈表格）
5. 提交 Git commit

**验收**：只修改一个文件，不影响其他任何技能或智能体。

## 四、约束条件

1. **格式约束**：所有文件必须是 Markdown 格式
2. **存储约束**：使用 Git 仓库管理，不使用数据库
3. **访问约束**：不开发 Web 界面，通过 Git 仓库或文件系统直接访问
4. **工具约束**：第一阶段不开发自动格式转换工具，手工维护多平台适配
5. **语言约束**：主要使用中文撰写，必要时保留英文术语（如 Claude Code、Cursor）
