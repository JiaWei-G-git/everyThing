# AI 技能知识库平台 —— 实施路线图

## 一、总体策略

采用**渐进式交付**策略：
1. 先搭建基础设施和框架（让系统"能用"）
2. 再填充核心内容（让系统"好用"）
3. 最后扩展和优化（让系统"常用"）

不追求一次性完成所有 17 个智能体和 80 个技能，而是按优先级分批交付。

## 二、阶段划分

### 阶段一：基础设施搭建（第 1-2 周）

**目标**：建立完整的目录结构、模板体系和文档规范，验证信息架构的可用性。

#### 交付物

| 编号 | 交付物 | 说明 |
|:---|:---|:---|
| D1-1 | 根目录 INDEX.md | 总索引，包含按角色和按场景两个入口 |
| D1-2 | 全部集群文件夹 | 6 个集群的文件夹结构和 INDEX.md |
| D1-3 | 智能体模板 | `templates/智能体模板.md`，含完整的示例和注释 |
| D1-4 | 技能模板 | `templates/技能模板.md`，含完整的示例和注释 |
| D1-5 | 全部 17 个智能体定义文件 | 只有框架内容（角色定位 + 技能清单），技能文件先留空链接 |
| D1-6 | 文档规范 | `docs/贡献指南.md`、`docs/命名规范.md` |

#### 具体任务

- [ ] 创建完整的目录结构
- [ ] 编写总索引 INDEX.md
- [ ] 编写 6 个集群的 INDEX.md
- [ ] 编写 17 个智能体的框架文件（角色定位 + 技能清单表格）
- [ ] 编写智能体模板和技能模板
- [ ] 编写贡献指南和命名规范
- [ ] 邀请 2-3 名团队成员浏览结构，收集反馈

#### 验收标准

- [ ] 新用户能在 3 分钟内从 INDEX.md 找到任意一个智能体
- [ ] 新用户能理解目录结构的组织逻辑
- [ ] 所有链接有效，无 404
- [ ] 模板文件可以直接复制使用

#### 风险与应对

| 风险 | 影响 | 应对 |
|:---|:---:|:---|
| 目录结构不被团队接受 | 高 | 提前与关键用户沟通，做纸质原型测试 |
| 集群划分有争议 | 中 | 预留调整空间，用序号前缀方便重新排序 |
| 链接维护困难 | 低 | 使用工具检查链接有效性，纳入 CI |

#### 退出条件

若框架不被团队接受，则调整结构后再进入阶段二。

---

### 阶段二：MVP 技能集（第 3-6 周）

**目标**：填充最高频使用的 15-20 个技能，覆盖核心工作场景，验证技能模板的有效性。

#### 交付物

| 编号 | 交付物 | 说明 |
|:---|:---|:---|
| D2-1 | 15-20 个完整的技能文件 | 按优先级列表填充 |
| D2-2 | 至少 1 个完整的智能体 | 所有技能齐全，可整体安装试用 |
| D2-3 | 内部试用报告 | 3-5 名用户的试用反馈 |

#### 技能填充优先级

**P0（必须先做，6-8 个）**：

| 技能 | 所属集群 | 为什么优先 |
|:---|:---|:---|
| 需求拆解 | 需求工程 | 所有项目的起点 |
| SRS文档生成 | 需求工程 | 需求阶段的最终交付物 |
| 根据需求生成原型 | 设计工程 | 需求到设计的桥梁 |
| API设计 | 开发工程 | 前后端协作的核心 |
| React组件开发 | 开发工程 | 前端最高频任务 |
| CRUD接口开发 | 开发工程 | 后端最高频任务 |
| 测试用例生成 | 测试工程 | 测试工作的起点 |
| 代码审查检查清单 | 专项能力 | 质量保障的通用需求 |

**P1（紧接着做，6-8 个）**：

| 技能 | 所属集群 | 补充说明 |
|:---|:---|:---|
| 需求去重 | 需求工程 | 需求整理时高频使用 |
| 边界划分 | 需求工程 | 与需求拆解配套 |
| 根据需求编写设计文档 | 设计工程 | 与原型生成配套 |
| 架构图生成 | 开发工程 | 架构设计阶段使用 |
| 认证授权设计 | 开发工程 | API 设计的安全补充 |
| UI自动化测试 | 测试工程 | 与测试用例生成配套 |
| 接口自动化测试 | 测试工程 | 后端开发后的必做任务 |
| OWASP检测 | 专项能力 | 上线前的必做检查 |

> P1 技能根据阶段二实际进度，可延后至阶段三填充。

**P2（后续补充）**：
其余技能根据阶段二的反馈和实际使用频率决定填充顺序。

#### 具体任务

- [ ] 按优先级列表逐个编写技能文件
- [ ] 每个技能文件至少经过 1 人评审
- [ ] 选择 1 个智能体（建议"需求分析智能体"），补齐所有技能，验证整体安装流程
- [ ] 组织 3-5 名同事试用，收集反馈

#### 验收标准

- [ ] P0 技能全部完成，内容完整可用
- [ ] 至少有 1 人成功使用某个技能完成了实际工作
- [ ] 试用反馈中，"找不到"和"不会用"的问题占比 < 20%
- [ ] 技能文件的平均评分 ≥ 6/10

#### 风险与应对

| 风险 | 影响 | 应对 |
|:---|:---:|:---|
| 技能写得太泛，不实用 | 高 | 每个技能必须基于真实项目经验编写 |
| Prompt 效果不稳定 | 中 | 每个 Prompt 模板必须经过实际测试 |
| 团队不愿意试用 | 中 | 选择有痛点的同事，解决他们的问题 |
| 维护者时间不足 | 高 | 限制阶段二范围，只做 P0 技能 |

#### 退出条件

若到第 6 周末 P0 技能未完成，则削减 P1 范围。

---

### 阶段三：全集群覆盖（第 7-10 周）

**目标**：补齐所有集群的核心技能，覆盖从需求到运维的全链路。

#### 交付物

| 编号 | 交付物 | 说明 |
|:---|:---|:---|
| D3-1 | 40-50 个完整技能 | 每个集群至少 5-8 个技能 |
| D3-2 | 至少 3 个完整智能体 | 可整体安装使用 |
| D3-3 | 平台适配版本 | 高频技能提供 Claude Code + Cursor 两种格式 |
| D3-4 | 使用数据报告 | 哪些技能被用得最多，哪些没人用 |

#### 填充策略

按集群逐个补齐：

1. **第 7 周**：需求工程 + 设计工程（补齐剩余的）
2. **第 8 周**：开发工程（补齐前端、后端、架构）
3. **第 9 周**：测试工程 + 专项能力
4. **第 10 周**：运维交付 + 平台适配

#### 验收标准

- [ ] 每个集群至少有 5 个可用技能
- [ ] 至少有 3 个智能体可以完整安装使用
- [ ] 高频技能（周使用 ≥ 2 次）有 Claude Code 和 Cursor 两种格式
- [ ] 用户可以在 2 分钟内找到任意集群的索引

#### 退出条件

若某集群技能填充率 < 50%，则延期 1 周。

---

### 阶段四：优化与扩展（第 11-16 周）

**目标**：基于使用数据优化现有技能，扩展平台适配，建立持续更新机制。

#### 交付物

| 编号 | 交付物 | 说明 |
|:---|:---|:---|
| D4-1 | 技能优化版 | 根据反馈优化的 V1.1+ 版本 |
| D4-2 | 自动化工具（可选） | 如格式转换脚本、链接检查脚本 |
| D4-3 | 新技能 | 根据实际工作中发现的新场景补充 |
| D4-4 | 贡献指南完善版 | 让团队成员也能参与维护 |

#### 优化方向

1. **内容优化**：
   - 根据使用反馈，优化 Prompt 模板
   - 合并使用频率低、内容重复的技能
   - 拆分用户反馈"太复杂"的技能

2. **体验优化**：
   - 优化索引结构，让高频技能更容易找到
   - 补充更多的使用示例
   - 添加"常见问题"区块

3. **平台适配扩展**：
   - 指 P1/P2 平台如 VS Code Copilot、Windsurf 等，Claude Code 和 Cursor 适配在阶段三已完成
   - 高频技能补充 VS Code Copilot 版本
   - 探索 Windsurf 适配
   - 制作"一键安装"脚本（如 `install-claude-skills.sh`）

4. **社区建设**：
   - 完善贡献指南，鼓励团队成员提交改进
   - 建立定期评审机制（如双周技能评审会）
   - 收集并整理优秀 Prompt 案例

## 三、里程碑时间线

```
Week 1-2    [阶段一] 基础设施搭建
            ├── 目录结构
            ├── 索引系统
            ├── 模板文件
            └── 17 个智能体框架

Week 3-6    [阶段二] MVP 技能集
            ├── P0 技能（8个）
            ├── P1 技能（8个）
            └── 内部试用

Week 7-10   [阶段三] 全集群覆盖
            ├── 需求+设计集群补齐
            ├── 开发集群补齐
            ├── 测试+专项能力补齐
            └── 运维交付补齐

Week 11-16  [阶段四] 优化与扩展
            ├── 技能优化
            ├── 平台适配扩展
            └── 贡献机制建立
```

## 六、关键决策点

### 决策点 1：阶段一结束（第 2 周末）

**问题**：目录结构和信息架构是否被团队接受？

**决策选项**：
- A. 接受当前结构，进入阶段二
- B. 调整结构，延期 1 周
- C. 重新定义范围，缩小集群数量

**决策依据**：3-5 名关键用户的反馈，"找不到"和"看不懂"的反馈占比 < 30%

### 决策点 2：阶段二结束（第 6 周末）

**问题**：MVP 技能集是否有实际价值？

**决策选项**：
- A. 继续投入，进入阶段三
- B. 暂停扩展，先优化现有技能
- C. 调整方向，聚焦最有价值的集群

**决策依据**：至少 1 人使用技能完成了实际工作，且愿意继续使用

### 决策点 3：阶段三结束（第 10 周末）

**问题**：是否需要开发自动化工具？

**决策选项**：
- A. 手工维护
- B. 开发简单转换脚本
- C. 暂不决定，视阶段三的维护负担评估

**决策依据**：技能数量 > 30 且维护负担明显时，选 B；否则选 A 或 C，待后续评估
