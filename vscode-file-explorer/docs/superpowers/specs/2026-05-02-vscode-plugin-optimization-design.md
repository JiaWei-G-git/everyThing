# VS Code 插件 v0.2.0 优化设计文档

## 项目信息

- **日期**: 2026-05-02
- **主题**: 基于 200 个内部智能体方案的需求洞察，优化 ai-knowledge-base VS Code 插件
- **版本**: v0.2.0
- **状态**: 待实现

---

## 背景与目标

### 原始输入

收集了公司内部约 200 个智能体方案，覆盖开发、项目管理、数据分析、运维、售前、电力行业等十几个角色领域。这些方案反映了各部门的真实工作痛点和 AI 使用需求。

### 核心洞察（经需求分析提炼）

1. **用户按任务场景找智能体，而非按角色身份**——大量智能体是"多面手"，单一角色分类不够用
2. **文档/材料标准化生成是最大痛点**（~30%方案）——零散输入 → 按模板整理 → 标准格式输出
3. **数据开发和 SQL 是开发类的重要诉求**（~40%开发类方案）——开发工程集群需围绕数据开发强化
4. **行业专属方案占比高且高度场景化**——需要支持"领域知识包"机制
5. **运维/故障排查的价值在于流程标准化**——最佳载体是"可执行检查清单"而非纯 Prompt

### 设计目标

1. 支持**双分类导航**（角色 + 场景），让用户既能按身份浏览也能按任务查找
2. 开发工程集群作为**核心首位**，数据开发和测试作为子分类
3. 新增**行业知识包**加载机制，电力行业作为首个行业包
4. 新增**检查清单式 Skill**交互模式，支撑运维/故障排查类场景
5. 保持**混合门户架构**：核心层（~30 个精选 Skill 内置）+ 扩展层（批量导入）
6. 使用统计仅做**本地个人维度**，基于 globalState 持久化

## 核心功能 1：行业知识包机制

### 目录结构

```
vscode-file-explorer/
├── templates/
│   └── clusters/              # 核心模板（6个集群）
└── industry-packages/         # 行业知识包目录（用户可配置路径）
    └── 电力行业/
        ├── package.json       # 行业包元数据
        └── clusters/
            ├── 01-电网规约/
            ├── 02-采集计量/
            └── 03-线损分析/
```

### 行业包元数据格式

```json
{
  "name": "电力行业",
  "version": "1.0.0",
  "description": "南方电网电力行业专属智能体包",
  "clusters": ["电网规约", "采集计量", "线损分析"]
}
```

### 设置页扩展

新增"行业知识包"管理区域：

- 显示已加载的行业包列表（名称、版本、集群数、Skill 数）
- 支持启用/禁用开关
- 支持添加新的行业包路径
- 支持移除行业包

### 加载逻辑

1. KnowledgeService 启动时先加载核心模板
2. 读取设置中配置的行业包路径列表
3. 遍历已启用的行业包路径，解析其目录结构
4. 将行业包的 Skill 合并到知识库中，标记 `source: 'industry'` 和 `industryPackage`
5. 同名集群自动合并展示（核心 + 行业的 Skill 在同一集群下）

## 核心功能 2：检查清单式 Skill

### 适用场景

运维/故障排查类 Skill，核心价值是"按步骤执行检查"而非"生成一段文字"。

### Markdown 扩展格式

```markdown
---
name: 采集故障排查指引
description: 针对终端离线、召测失败等问题的标准化排查流程
type: checklist
tags: [运维, 采集, 故障排查]
---

## 步骤 1：确认故障现象
- [ ] 终端完全离线（ping 不通）
- [ ] 仅召测失败（ping 通但业务不通）

## 步骤 2：检查网络层
- [ ] Redis 集群连接状态正常
- [ ] 采集服务进程在运行
```

### 前端渲染

技能详情页检测到 `type: checklist` 时，渲染为交互式检查清单：

- 顶部显示进度条（完成百分比）
- 每个 `##` 标题作为一个可折叠的步骤组
- 每个 `- [ ]` 渲染为可勾选的复选框
- 步骤组按顺序展开，已完成的组可折叠
- 全部完成后显示"排查完成 ✅"提示

### 状态管理

v0.2.0：勾选状态仅保存在内存中，页面刷新后重置  
v0.2.1+：可持久化到 globalState（以 `skillId + timestamp` 为 key）

## 核心功能 3：使用统计

### 统计维度

仅本地个人统计，存储在 `globalState`：

```typescript
interface UsageStats {
  skillUsage: Record<string, number>;      // skillId -> 点击/使用次数
  scenarioUsage: Record<string, number>;   // scenarioTag -> 使用次数
  lastUsed: number;                        // 最后活跃时间戳
}
```

### 触发时机

- 用户点击 Skill 进入详情页：`skillUsage[skillId]++`
- 用户点击场景标签：`scenarioUsage[tag]++`
- 用户通过"快速创建"生成文档：对应场景计数 +1
- 用户安装 Skill 到 Claude/Cursor：对应 Skill 计数 +1

### "我的常用场景"计算

取 `scenarioUsage` 中频次最高的前 5 个场景，按 Dashboard 首页展示。

## 内容分层策略

### 核心层（内置，~30 个 Skill）

从 200 个方案中去重合并后，精选高价值、无重复的智能体：

| 集群 | 精选 Skill 示例 |
|------|----------------|
| 开发工程 | 代码结构化分析、SQL 优化、代码重构、Bug 急速定位、跨平台代码迁移 |
| 需求与产品 | 需求拆解器、用户故事拆解器、PRD 拆解、PRD 细节检查器 |
| 设计与数模 | 设计小助手、数模智能管家、表结构提取器、原型转概要设计 |
| 项目管理与交付 | 项目管理助手、会议纪要专属智能体、周报助手、项目材料自动编排 |
| 运维与数据分析 | 采集故障排查指引、巡检报告助手、数据智能统计分析、运维轻智体 |
| 行业专属（电力）| 电力规约智析、线损智能诊断、户变识别线损还原 |

### 扩展层（导入，~170 个 Skill）

其余全部方案通过**批量导入**进入用户本地知识库：

- **导入格式**：支持目录结构导入（zip 解压后的文件夹）或 JSON 清单导入
- **导入入口**：设置页中的"批量导入"按钮
- **存储位置**：用户配置的本地知识库路径（`aiKnowledgeBase.localPath`）
- **与核心层关系**：扩展层 Skill 和核心层 Skill 在界面上无区别，合并展示

## Extension ↔ Webview 通信协议扩展

### 新增消息类型

**Extension → Webview：**

| 消息类型 | 数据 | 触发时机 |
|---------|------|---------|
| `init:industryPackages` | `IndustryPackage[]` | 加载完行业包后 |

**Webview → Extension：**

| 消息类型 | 数据 | 说明 |
|---------|------|------|
| `switch:viewMode` | `{ mode: 'role' \| 'scenario' }` | 用户切换浏览模式 |
| `toggle:industryPackage` | `{ packageId, enabled }` | 启用/禁用行业包 |
| `add:industryPackage` | `{ path }` | 添加行业包路径 |
| `remove:industryPackage` | `{ packageId }` | 移除行业包 |
| `import:skills` | `{ path }` | 批量导入扩展层 Skill |

## 错误处理

| 场景 | 处理方式 |
|------|---------|
| 行业包加载失败 | 跳过该包，记录警告日志，Webview 显示"部分行业包加载失败"提示 |
| 行业包格式错误 | 提示用户检查 package.json 和目录结构 |
| 批量导入失败 | 显示具体错误文件，已成功导入的部分保留 |
| 检查清单渲染失败 | 降级为普通 Markdown 文本展示 |
