# SDLC Agent 平台 PRD v2.0

## 1. 产品概述

### 1.1 产品定义

SDLC Agent 是一个**跨平台、模块化**的软件开发生命周期自动化平台。核心产物基于**标准化 Markdown 格式**和**通用项目目录结构**，可运行在 **Kimi Work、Claude Code、Cursor 等任何支持 LLM 的工具**中。面向**南方电网项目**（计量系统、电能量系统等）优化，但支持通用企业后台管理系统作为验证。

### 1.2 目标用户

| 角色 | 技能背景 | 使用方式 |
|------|---------|---------|
| 用户（你） | 产品经理 + 项目经理 + 技术决策者 | 输入需求、阶段审核、最终验收 |

### 1.3 核心价值主张

**一句话**：通过半自动化的流水线，将企业后台管理系统从需求到工程骨架的交付周期缩短 50% 以上，同时保证产物一致性、跨项目经验复用，生成的代码定位为**工程骨架**（需人工完善业务逻辑）。

### 1.4 产品边界

**在范围内**：
- 需求发现 → 需求评审 → PRD → 原型 → UI 规范 → 技术架构 → 前端编码 → 后端编码 → 测试
- 模板学习（从现有文档提取格式标准）
- 跨项目经验复用（技术栈、踩坑、规范）
- 需求变更的半自动影响分析（人工确认后更新）
- 跨平台运行（Kimi Work / Claude Code / Cursor / 其他 LLM 工具）
- 模块化独立使用（任一阶段可单独触发，不依赖全流程）
- 细粒度模块拆分（如后端编码模块可独立使用）
- 自动化校验（代码编译、接口一致性、路由冲突、数据库覆盖检查）
- 需求追踪链（RTM）管理

**不在范围内**：
- 竞品分析（企业后台系统无需）
- 部署运维（后续扩展）
- 直接生成 .rp 原型文件（Axure 原生格式）
- 直接对接 Figma/Sketch 等设计工具
- 工具本身的安装包/插件开发（产物是标准化文档和代码，由用户自行在工具中加载）
- 生成可直接投产的生产代码（定位为工程骨架，需人工完善）
- 全自动影响追踪（采用半自动：LLM 标注可能影响范围，人工确认后更新）

## 2. 跨平台与模块化设计

### 2.1 设计原则

**核心原则**：产物与工具解耦。Agent 不依赖任何特定工具，所有产物使用标准化格式，可人工阅读和修改，可在任何支持 LLM 的工具中执行。

**代码定位**：所有生成的代码定位为**工程骨架**（Scaffold），包含：
- 完整的项目结构
- 数据库表定义（DDL）
- API 接口定义（路由、参数、响应格式）
- 页面组件框架（路由、布局、表格/表单结构）
- 权限控制框架（RBAC + 数据权限接口）
- 通用功能（登录、日志、异常处理、分页）

**不包含**：复杂业务逻辑、性能优化、安全加固、国产化适配细节、第三方系统集成。

### 2.2 跨平台运行

| 工具 | 运行方式 | 完整功能支持 |
|------|---------|------------|
| **Kimi Work** | 加载为 Skills，使用 Tools 读写文件 | ✅ 原生支持完整流程（文件操作、子 Agent、定时任务） |
| **Claude Code** | 将产物 Markdown 作为 prompt 上下文，使用 `read`/`write` 操作文件 | ✅ 支持所有阶段（需手动复制产物文件） |
| **Cursor** | 将产物文件作为 `.mdc` 规则或普通文件加载，通过编辑器操作 | ✅ 支持所有阶段（需手动管理文件） |
| **其他 LLM 工具** | 任何支持读取 Markdown 文件的工具 | ⚠️ 支持产物生成，文件操作需手动完成 |
| **纯 Web 聊天工具**（如 ChatGPT 网页版） | 无法直接操作文件 | ❌ 仅支持产物以代码块输出，用户手动保存 |

**降级模式**：工具不支持文件操作时，产物以 Markdown 代码块输出，用户手动保存到文件系统。

### 2.3 模块化独立运行

每个模块可**完全独立**使用，不依赖其他模块：

| 使用场景 | 调用的模块 | 前置产物（可选） | 最小输入模板 |
|---------|----------|----------------|------------|
| 需求访谈 | 需求发现 | 无 | 项目描述（一句话即可） |
| 生成 PRD | PRD 生成 | 需求发现报告（或手动输入） | 业务目标 + 核心功能列表 + 用户角色 |
| 设计原型 | 原型设计 | PRD（或手动输入） | 页面清单（名称 + 类型 + 核心操作） |
| 设计 UI | UI 规范 | 原型 + PRD（或手动输入） | 设计偏好（主色 + 组件库） |
| 设计架构 | 技术架构 | PRD + 配置（或手动输入） | 技术栈 + 数据实体清单 |
| 生成前端 | 前端编码 | 原型 + UI + 架构（或手动输入） | 页面结构 + API 清单 + 技术栈 |
| 生成后端 | 后端编码 | 架构 + PRD（或手动输入） | 表结构 + API 规范 + 技术栈 |
| 生成测试 | 测试验证 | 前后端 + PRD（或手动输入） | 功能点清单 + API 清单 |

**模块独立使用方式**：
1. 读取已有产物（Markdown 文件）作为输入
2. 无产物时，用户手动输入关键信息（如"请生成用户管理的后端代码，表结构如下..."）
3. 模块按自身逻辑生成产物，不依赖主控协调器

### 2.4 细粒度模块拆分

模块内部可进一步拆分：

| 模块 | 可拆分子模块 | 使用场景 |
|------|------------|---------|
| 后端编码 | 数据库设计、API 设计、业务逻辑、权限实现 | 只需要生成 DDL 时 |
| 前端编码 | 路由配置、页面组件、通用组件、权限控制 | 只需要生成表格页面时 |
| 技术架构 | 技术选型、数据库设计、API 规范、模块划分 | 只需要选型建议时 |
| 原型设计 | 页面结构清单、低保真 HTML | 只需要清单时 |
| 测试验证 | 功能测试用例、接口测试用例、权限测试用例 | 只需要接口测试时 |

**细粒度使用方式**：用户指定子模块："请生成数据库设计部分"、"请生成用户管理页面的前端代码"。

## 3. 需求追踪与一致性保障

### 3.1 需求 ID 体系

PRD 中每个需求条目分配唯一 ID：

```
REQ-001: 用户管理模块
REQ-002: 角色权限管理
REQ-003: 数据采集功能
...
```

**下游产物必须显式引用上游 ID**：
- 原型页面 → `关联需求：REQ-001, REQ-003`
- 数据库表 → `关联需求：REQ-002`
- API 接口 → `关联需求：REQ-001, REQ-004`
- 前端页面 → `关联需求：REQ-001`
- 测试用例 → `关联需求：REQ-001`

### 3.2 追踪矩阵（RTM）

产物：`artifacts/traceability-matrix.md`

记录需求 → 设计 → 代码 → 测试的完整映射：

| 需求 ID | 需求描述 | 原型页面 | 数据库表 | API 接口 | 前端页面 | 测试用例 | 状态 |
|---------|---------|---------|---------|---------|---------|---------|------|
| REQ-001 | 用户管理 | 用户列表 | sys_user | GET /users | /system/user | TC-001 | 已覆盖 |
| REQ-002 | 角色权限 | 角色列表 | sys_role | GET /roles | /system/role | TC-002 | 已覆盖 |

### 3.3 跨阶段契约文件

产物：`artifacts/api-contract.yaml`

作为前后端、原型、测试的共同输入，确保一致性：

```yaml
api_contract:
  version: "v1"
  base_path: "/api/v1"
  endpoints:
    - id: API-001
      path: "/users"
      method: "GET"
      request: { page: int, size: int, keyword: string }
      response: { data: [], total: int }
      related_req: [REQ-001]
      related_page: [用户列表]
      related_table: [sys_user]
      
    - id: API-002
      path: "/users/{id}"
      method: "PUT"
      request: { id: long, username: string, role_ids: [] }
      response: { success: boolean }
      related_req: [REQ-001]
```

**契约文件由技术架构阶段生成，后续阶段只读取不修改。**

### 3.4 自动化一致性校验

每个阶段门禁增加自动校验脚本 `gate-check.sh`：

| 阶段 | 自动校验内容 | 校验脚本 |
|------|------------|---------|
| 原型设计 | 比对 PRD 页面清单 vs 原型页面清单，检测遗漏 | `check-wireframe-coverage.py` |
| 技术架构 | 比对 PRD 数据实体 vs 数据库设计，检测遗漏 | `check-db-coverage.py` |
| 前端编码 | 扫描路由定义，检测重复或冲突；比对 API 契约 vs 前端调用 | `check-route-conflict.py` + `check-api-contract.py` |
| 后端编码 | 比对 API 契约 vs 后端实现路由；比对数据库设计 vs DDL | `check-api-contract.py` + `check-ddl-coverage.py` |
| 测试验证 | 比对 PRD 功能点 vs 测试用例，检测遗漏 | `check-test-coverage.py` |

**校验失败时**：
1. 阻止进入下一阶段
2. 将错误日志反馈给 LLM 自修复
3. 用户收到差异报告，确认后允许跳过（记录风险）

### 3.5 需求变更影响分析（半自动）

**策略**：放弃全自动影响追踪，采用**半自动**方式：

1. **变更标注**：用户提出变更（如"用户管理增加手机号字段"），Agent 基于 RTM 标注可能受影响范围：
   - 数据库表 sys_user（新增字段）
   - API 接口 API-001（请求参数变更）
   - 前端页面 用户列表（展示列变更）
   - 测试用例 TC-001（新增字段验证）
2. **人工确认**：用户确认影响范围，勾选需要更新的模块
3. **增量更新**：仅更新确认范围内的模块，生成新的 draft 版本
4. **Git 管理**：每次变更形成 commit，支持回滚与 diff 比对

### 4.2 功能详述

#### 4.2.1 需求发现模块（sdlc-discovery）

**功能点**：
1. **领域模板匹配**：根据项目描述匹配领域（energy-metering / general-admin），加载对应领域模板
2. **结构化访谈**：按预设问题清单引导用户澄清需求，问题清单支持领域定制
3. **需求发现报告生成**：将访谈结果整理为标准化 Markdown 文档，包含：
   - 业务背景、核心目标、约束与假设
   - 需求访谈记录（问题 + 回答）
   - 初步功能清单（无详细设计，仅高层描述）

**输入**：项目描述（一句话或详细描述）
**输出**：`artifacts/01-discovery/draft/discovery.md`

**门禁检查项**：
- □ 业务背景清晰（自动：关键词提取检查是否包含业务领域、用户群体）
- □ 核心目标明确（自动：目标数量 ≤ 5，每个目标可衡量）
- □ 关键约束已识别（人工：确认约束列表完整）
- □ 用户确认无遗漏需求（人工：最终确认）

#### 4.2.2 需求评审模块（sdlc-review）【新增】

**功能点**：
1. **可行性评估**：基于需求发现报告，评估技术可行性、资源需求、风险等级
2. **范围确认**：明确需求边界，识别模糊或冲突的需求
3. **风险登记**：记录识别的风险及缓解措施

**输入**：需求发现报告
**输出**：`artifacts/02-review/draft/review.md`

**门禁检查项**：
- □ 需求范围与业务目标一致（自动：比对目标 vs 功能清单，检测偏离）
- □ 技术可行性已评估（人工：确认技术方案可行）
- □ 关键风险已识别并记录（人工：确认风险清单完整）
- □ 用户确认进入下一阶段（人工：最终确认）

#### 4.2.3 PRD 生成模块（sdlc-prd）

**功能点**：
1. **模板学习**：读取用户上传的现有文档（PRD 样本），提取格式标准（章节结构、表格样式、命名规范）保存为 `style-guide.md`
2. **需求 ID 分配**：为每个需求条目分配唯一 ID（REQ-001 格式）
3. **RTM 初稿生成**：建立需求与功能模块的初步映射
4. **结构化 PRD 生成**：按模板/标准生成 PRD，包含：
   - 产品概述（定义、用户、价值主张）
   - 功能模块（模块清单 + 功能点详述，每个功能点带 REQ-ID）
   - 数据实体（数据模型初稿，带关联 REQ-ID）
   - 权限矩阵（角色 × 页面 × 操作，带关联 REQ-ID）
   - 接口概览（非详细 API，仅有哪些接口、做什么，带关联 REQ-ID）
5. **版本管理**：支持 `draft/` 迭代草稿和 `vN/` 正式版本

**输入**：需求发现报告 + 需求评审报告 + 项目配置 + 风格指南（可选）
**输出**：`artifacts/03-prd/draft/prd.md` + `artifacts/03-prd/draft/rtm-initial.md`

**门禁检查项**：
- □ 所有功能模块分配 REQ-ID（自动：检查所有功能点有 ID）
- □ 功能模块完整覆盖需求（自动：比对需求访谈功能清单 vs PRD 功能清单）
- □ 数据实体定义清晰（人工：确认实体清单完整）
- □ 权限矩阵已定义（人工：确认角色和操作完整）
- □ 无遗漏关键功能点（人工：最终确认）

#### 4.2.4 原型设计模块（sdlc-wireframe）

**功能点**：
1. **页面结构清单生成**：从 PRD 提取所有页面，生成结构化清单，每个页面标注关联 REQ-ID
2. **低保真 HTML 原型**：生成可点击的 HTML 页面，展示页面布局和信息架构
3. **Axure 衔接说明书**：生成供 Axure 参考的结构化说明书（页面清单、交互规则）
4. **一致性校验**：自动比对 PRD 页面清单 vs 原型页面清单，检测遗漏

**页面结构清单字段**：
- 页面名称、路由、页面类型、核心组件、数据源、操作权限、交互规则、关联页面、**关联 REQ-ID**

**输入**：PRD 文档 + RTM 初稿
**输出**：`artifacts/04-wireframe/draft/wireframe.md` + `artifacts/04-wireframe/draft/prototype.html`

**门禁检查项**：
- □ 所有 PRD 中的页面已定义（自动：`check-wireframe-coverage.py` 比对）
- □ 页面路由无冲突（自动：扫描路由，检测重复）
- □ 权限控制已体现（人工：确认角色权限分配）
- □ 交互规则清晰可执行（人工：确认交互逻辑完整）

#### 4.2.5 UI 设计模块（sdlc-ui-design）

**功能点**：
1. **设计令牌生成**：生成颜色系统、字体系统、间距系统
2. **组件使用规范**：定义按钮、表格、表单、弹窗、抽屉等组件的样式和交互规则
3. **编码就绪输出**：确保设计令牌可直接用于生成 CSS/样式代码

**输入**：PRD + 原型设计 + 项目配置（设计系统偏好）
**输出**：`artifacts/05-ui-design/draft/ui-design.md`

**门禁检查项**：
- □ 颜色系统覆盖所有状态（主色、辅助、成功、警告、错误、信息）
- □ 字体层级满足后台系统需求（标题/正文/辅助）
- □ 表格组件规范适合数据密集型场景
- □ 表单组件规范支持复杂输入（日期、选择、级联等）

#### 4.2.6 技术架构模块（sdlc-tech-arch）

**功能点**：
1. **技术栈确认**：读取项目配置中的技术栈，如未指定则根据项目特征推荐
2. **架构设计**：
   - 系统架构图（文本描述 + 层级说明）
   - 数据库设计（ER 图描述 + 表结构定义，带关联 REQ-ID）
   - **API 契约生成**：`artifacts/api-contract.yaml`（包含所有接口定义，带关联 REQ-ID）
   - 模块依赖关系
3. **项目配置验证**：确保技术栈选择合理（如南网项目是否需要国产化组件）
4. **一致性校验**：
   - 自动比对 PRD 数据实体 vs 数据库设计，检测遗漏
   - 自动比对 PRD 接口概览 vs API 契约，检测遗漏

**输入**：PRD + RTM + 项目配置
**输出**：`artifacts/06-tech-arch/draft/tech-arch.md` + `artifacts/06-tech-arch/draft/db-design.md` + `artifacts/api-contract.yaml`（契约文件，后续阶段只读）

**门禁检查项**：
- □ 数据库设计覆盖所有 PRD 数据实体（自动：`check-db-coverage.py` 比对）
- □ API 契约覆盖所有页面操作（自动：比对 PRD 接口概览 vs 契约）
- □ 权限模型与 PRD 权限矩阵一致（自动：比对矩阵 vs 架构设计）
- □ 技术栈选择有充分理由（人工：确认选型合理）

#### 4.2.7 前端编码模块（sdlc-frontend）

**功能点**：
1. **代码生成**：按设计令牌 + 技术架构生成前端**工程骨架**（非生产代码）
2. **页面实现**：
   - 路由配置（基于页面结构清单）
   - 页面组件（列表页、表单页、详情页、仪表盘）——**仅框架，无业务逻辑**
   - 通用组件（表格、表单、搜索、分页、弹窗、抽屉）
   - 权限控制（路由守卫、按钮级权限）——**框架实现，待填充业务规则**
   - API 调用封装（按 API 契约生成 service 层）
3. **代码质量**：遵循项目配置中的编码规范
4. **幻觉防御**：
   - 生成后执行 `npm install` + `npm run build`，编译失败则阻断并反馈 LLM 自修复
   - 比对 API 契约 vs 前端调用，检测调用不存在的接口
   - 检查依赖是否存在（通过 npm registry 查询）
5. **一致性校验**：
   - 扫描路由定义，检测重复或冲突
   - 比对 API 契约 vs 前端调用

**输入**：UI 设计 + 技术架构 + 原型设计 + API 契约
**输出**：`artifacts/07-frontend/` 代码目录

**门禁检查项**：
- □ 所有页面已生成可运行代码（自动：编译通过）
- □ API 调用与后端契约一致（自动：`check-api-contract.py` 比对）
- □ 权限控制已实现（自动：扫描权限相关代码）
- □ 代码可编译运行（自动：`npm run build`，无语法错误）
- □ 依赖真实存在（自动：检查 package.json 依赖有效性）

#### 4.2.8 后端编码模块（sdlc-backend）

**功能点**：
1. **数据库定义**：生成数据库表创建脚本（DDL），基于数据库设计文档
2. **API 实现**：
   - 路由定义（按 API 契约）
   - 控制器/Handler 层——**仅框架，无业务逻辑**
   - 业务逻辑层（Service/Business）——**接口定义，待实现**
   - 数据访问层（DAO/Repository）——**基础 CRUD**
   - DTO/VO 定义
3. **权限实现**：RBAC 模型（角色、权限、用户）+ **数据权限接口（待实现）**
4. **通用功能**：登录认证、日志、异常处理、分页封装——**框架实现**
5. **幻觉防御**：
   - 生成后执行编译（`mvn compile` / `go build` / `npm run build`），失败则阻断并反馈 LLM 自修复
   - 比对 API 契约 vs 后端实现路由，检测遗漏或虚构接口
   - 检查数据库语法（针对特定数据库类型，如 DWS 与标准 SQL 差异）
   - 安全编码检查：检测 SQL 注入风险、明文密钥存储、XSS 漏洞
6. **一致性校验**：
   - 比对 API 契约 vs 后端实现
   - 比对数据库设计 vs DDL

**输入**：技术架构 + PRD + API 契约
**输出**：`artifacts/08-backend/` 代码目录 + DDL 脚本

**门禁检查项**：
- □ 数据库表覆盖所有数据实体（自动：`check-ddl-coverage.py` 比对）
- □ API 实现覆盖所有契约接口（自动：`check-api-contract.py` 比对）
- □ 权限模型与 PRD 一致（自动：比对 RBAC 实现 vs 权限矩阵）
- □ 代码可编译运行（自动：编译检查）
- □ 无 SQL 注入风险（自动：扫描拼接 SQL）
- □ 无明文密钥存储（自动：扫描密钥相关代码）

#### 4.2.9 测试验证模块（sdlc-test）

**功能点**：
1. **测试用例生成**：

   - 接口测试用例（按 API 契约）
   - 权限测试用例（按权限矩阵）
   - 数据权限测试用例（按数据权限模型）
2. **测试执行**（可选，需测试环境）：
   - 生成 `docker-compose.yml` 模板，自动拉起测试数据库
   - 生成自动化测试脚本（JUnit / pytest / Playwright）
   - 基于 PRD 数据实体自动生成 Mock 数据
   - 执行测试并生成报告
3. **测试覆盖率检查**：
   - 功能覆盖率 ≥ 80%（自动比对 PRD 功能点 vs 测试用例）
4. **幻觉防御**：
   - 比对 PRD 功能点 vs 测试用例，检测遗漏
   - 比对 API 契约 vs 接口测试，检测遗漏

**输入**：PRD + RTM + 技术架构 + 前后端代码 + API 契约

**门禁检查项**：

- □ 权限测试覆盖所有角色（自动：扫描测试用例中的角色覆盖）
- □ 关键业务流程有端到端测试（人工：确认核心流程）
- □ 接口测试覆盖所有契约接口（自动：比对 API 契约 vs 接口测试）

#### 4.2.10 主控协调模块（sdlc-orchestrator）

**功能点**：
1. **项目初始化**：
   - 创建项目目录结构（含 `draft/` 和 `vN/` 目录）
   - 生成 `project-config.yaml`
   - 检查知识库中是否有可复用的经验
2. **阶段调度**：
   - 按顺序执行阶段 1 → 2 → 3 → 4 → 5 → 6 → 7 → 8 → 9
   - 支持从任意阶段启动（检查前置阶段产物）
   - 支持单阶段独立触发
   - 支持子模块独立触发（如"只生成数据库设计"）
3. **门禁管理**：
   - 阶段完成后暂停，等待用户审核
   - 先执行**自动校验脚本**，通过后进入人工审核
   - 支持三种审核模式：简单确认 / 结构化检查清单 / 直接修改文件
   - 记录门禁审核结果到 `gate-review.md`
4. **状态跟踪**：
   - 维护 `sdlc.log` 执行日志
   - 跟踪当前阶段、版本、审核状态
5. **需求变更处理**：
   - 基于 RTM 标注可能受影响范围
   - 人工确认影响范围后，触发增量更新
   - 生成 Git commit，支持回滚

**输入**：用户指令（启动项目、进入下一阶段、修改需求等）
**输出**：阶段产物 + 执行日志 + 门禁记录

#### 4.2.11 经验复用模块（sdlc-knowledge）【增强】

**功能点**：
1. **模板学习**：从用户上传的文档中提取格式标准，保存为 `style-guide.md`
2. **项目归档**：项目结束时自动归档到 `memory/`，使用固定模板：

```markdown
- **方案**：解决方案
```

3. **经验检索**：

## 6. 权限矩阵

本系统为单人使用（用户自己），无多用户权限控制。但产物中需要生成用户项目的权限矩阵。

**用户项目权限模型（增强版）**：

### 6.1 功能权限（RBAC）

| 角色 | 页面访问 | 操作权限 |
|------|---------|---------|
| 系统管理员 | 全部 | 全部 |
| 地市管理员 | 本市数据相关页面 | 增删改查（仅本市数据） |
| 区县管理员 | 本区县数据相关页面 | 增删改查（仅本区县数据） |
| 班组长 | 本班组数据相关页面 | 查询 + 部分修改（仅本班组） |
| 普通用户 | 指定页面 | 仅查询（指定范围） |

### 6.2 数据权限（南网特定）

| 层级 | 数据隔离范围 | 实现方式 |
|------|------------|---------|
| 省级 | 全省数据 | 无过滤 |
| 地市 | 本市数据 | 行级过滤（where org_id = 地市ID） |
| 区县 | 本区县数据 | 行级过滤（where org_id = 区县ID） |
| 班组 | 本班组数据 | 行级过滤（where team_id = 班组ID） |
| 供电所 | 本供电所数据 | 行级过滤（where station_id = 供电所ID） |

**数据权限实现框架**：
- 后端：MyBatis 拦截器 / AOP 切面，自动注入数据权限过滤条件
- 前端：表格列脱敏（敏感字段根据角色显示/隐藏）
- 配置：数据权限规则通过数据库配置，非硬编码


**模块拆分规则**：
- 按业务域拆分（用户管理、计量、数据、报表等）
- 每个模块独立生成代码，共享通用组件
- 模块间通过 API 契约对接，不直接依赖

## 9. 非功能需求

### 9.1 性能

- 单个阶段首轮生成 < 10 分钟（LLM 生成）
- 后续修改基于增量生成，< 5 分钟
- 大型项目自动拆分为子任务，每子任务 < 10 分钟

### 9.2 上下文管理

**策略**：每个模块每次只处理有限数量文件，通过文件路径传递中间结果，不将所有内容加载到 LLM 上下文。

| 场景 | 上下文策略 |
|------|----------|
| 小型项目（< 20 页面） | 全量加载 PRD + 产物 |
| 中型项目（20-50 页面） | 分模块加载（每次 1 个模块） |
| 大型项目（> 50 页面） | 分模块 + 分页面加载（每次 1 个页面） |
| 代码生成 | 只加载当前模块的 API 契约 + 设计规范，不加载全部代码 |

**上下文压缩**：
- 历史产物通过文件引用，不重复加载全文
- 使用摘要代替全文（如"用户管理模块：5 个页面，已生成"）
- 大型 PRD 只加载当前阶段相关章节

### 9.3 可用性

- 所有产物使用 Markdown 格式，便于人工阅读和修改
- 代码产物遵循标准项目结构，可直接用 IDE 打开
- 自动校验失败时提供明确的错误日志和修复建议

### 9.4 可扩展性

- 新阶段可作为插件插入流水线
- 新领域模板可通过 `domain-templates/` 添加
- 技术栈切换通过 `project-config.yaml` 配置
- 新校验规则可通过 `scripts/` 添加

### 9.5 可靠性

- 每个阶段产物必须持久化到文件系统，不依赖内存状态
- 执行日志完整记录，支持异常恢复
- 自动校验失败阻断进入下一阶段，防止错误累积
- 需求变更基于 Git commit，支持回滚

### 9.6 安全性

- 生成代码安全编码检查：
  - SQL 注入风险扫描（检测拼接 SQL）
  - XSS 漏洞扫描（检测未转义输出）
  - 明文密钥存储检测
  - 硬编码密码检测
- 南网项目增加等保合规检查清单
- 代码定位为工程骨架，不直接用于生产环境

### 10.2 依赖关系

- **LLM 环境**：任何支持大模型上下文和文件操作的工具（Kimi Work / Claude Code / Cursor / 其他）
- **文件系统**：用于存储产物和读取已有文档
- **产物标准化**：所有产物为 Markdown / 代码 / YAML 文本格式，无工具专属依赖
- **Git**：用于需求变更的版本控制和回滚（可选，但强烈建议）
- **Docker**：用于测试环境拉起（可选）

### 10.3 跨平台兼容性说明

- **Kimi Work**：使用 Skill 系统加载，通过 Tools 读写文件，原生支持完整流程（自动校验、子 Agent、文件操作）
- **Claude Code**：将 `SKILL.md` 作为 `.md` 文件加载，通过 `read`/`write` 操作产物文件，支持所有阶段（自动校验需手动执行脚本）
- **Cursor**：将产物文件作为 `.mdc` 规则或普通文件加载，通过编辑器操作，支持所有阶段（自动校验需手动执行脚本）
- **其他工具**：只要支持读取 Markdown 和文件操作，即可运行（降级模式：产物以代码块输出）

**核心保证**：所有产物不依赖任何工具专属格式或 API，纯文本即可运行。自动校验脚本为通用 Python/Bash 脚本，可在任何环境执行。

### 10.4 已知风险与缓解

| 风险 | 影响 | 缓解措施 | 状态 |

| 需求理解偏差 | 高 | 多轮访谈 + 结构化门禁 + 用户直接修改文件 + RTM 追踪 | 已缓解 |
| 上下文长度限制 | 中 | 按模块分批生成；使用文件传递而非内存；上下文压缩策略 | 已缓解 |
| 技术栈知识缺失 | 中 | 项目配置提供技术栈文档链接；Agent 学习已生成代码；幻觉防御（依赖检查、语法检查） | 已缓解 |
| 产物一致性难以保证 | 高 | 跨阶段契约文件（api-contract.yaml）+ 自动化校验脚本 + 门禁阻断 | 已缓解 |
| 需求变更增量更新过于理想化 | 高 | 半自动影响分析（LLM 标注 + 人工确认）+ Git 版本控制 | 已缓解 |
| 经验复用检索能力不足 | 中 | 结构化记忆模板 + 关键词索引 + 向量检索（RAG） | 已缓解 |
| 跨平台执行差异 | 中 | 产物标准化 + 各平台适配指南 + 降级模式 | 已缓解 |
| 模块独立使用边界模糊 | 中 | 定义最小输入模板 + 输入完整性校验 + 明确独立使用场景 | 已缓解 |
| 门禁过度依赖人工 | 中 | 自动校验前置 + 人工仅处理语义性问题 | 已缓解 |
