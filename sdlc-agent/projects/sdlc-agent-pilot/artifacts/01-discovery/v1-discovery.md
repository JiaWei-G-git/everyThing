# SDLC Agent 需求发现报告

## 1. 项目背景

### 1.1 业务背景

用户正在开发一个**SDLC Agent（软件开发生命周期 Agent）**，用于项目开发全流程自动化。该 Agent 运行在 **Kimi Work** 环境中，通过 Kimi 的 Skills 系统实现。

### 1.2 用户当前状况

- **当前项目**：一个通用后端管理系统（作为试点项目，用于验证 Agent 流程）
- **长期业务**：南方电网项目（计量系统、电能量系统等），**这是核心目标领域**
- **工作性质**：企业级后台管理系统（B端 Admin 系统）开发，但专注优化南网相关领域
- **核心特征**：数据密集型、复杂权限体系、大量表格/表单、可能有实时数据

### 1.3 目标用户

| 角色 | 职责 | 使用方式 |
|------|------|---------|
| 用户（你） | 产品经理 + 项目经理 + 技术决策者 | 需求输入方、阶段审批人、最终验收者 |
| Kimi（SDLC Agent） | 自动化执行团队 | 按阶段产出、等待审核、执行迭代 |

---

## 2. 核心目标

构建一个**专注型 + 可配置**的 SDLC Agent 平台，满足：

1. **领域聚焦**：面向**南方电网项目**（计量系统、电能量系统等）优化流程模板，企业后台管理系统仅作为测试验证
2. **技术中立**：无默认技术栈，全部在项目级别配置
3. **半自动执行**：Agent 按阶段推进，每阶段结束需用户审核确认
4. **模板学习**：可从现有文档中提取格式标准，后续按该标准生成
5. **经验复用**：跨项目积累技术栈选择、踩坑指南、规范偏好
6. **变更支持**：支持需求频繁变更，产物支持版本控制

## 4. 关键设计原则

### 4.1 零默认技术栈

- Agent 不内置任何默认技术栈偏好
- 每个项目启动时通过**项目配置**指定技术栈
- 技术栈配置包含：前端框架、后端语言、数据库、UI 组件库、构建工具等

### 4.2 项目级配置

每个项目有一份 `project-config.yaml`：

```yaml
project_name: "南网计量系统 V2"
domain: "energy-metering"  # 领域标识，用于匹配经验模板
tech_stack:
  frontend: "Vue 3 + Element Plus"
  backend: "Java + Spring Boot"
  database: "PostgreSQL"
  ui_component_library: "Element Plus"
  build_tool: "Vite"
design_system:
  primary_color: "#409EFF"
  font_family: "系统默认"
  table_style: "紧凑表格，操作列右对齐"
  form_layout: "两列/三列自适应"
style_guide:
  prd_template: "learned-from-sample"  # 指向模板学习产物
  wireframe_format: "页面结构清单 + 低保真 HTML"
  naming_convention: "驼峰命名"
```

### 4.3 模板学习功能

- 项目启动时可上传现有文档（PRD、原型、代码等）
- Agent 分析并提取格式标准：
  - 章节结构（PRD 有哪些章节、顺序）
  - 表格样式（字段定义表格的列名）
  - 命名规范（数据库表名、API 路由风格）
  - 设计规范（颜色、字体、间距偏好）
- 提取结果保存为 `project-style-guide.md`
- 后续所有阶段按该标准生成

### 4.4 需求变更支持

- 产物支持**版本控制**（v1, v2, v3...）
- 支持变更后**增量更新**（只改受影响的部分，不推倒重来）
- 变更触发时，Agent 评估影响范围并建议：
  - 哪些下游产物需要同步更新
  - 变更优先级和风险

### 5.1 版本管理规则

**两种多版本场景：**

| 场景 | 处理方式 | 目录 |
|------|---------|------|
| **迭代草稿** | 同一需求，产物不满意 → 修改 → 再改 | `draft/` 下覆盖最新 |
| **需求变更** | 需求变了，产物要跟着变 | 新建 `vN/` 正式版本，永久保留 |

**门禁流程：**
1. Agent 产出 → 写入 `draft/`
2. 用户审核 → 不满意 → 修改意见 → Agent 覆盖 `draft/` → 再审
3. 用户审核 → 满意 → 通过门禁 → 从 `draft/` 移入 `vN/`（新建版本目录）
4. 需求变更 → 读取最新 `vN/` → 修改 → 写入 `draft/` → 通过门禁 → 移入 `vN+1/`

## 6. 阶段门禁审核机制

### 6.1 审核模式

**模式 A：简单确认**（默认）
- Agent 完成阶段后问："是否满意？满意回复 Y，不满意说修改意见"
- 用户回复 Y → 通过门禁，从 `draft/` 移入 `vN/`
- 用户说修改意见 → 迭代修改 → 覆盖 `draft/` → 再问

**模式 B：结构化检查清单**（可选）
- 用户说"要改" → Agent 展示检查清单逐项讨论
- 每个阶段有预设检查项（如 PRD 阶段：□ 功能模块完整 □ 权限矩阵已定义 □ 数据实体已列出）

**模式 C：直接修改文件**
- Agent 产物输出到 `draft/`
- 用户直接在文件中修改
- 用户告诉 Agent"改完了，继续" → Agent 读取修改后的 `draft/` → 进入下一阶段

### 6.2 审核产物

每个阶段门禁产出：
- `vN/gate-review.md`（门禁审核记录）
- 记录：审核人、审核日期、通过/修改意见、修改内容摘要

## 7. 项目记忆与经验复用

### 7.1 记忆内容

每个项目结束时自动归档：

| 记忆类型 | 内容 | 复用方式 |
|---------|------|---------|
| 技术栈选择 | 本项目用什么技术栈、选择原因 | 新项目推荐 |
| 踩坑指南 | 开发中遇到的问题和解决方案 | 同类项目预警 |
| 设计决策 | 关键架构决策及理由 | 新项目参考 |
| 规范文件 | 编码规范、命名规范、接口规范 | 新项目沿用 |
| 常用模式 | 重复出现的代码模式、组件模式 | 新项目模板 |

### 7.2 查询方式

- **自动推荐**：新项目启动时，Agent 根据项目描述匹配相关项目，自动推荐经验
- **被动查询**：用户主动问"看看之前计量系统项目用了什么表结构"
- **两者结合**

### 7.3 领域模板

按领域（如 energy-metering、general-admin）积累标准模板：
- 领域专属数据模型（如南网计量系统通用的台区、计量点、电能表模型）
- 领域专属功能清单（如计量系统通常需要：数据采集、线损计算、异常预警等）
- 领域专属需求访谈问题

## 8. 原型设计方式

### 8.1 输出形式

- **主要输出**：页面结构清单（Markdown 表格，描述每个页面的路由、包含组件、数据来源、交互逻辑）
- **辅助输出**：低保真 HTML 原型（可点击预览，作为 Axure 搭建的参考）
- **与 Axure 的衔接**：不直接生成 .rp 文件，生成供 Axure 参考的结构化说明书

### 8.2 页面结构清单字段

| 字段 | 说明 |
|------|------|
| 页面名称 | 如"用户管理" |
| 路由 | /system/user |
| 页面类型 | 列表页 / 表单页 / 详情页 / 仪表盘 |
| 核心组件 | 表格、搜索栏、操作按钮、分页器 |
| 数据源 | 对应后端 API |
| 操作权限 | 哪些角色可访问/操作 |
| 交互规则 | 如"点击编辑按钮弹出抽屉表单" |
| 关联页面 | 该页面跳转到哪些其他页面 |

## 9. UI 设计输出

- 不直接生成高保真视觉稿（Figma 等）
- 输出**设计令牌（Design Tokens）**和**组件使用规范**：
  - 颜色系统（主色、辅助色、状态色、中性色）
  - 字体系统（字号层级、字重、行高）
  - 间距系统（栅格、内边距、外边距）
  - 组件规范（按钮、表格、表单、弹窗、抽屉的样式和交互规则）
- 编码阶段直接按设计令牌生成 CSS/样式代码

## 10. 约束与假设

### 10.1 技术约束

- 运行在 Kimi Work 环境中
- 使用 Kimi Skills 系统实现（SKILL.md + scripts + references + assets）
- 代码生成通过 PythonRun 或子 Agent 执行
- 文件操作通过 Kimi Work 的读写工具
- 产物存储在本地文件系统（`projects/` 目录）

### 10.2 假设

- 用户具备技术背景，能理解技术架构决策并做审核
- 用户有最终代码修改和部署能力（Agent 产出初稿，用户可调整）
- 技术栈由用户在项目启动时指定（或从经验库推荐中选择）
- 原型设计阶段不替代 Axure，而是提供 Axure 的输入参考

### 10.3 已知局限

- 不直接生成 .rp 原型文件
- 不直接对接外部设计工具（Figma、Sketch）
- 部署阶段需要用户手动操作（或后续扩展）
- 代码质量依赖 LLM 能力，复杂业务逻辑仍需人工审核

## 11. 成功标准（如何衡量 Agent 成功）

| 指标 | 定义 | 目标 |
|------|------|------|
| 阶段通过率 | 每个阶段一次通过门禁的比例 | > 80% |
| 需求返工率 | 因需求理解偏差导致的返工次数 | < 20% |
| 编码效率 | 相比手写代码，节省的时间比例 | > 50% |
| 项目复用率 | 新项目复用已有项目经验的比例 | > 60% |
| 变更响应速度 | 需求变更后更新所有下游产物的时间 | < 30 分钟 |
