# 资产提炼技能 (Asset Extractor)

**版本**: v1.0.0
**类型**: Skill · 知识资产提炼器

当用户说 **"提炼资产"**、**"提取 Prompt/Skill"** 或表达类似意图时，AI 自动匹配本 Skill。

> 💡 **手动调用**：`/skill:asset-extractor`

---

## 功能说明

从 Vault 原料层（`00-Inbox` / `01-Work`）的工作记录中，调用 LLM 提取可复用的知识资产草稿，输出到 `00-Inbox-收件箱/待提炼/` 等待人工确认。

### 与 `pattern-miner` 的分工

| Skill | 适用场景 | 提炼方式 | 资产质量 |
|-------|---------|---------|----------|
| **`asset-extractor`** | 单条原料（外部教程、即兴笔记、某次特定会话） | 从单条记录中改写提取 | 适合轻量 Prompt，可能存在伪通用风险 |
| **`pattern-miner`** | 多条同类会话（同一问题出现 ≥3 次） | 跨案例抽象，归纳共性 | 适合通用 Skill，经过重复验证 |

**使用建议**：
- 从外部文章、教程、单条灵感中提炼 → 用 `asset-extractor`
- 从工作记录中发现反复出现的问题模式 → 用 `pattern-miner`

---

## 执行步骤

### 1. 确定输入原料并判断执行方式

解析用户输入，判断是**指定文件**还是**批量扫描**：

| 用户输入特征 | 执行方式 | 示例 |
|-------------|---------|------|
| 包含具体文件路径、@文件、`提炼这份记录` | **方式 A — 指定文件** | `@01-Work-工作记录/xx/会话记录-xxx.md` |
| 无具体路径，说`提炼资产`、`批量提炼`、`扫描未提炼` | **方式 B — 批量扫描** | `提炼本周资产` |

**方式 A — 指定文件**：
- 直接定位用户指定的原料文件
- 不对该文件做预筛选（用户明确要求提炼，即使长度较短也执行）
- 仅对该文件调用 LLM 提取

**方式 B — 批量扫描**：
- 扫描 `00-Inbox` 和 `01-Work` 下所有 `.md` 文件
- 排除已标记 `status: 已提炼` 的原料
- 对扫描结果执行预筛选（步骤 2.1）

### 2. 读取原料内容

读取原料文件的完整 Markdown 内容，作为 LLM 分析的输入。

#### 2.1 预筛选（仅方式 B 执行）

**方式 A（指定文件）跳过本步骤，直接进入步骤 3。**

**方式 B（批量扫描）**在调用 LLM 前，先对原料进行规则预筛选，**不通过筛选的原料直接跳过**，不调用 LLM：

| 筛选条件 | 规则 | 未通过处理 |
|---------|------|-----------|
| **最小长度** | 正文（不含 YAML Frontmatter）< 200 字 | 跳过，标记 `status: 内容过短，无需提炼` |
| **内容质量** | 不含任何代码块（```）、编号步骤、指令模板、问题分析 | 跳过，标记 `status: 无有效内容，无需提炼` |

**通过筛选的原料**进入步骤 3 调用 LLM。

> **方式 A 例外**：用户明确指定的文件不受预筛选限制，直接进入步骤 3。

### 3. 调用 LLM 提取资产

发送提炼指令 + 原料内容给 LLM，要求识别并提取以下 2 类资产：

| 资产类型    | 目标目录      | 说明                                   |
| ----------- | ------------- | -------------------------------------- |
| 优质 Prompt | `10-Prompts/` | 有效的提问模板，带使用场景             |
| 原子 Skill  | `30-Skills/`  | 可复用的操作流程，带触发条件和执行步骤 |

> **注意**：当前 MVP 版本仅实现 prompt 和 skill 两类资产的自动提取，agent/mcp/workflow/tutorial 类资产需人工识别后手动归档。

**LLM 系统提示词**（发送给 LLM 的 system prompt）：

````
你是一位知识库管理员，负责从工作记录中识别并结构化提取可复用的知识资产。

## 输入
下面会提供一段工作记录（可能包含对话日志、操作步骤、问题解决过程等）。

## 提取目标
仅提取以下两类资产，忽略所有其他信息：

### 1. 优质 Prompt
定义：在工作记录中实际使用过、且明确产生了有效结果的提问或指令模板。
提取标准：
- 该 Prompt 曾引导出高质量回答或正确完成了特定任务
- 具备可迁移性：替换具体业务参数后可在同类场景中复用
- 排除一次性的、高度依赖特定上下文而无法泛化的提问

提取要求：
- scenario：写明该 Prompt 适用的具体任务类型（如"代码审查""文档摘要""数据清洗规则生成"），而非笼统的"日常使用"
- content：提供完整的 Prompt 模板文本，用 `{{变量名}}` 标注需要替换的参数部分

### 2. 原子 Skill
定义：工作记录中出现的、可独立执行的操作流程，粒度为单一职责（一个 Skill 只解决一个具体问题）。
提取标准：
- 包含明确的触发条件（什么情况下需要执行）
- 包含可按顺序执行的具体步骤（至少2步，每步是一个可验证的动作）
- 排除过于宽泛的流程描述（如"优化系统性能"）和过于琐碎的单步操作（如"打开某个页面"）

提取要求：
- scenario：写明触发该 Skill 的具体条件或前置状态（如"当 API 响应时间超过 2s 且日志中出现连接池耗尽警告时"）
- content：按编号列出执行步骤，每步包含具体动作和预期结果

## confidence 判定规则
- high：工作记录中有明确证据表明该资产被成功使用过（如产出了正确结果、解决了问题）
- medium：工作记录中出现了该模式但缺少明确的成功验证证据
- low：从上下文推断可能有复用价值，但记录中未直接体现使用效果

## 输出格式
直接输出标准 Markdown 草稿。每条资产之间用 `---` 分隔。

**Prompt 资产格式**：
```markdown
---
type: prompt
name: 动词+宾语，不超过15字
scenario: 具体使用场景
content: 完整 Prompt 模板文本
confidence: high|medium|low
source: 来源原料文件路径
---

# {name}

## 场景
{scenario}

## 模板
{content}
```

**Skill 资产格式**：
```markdown
---
type: skill_candidate
name: 动词+宾语，不超过15字
scenario: 具体触发条件
content: |
  1. 步骤一...
  2. 步骤二...
confidence: high|medium|low
source: 来源原料文件路径
---

# {name}

## 触发条件
{scenario}

## 执行步骤
{content}
```

**边界规则**：
- 如果工作记录中没有符合上述提取标准的资产，直接回复：**无**
- 宁可少提取也不降低质量标准：模糊的、无法泛化的、缺少关键细节的内容不提取
- 若同一段记录中存在多个独立资产，逐条输出，用 `---` 分隔

````

**LLM 调用参数**：
- temperature: 0.3
- 超时: 60 秒
- 接口: `{LLM_BASE_URL}/chat/completions`

**Markdown 处理规则**：
1. 若 LLM 返回"无"或空内容，跳过该文件，不保存任何草稿
2. 若包含多条资产，按 `---` 分隔符拆分为独立内容块
3. 每条资产保存为一个独立的 `.md` 文件
4. 若格式异常，向用户汇报并跳过该文件，不中断流程

### 4. 保存草稿

将 LLM 返回的 Markdown 内容直接保存为草稿文件：

````

00-Inbox-收件箱/待提炼/Prompt-{名称}.md
00-Inbox-收件箱/待提炼/Skill-{名称}.md

````

**Prompt 草稿模板**：使用 `90-Templates/prompt_asset.md`，包含：
- title, category, source, extracted_date, applicable_scenario, maturity: draft

**Skill 草稿模板**：内联生成，包含：
- type: skill_candidate, name, source, extracted_date, status: draft
- 触发条件 / 执行步骤

### 5. 标记原料状态

对已完成提炼的原料，在 YAML Frontmatter 中更新 `status: 已提炼`（原料文件本身保留不删除）。

- **方式 A**：仅标记用户指定的那一条原料
- **方式 B**：标记本次扫描并通过提炼的所有原料

### 6. 归档原料

提炼完成后，将原料从原目录移动到归档区，实现物理隔离：

**归档路径规则**：
```
99-Archive/原料备份/{YYYY-MM}/{原文件名}
```

**执行动作**：
1. 读取原料 YAML Frontmatter 中的 `date` 字段，提取年月（如 `2026-04-25` → `2026-04`）
2. 若 `date` 不存在，使用当前年月
3. 创建归档目录（如不存在）：`99-Archive/原料备份/2026-04/`
4. 在原料 Frontmatter 末尾追加归档字段：
   ```yaml
   archived: true
   archive_date: "2026-04-26"
   original_path: "01-Work-工作记录/XX项目/会话记录/会话记录-YYYY-MM-DD-xxx.md"
   ```
5. 移动原料文件到归档目录

**source 字段处理**：
- 草稿文件中的 `source` 字段保持原值不变
- 通过 `original_path` 可追溯原料最初来源位置

**边界情况**：
- 若原料无 YAML Frontmatter（如外部导入的纯 Markdown），直接在文件开头插入完整 Frontmatter 块（含 `archived` / `archive_date` / `original_path`），再移动
- 若归档目录已存在同名文件，追加序号去重（如 `-1`）

### 7. 输出提炼摘要

向用户汇报：
- 处理了多少条原料
- 提取了多少条候选资产
- 候选资产列表及建议归属目录
- 提醒用户前往 `待提炼/` 目录审阅

> 📌 **维护提醒**：如提取出新资产，别忘了同步更新 `30-Skills/README.md` 或 `10-Prompts/README.md` 的资产索引

---

## 核心规则（不可违反）

1. **只产出草稿，不直接写入资产区** — 所有 LLM 提取结果必须进入 `待提炼/`，经人工确认后才可迁移；
2. **资产来源可追溯** — 每条草稿必须标注 `source`（来源原料文件路径）；
3. **原料不删除** — 提炼后原料保留，仅更新 `status` 标签；
4. **支持空产出** — 如原料质量不足，允许返回空数组，不强行凑数；
5. **异常回退** — LLM 调用失败时保留原始原料，向用户汇报错误。

## 使用示例

**用户**: "提炼这份会话记录 @01-Work-工作记录/XX项目/会话记录/xxx.md"

**执行**:
1. 读取指定原料文件；
2. 调用 LLM 分析并提取资产；
3. 渲染草稿并保存到 `00-Inbox-收件箱/待提炼/`；
4. 更新原料状态为 `已提炼`；
5. 向用户汇报提炼结果。
