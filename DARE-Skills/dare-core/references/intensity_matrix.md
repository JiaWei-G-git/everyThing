# 全阶段全强度配置矩阵

本文档定义D.A.R.E.框架中四个审查阶段（REQ/ARCH/CODE/TEST）在五个强度级别（Lv.1-Lv.5）下的详细配置参数。

---

## 强度级别定义

### Lv.1 温和模式（Gentle）

| 属性 | 配置值 |
|------|--------|
| 语言风格 | "建议考虑..." / "或许可以思考..." |
| 判定标准 | 仅标记明显问题 |
| 阻塞性 | 不阻塞 |
| Devil挑战数量 | 1-2个 |
| 证据要求 | 引用即可，无需详细论证 |
| 适用场景 | 快速检查、学习目的、初步想法验证 |

**Prompt前缀模板：**
```
请以温和、建设性的语气进行审查。你的目标是帮助改进方案，而非挑错。
每个建议使用鼓励性语言，重点关注最明显的问题。
避免过于苛刻的措辞，优先考虑合作而非对抗。
```

---

### Lv.2 标准模式（Standard）

| 属性 | 配置值 |
|------|--------|
| 语言风格 | "此方案存在以下缺陷..." / "需要说明理由..." |
| 判定标准 | 要求对中等风险问题提供解释 |
| 阻塞性 | 不阻塞，标记警告 |
| Devil挑战数量 | 3-5个 |
| 证据要求 | 需引用具体内容并简要说明 |
| 适用场景 | 日常PR审查、常规文档审查 |

**Prompt前缀模板：**
```
请以标准审查语气进行审查。客观指出方案中的缺陷和风险点，
要求对每个被标记的问题提供简要说明。区分问题的严重级别，
对高风险问题要求额外解释。使用专业但直接的措辞。
```

---

### Lv.3 严格模式（Strict）

| 属性 | 配置值 |
|------|--------|
| 语言风格 | "必须证明该方案在X场景下的可行性" |
| 判定标准 | 对高风险问题要求穷举反例或提供替代方案 |
| 阻塞性 | 阻塞高风险通过 |
| Devil挑战数量 | 5-8个 |
| 证据要求 | 必须引用具体证据并穷举反例 |
| 适用场景 | 关键模块审查、架构评审、安全审查 |

**Prompt前缀模板：**
```
请以严格审查语气进行审查。对每个高风险问题，要求提供：
1. 具体的反例场景
2. 替代方案对比
3. 风险缓解措施
如果不满足以上要求，Issue将被标记为阻塞项。
语言直接有力，不接受模糊解释。
```

---

### Lv.4 激进模式（Aggressive）

| 属性 | 配置值 |
|------|--------|
| 语言风格 | "该设计在以下维度存在致命缺陷..." |
| 判定标准 | 多维度同步攻击，要求逐一回应 |
| 阻塞性 | 阻塞直到致命问题解决 |
| Devil挑战数量 | 8-12个 |
| 证据要求 | 多维度证据链，要求逐一回应 |
| 适用场景 | 关键架构决策、事故复盘、安全审计 |

**Prompt前缀模板：**
```
请以激进审查语气进行审查。从以下维度同步攻击方案：
- 安全性：攻击面和漏洞
- 性能：瓶颈和资源消耗
- 可靠性：容错和降级能力
- 可维护性：复杂度和耦合
- 兼容性：接口和版本兼容性

对每个"致命缺陷"，要求逐一回应并提供修复证据。
在致命问题解决前，审查不通过。
```

---

### Lv.5 极端模式（Extreme）

| 属性 | 配置值 |
|------|--------|
| 语言风格 | "零容忍：发现以下不可接受的风险..." |
| 判定标准 | 零漏洞容忍，全量攻击面覆盖 |
| 阻塞性 | 强制人工介入 |
| Devil挑战数量 | 全量扫描，无上限 |
| 证据要求 | 全覆盖验证，零容错 |
| 适用场景 | 最高安全级别、合规审计、核心基础设施 |

**Prompt前缀模板：**
```
以零容忍态度进行全面审查。任何风险，无论多小，都必须标记。
覆盖全部攻击面，包括但不限于：
- 所有输入验证点
- 所有权限检查点
- 所有错误处理路径
- 所有并发场景
- 所有边界条件
- 所有依赖项的安全状态

发现任何不可接受的风险时，立即标记并强制要求人工介入。
```

## 全阶段 × 全强度配置矩阵

### REQ（需求阶段）

| 强度 | 审查焦点 | Devil关注点 | 判定标准 | 默认启用 |
|------|----------|-------------|----------|----------|
| Lv.1 | 需求完整性 | 明显缺失的需求项 | 标记明显遗漏 | 否 |
| Lv.2 | 需求一致性 | 矛盾需求、模糊定义 | 要求澄清模糊项 | **是（默认）** |
| Lv.3 | 需求可行性 | 技术可行性、资源约束 | 要求可行性证明 | 否 |
| Lv.4 | 需求风险面 | 业务风险、合规风险 | 多维度风险评估 | 否 |
| Lv.5 | 需求全覆盖 | 攻击面全扫描 | 零遗漏容忍 | 否 |

### ARCH（架构阶段）

| 强度 | 审查焦点 | Devil关注点 | 判定标准 | 默认启用 |
|------|----------|-------------|----------|----------|
| Lv.1 | 架构清晰度 | 明显不合理的设计 | 标记明显问题 | 否 |
| Lv.2 | 架构一致性 | 与标准实践的一致性 | 要求解释偏差 | 否 |
| Lv.3 | 架构健壮性 | 扩展性、容错性、性能 | 要求反例和替代方案 | **是（默认）** |
| Lv.4 | 架构风险面 | 多维度架构风险 | 阻塞致命缺陷 | 否 |
| Lv.5 | 架构全覆盖 | 全量架构假设验证 | 零假设容忍 | 否 |

### CODE（代码阶段）

| 强度 | 审查焦点 | Devil关注点 | 判定标准 | 默认启用 |
|------|----------|-------------|----------|----------|
| Lv.1 | 代码规范 | 明显违规 | 标记明显问题 | 否 |
| Lv.2 | 代码质量 | 异味、重复、复杂度 | 标记警告 | 否 |
| Lv.3 | 代码安全性 | 漏洞、注入、越权 | 阻塞高风险 | **是（默认）** |
| Lv.4 | 代码健壮性 | 边界条件、并发、错误处理 | 阻塞致命缺陷 | 否 |
| Lv.5 | 代码全覆盖 | 全路径覆盖、全攻击面 | 强制人工介入 | 否 |

### TEST（测试阶段）

| 强度 | 审查焦点 | Devil关注点 | 判定标准 | 默认启用 |
|------|----------|-------------|----------|----------|
| Lv.1 | 测试存在性 | 明显缺失的测试 | 标记遗漏 | 否 |
| Lv.2 | 测试充分性 | 覆盖率、边界条件 | 标记不足 | **是（默认）** |
| Lv.3 | 测试有效性 | 测试质量、Mock合理性 | 阻塞无效测试 | 否 |
| Lv.4 | 测试完备性 | 异常路径、并发场景 | 阻塞致命遗漏 | 否 |
| Lv.5 | 测试全覆盖 | 全路径、全攻击面 | 强制人工介入 | 否 |

## 强度动态调整规则

审查过程中可根据以下规则动态调整强度：

### 升级触发条件
1. 发现严重安全漏洞 → 自动升级至Lv.4
2. 发现数据完整性风险 → 自动升级至Lv.4
3. 涉及用户隐私处理 → 最低Lv.3
4. 涉及资金交易 → 最低Lv.4

### 降级触发条件
1. 连续3个Issue被Judge驳回 → 可考虑降级
2. 审查对象为非核心模块 → 可酌情降级
3. 时间压力极大 → 与用户确认后降级

### 阶段间强度继承
```
REQ Confirmed Issue → ARCH 自动至少Lv.3
ARCH Confirmed Issue → CODE 自动至少Lv.4
CODE Confirmed Issue → TEST 自动至少Lv.3
```

## 强度-成本对照表

| 强度 | 平均Token消耗 | 平均响应时间 | 每Issue审查成本 |
|------|-------------|-------------|----------------|
| Lv.1 | 1x | 15-30s | 低 |
| Lv.2 | 2x | 30-60s | 中低 |
| Lv.3 | 4x | 1-3min | 中 |
| Lv.4 | 8x | 3-8min | 高 |
| Lv.5 | 16x | 8-20min | 极高 |

**建议：** 大多数日常审查使用Lv.2-3，仅在关键路径和核心模块使用Lv.4-5。
