# 编排模式详细说明

本文档详细描述D.A.R.E.框架的三种编排模式：单Agent自对抗、双Agent辩论和多Agent评审委员会。

---

## 模式A：单Agent自对抗（Single-Agent Self-Adversarial）

### 概述
单个Agent同时扮演Devil和Advocate两个角色，通过内部自问自答完成审查。成本低、速度快，适用于快速扫描场景。

### 参与角色
- 1个Agent（角色切换：Devil ↔ Advocate）

### 执行流程

```
┌─────────────────────────────────────────────┐
│           输入: 待审查工件                     │
└──────────────────┬──────────────────────────┘
                   │
                   ▼
┌─────────────────────────────────────────────┐
│  Round 1: Devil模式 — 提出3-5个核心质疑       │
│  Prompt: "作为Devil，列出该方案的主要缺陷..."  │
└──────────────────┬──────────────────────────┘
                   │
                   ▼
┌─────────────────────────────────────────────┐
│  Round 1: Advocate模式 — 回应质疑            │
│  Prompt: "作为Advocate，回应以上每个质疑..."   │
└──────────────────┬──────────────────────────┘
                   │
                   ▼
┌─────────────────────────────────────────────┐
│  Round 2: Devil模式 — 追加攻击               │
│  Prompt: "针对回应，追加反例和证据..."         │
└──────────────────┬──────────────────────────┘
                   │
                   ▼
┌─────────────────────────────────────────────┐
│  Round 2: Advocate模式 — 强化辩护或承认       │
│  Prompt: "回应追加攻击，承认或反驳..."         │
└──────────────────┬──────────────────────────┘
                   │
                   ▼
┌─────────────────────────────────────────────┐
│  Judge模式 — 自评打分                        │
│  Prompt: "综合以上辩论，对每个Issue评分..."    │
└──────────────────┬──────────────────────────┘
                   │
                   ▼
┌─────────────────────────────────────────────┐
│           输出: 结构化审查报告                 │
└─────────────────────────────────────────────┘
```

### Prompt模板

**Devil角色Prompt：**
```
你现在是D.A.R.E.框架中的Devil（批判者）。你的任务是对以下方案进行严格审查。

待审查内容：
{input_artifact}

要求：
1. 列出3-5个核心质疑点
2. 每个质疑必须引用具体内容（行号、组件名、需求ID等）
3. 每个质疑必须附带建设性建议
4. 当前强度等级：Lv.{intensity}

请以结构化格式输出质疑清单。
```

**Advocate角色Prompt：**
```
你现在是D.A.R.E.框架中的Advocate（维护者）。以下是你需要辩护的方案，以及Devil提出的质疑。

方案内容：
{input_artifact}

Devil的质疑：
{devil_challenges}

要求：
1. 对每个质疑正面回应
2. 提供数据或先例支持辩护
3. 无法辩护时承认问题并提出修改建议
4. 不捍卫明显错误的决策

请以结构化格式输出回应。
```

### 优缺点

| 优点 | 缺点 |
|------|------|
| 无需多Agent协调，延迟最低 | 缺乏真正的对抗张力 |
| 成本低，适合高频使用 | 可能因Confirmation Bias遗漏盲点 |
| 实现简单，易于集成 | 自我辩论深度有限 |
| 适合快速扫描大量代码 | 复杂问题分析不够深入 |

### 适用场景清单
- [x] 日常代码提交前快速自查
- [x] 批量历史债务初步扫描
- [x] 文档格式和一致性检查
- [x] 简单Bug修复的影响面评估
- [x] 时间紧迫时的快速审查（< 30分钟）

## 模式B：双Agent辩论（Dual-Agent Debate）

### 概述
两个独立Agent分别扮演Devil和Advocate进行多轮辩论，Judge在最终轮介入做裁决。标准模式，平衡成本与深度。

### 参与角色
- Agent 1: Devil（批判者）
- Agent 2: Advocate（维护者）
- Agent 3: Judge（裁决者，仅最终轮介入）

### 执行流程

```
                    ┌──────────┐
     ┌─────────────→│  Devil   │
     │              │ (Agent 1)│
     │              └────┬─────┘
     │                   │ 提出质疑
     │                   ▼
┌────┴─────┐      ┌──────────┐
│  输入工件  │      │ Advocate │
│          │      │ (Agent 2)│←─────────────┐
└────┬─────┘      └────┬─────┘              │
     │                 │ 回应质疑              │
     │                 ▼                     │
     │              Round 1 完成              │
     │                 │                     │
     │                 ▼                     │
     │              Devil追加攻击            │
     │                 │                     │
     │                 ▼                     │
     │              Advocate再次回应          │
     │                 │                     │
     │                 ▼                     │
     │              Round 2 完成              │
     │                 │                     │
     │                 ▼                     │
     │    ┌─────────────────────┐            │
     │    │   Judge介入(Agent 3) │            │
     │    │   评分并判定Issue    │            │
     │    └─────────────────────┘            │
     │                 │                     │
     │                 ▼                     │
     │         检查收敛条件                   │
     │    ┌──────┐  ┌────────┐  ┌──────────┐│
     └───→│共识? │→│最大轮次?│→│无新论点? │┘
          └──┬───┘  └───┬────┘  └────┬─────┘
             │Y         │Y           │Y
             ▼          ▼            ▼
        ┌──────────────────────────────────┐
        │         输出结构化报告            │
        └──────────────────────────────────┘
```

### 对话流程示例

**Round 1 — Devil:**
> "在`UserService.authenticate()`方法第45行，密码哈希使用了MD5算法。这是不可接受的，因为MD5已被证明存在碰撞攻击。建议改用bcrypt或Argon2。"

**Round 1 — Advocate:**
> "感谢您的质疑。但此处MD5仅用于遗留系统的兼容性层，实际密码在`HashService`中已通过bcrypt处理（见第78行）。不过我承认这种双层设计可能混淆维护者，建议添加显式注释说明。"

**Round 2 — Devil:**
> "即使bcrypt在78行处理，如果45行的MD5层被绕过直接调用，仍然存在风险。请提供该MD5函数被调用的完整调用链，并说明如何确保不会单独使用。"

**Round 2 — Advocate:**
> "经过核查，MD5函数仅在`LegacyAuthAdapter`中被调用，且该Adapter已标记为@Deprecated。我同意增加运行时警告和单元测试确保不被单独使用。"

**Round 3 — Judge:**
> **Issue判定：**
> - ISSUE-001: "MD5用于密码相关场景" → **Probable**，存在绕过风险，建议增加防御性代码
> - 综合评分：6.5/10，需要补充措施

### 角色分工建议

| Devil子角色 | 关注点 | 适用场景 |
|-------------|--------|----------|
| Devil-Security | 安全漏洞、攻击面 | 所有涉及敏感数据的代码 |
| Devil-Performance | 性能瓶颈、资源泄漏 | 高并发、大数据量场景 |
| Devil-Maintainability | 代码异味、技术债务 | 长期维护的项目 |

### 优缺点

| 优点 | 缺点 |
|------|------|
| 真正的对抗性思维碰撞 | 需要多Agent协调，延迟较高 |
| 可分工覆盖不同维度 | 成本是模式A的2-3倍 |
| Judge独立裁决保证客观 | 简单问题可能过度分析 |
| 适合大多数标准审查场景 | 需要明确的上下文传递机制 |

### 适用场景清单
- [x] 日常Pull Request审查
- [x] 中等复杂度功能评审
- [x] 跨模块接口设计审查
- [x] 标准安全扫描
- [x] 技术方案选型评估

## 模式C：多Agent评审委员会（Multi-Agent Review Board）

### 概述
3-5个Agent组成评审委员会，Devil组从多维度同时攻击，Advocate组分工辩护，Judge综合裁决。最高成本，最深发现深度。

### 参与角色
- Devil组：2-3个Agent，分别覆盖安全、性能、可维护性等维度
- Advocate组：1-2个Agent，分工辩护不同模块
- Judge：1个Agent，综合裁决

### 执行流程

```
                    ┌─────────────────────────────────────┐
                    │          输入工件 + 上下文              │
                    └───────────────┬─────────────────────┘
                                    │
              ┌─────────────────────┼─────────────────────┐
              │                     │                     │
              ▼                     ▼                     ▼
       ┌─────────────┐      ┌─────────────┐      ┌─────────────┐
       │ Devil-Sec   │      │ Devil-Perf  │      │ Devil-Maint │
       │ (安全维度)   │      │ (性能维度)   │      │ (维护维度)   │
       └──────┬──────┘      └──────┬──────┘      └──────┬──────┘
              │                     │                     │
              └─────────────────────┼─────────────────────┘
                                    │
                                    ▼
                    ┌─────────────────────────────────────┐
                    │         多维度质疑汇总                │
                    │    (去重、聚类、优先级排序)            │
                    └─────────────────┬───────────────────┘
                                      │
              ┌───────────────────────┼───────────────────────┐
              │                       │                       │
              ▼                       ▼                       ▼
       ┌─────────────┐        ┌─────────────┐        ┌─────────────┐
       │ Advocate-A  │        │ Advocate-B  │        │ Advocate-C  │
       │ (模块A辩护)  │        │ (模块B辩护)  │        │ (通用辩护)   │
       └──────┬──────┘        └──────┬──────┘        └──────┬──────┘
              │                       │                       │
              └───────────────────────┼───────────────────────┘
                                    │
                                    ▼
                    ┌─────────────────────────────────────┐
                    │      多轮深度辩论 (2-3轮)            │
                    └─────────────────┬───────────────────┘
                                      │
                                      ▼
                    ┌─────────────────────────────────────┐
                    │           Judge (裁决者)              │
                    │    综合评估、交叉验证、最终判定         │
                    └─────────────────┬───────────────────┘
                                      │
                                      ▼
                    ┌─────────────────────────────────────┐
                    │         结构化审查报告                │
                    │    含跨维度风险评估矩阵               │
                    └─────────────────────────────────────┘
```

### 多维度攻击矩阵

| 维度 | Devil角色名 | 攻击范围 | 典型质疑 |
|------|------------|----------|----------|
| 安全性 | Devil-Sec | 认证/授权/数据/传输安全 | "此接口未做速率限制，存在暴力破解风险" |
| 性能 | Devil-Perf | 时间复杂度/空间复杂度/并发 | "该查询在数据量超过100万时响应时间会超2s" |
| 可维护性 | Devil-Maint | 代码复杂度/耦合度/文档 | "该函数 cyclomatic complexity = 25，需要拆分" |
| 可靠性 | Devil-Rel | 容错/降级/幂等/一致性 | "该操作非幂等，网络超时重试会导致重复数据" |
| 兼容性 | Devil-Compat | 接口兼容/版本兼容/平台兼容 | "此修改破坏v1 API的向后兼容性" |

### 协调协议

多Agent之间的协调通过以下机制实现：

1. **共享上下文板（Shared Context Board）：** 所有Agent可读写的问题跟踪板
2. **轮次同步信号：** 每轮结束时由Judge发送同步信号
3. **冲突去重规则：** 相同根因的Issue自动合并

### 优缺点

| 优点 | 缺点 |
|------|------|
| 多维度交叉验证，发现深度最大 | 成本高，延迟大 |
| 适合关键决策的高置信度需求 | 需要复杂的Agent协调机制 |
| 可并行攻击不同维度 | 可能产生过多Issue需要筛选 |
| 最接近真实专家评审会体验 | 配置和管理复杂度最高 |

- [x] 生产事故根因分析
- [x] 安全审计和合规审查
- [x] 重大版本发布的全面评估
- [x] 核心基础设施变更评审
- [x] 涉及多个团队的大型项目评审
