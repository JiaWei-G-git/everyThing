# Judge评分操作指南

本文档为D.A.R.E.框架中Judge（裁决者）角色的详细操作指南，确保评分的一致性和客观性。

---

## Judge角色定位

Judge是D.A.R.E.框架中的中立裁决者，不参与辩论过程，仅在辩论收敛阶段介入。Judge基于辩论双方提供的证据进行客观评分，不受任何一方立场影响。

**核心原则：**
- 基于证据权重而非角色立场做判定
- 对每个争议给出明确结论
- 生成结构化评分和优先级排序
- 在证据不足时要求补充而非臆断

---

## 评分流程

### Step 1: 证据收集与整理

在评分前，Judge需要完成以下准备工作：

1. **阅读辩论完整记录** — 确保了解每个Issue的攻防全过程
2. **提取关键证据** — 整理双方引用的行号、需求ID、架构组件名、数据等
3. **识别未回应的质疑** — 标记Advocate未正面回应的Devil质疑
4. **识别无效攻击** — 标记Devil无证据支持的质疑

### Step 2: 逐维度评分

对每个Issue，按以下五个维度分别评分（0-10分）：

#### 维度1: 证据充分性（权重30%）

| 分数 | 标准 | 示例 |
|------|------|------|
| 9-10 | 证据具体、可验证、有引用 | "第45行使用了MD5，NIST已不建议用于密码哈希（SP 800-132）" |
| 7-8 | 证据较具体，但缺少部分引用 | "使用了不安全的哈希算法"（未指明具体算法和引用） |
| 5-6 | 证据模糊，仅有一般性描述 | "这里可能有安全问题" |
| 3-4 | 几乎无证据，仅有主观判断 | "我觉得这个设计不好" |
| 0-2 | 无证据或证据明显错误 | "这个API不存在"（实际上存在） |

#### 维度2: 影响严重性（权重25%）

| 分数 | 标准 | 示例 |
|------|------|------|
| 9-10 | 灾难性影响：数据丢失、安全泄露、系统崩溃 | SQL注入导致全库暴露 |
| 7-8 | 严重影响：功能不可用、性能严重下降 | 核心交易接口超时 |
| 5-6 | 中等影响：部分功能受限、用户体验差 | 非核心功能响应慢 |
| 3-4 | 轻微影响：边缘情况、可绕过 | 日志格式不规范 |
| 0-2 | 几乎无影响：纯 cosmetic | 注释拼写错误 |

#### 维度3: 可修复性（权重20%）

| 分数 | 标准 | 示例 |
|------|------|------|
| 9-10 | 修复方案明确且工作量小 | 替换MD5为bcrypt（单行修改） |
| 7-8 | 修复方案明确但工作量中等 | 重构一个过长的函数 |
| 5-6 | 有修复方案但需权衡 trade-off | 缓存策略调整 |
| 3-4 | 修复方案不明确或工作量大 | 核心架构模式变更 |
| 0-2 | 几乎不可修复或修复成本极高 | 更换数据库类型 |

#### 维度4: 发生概率（权重15%）

| 分数 | 标准 | 示例 |
|------|------|------|
| 9-10 | 几乎必然发生 | 未做输入校验的接口被攻击 |
| 7-8 | 很可能发生 | 高并发场景下竞态条件触发 |
| 5-6 | 有可能发生 | 特定网络条件下的超时 |
| 3-4 | 很少发生 | 需要特定攻击者知识的利用 |
| 0-2 | 几乎不可能发生 | 需要物理接触的攻击 |

#### 维度5: 范围广度（权重10%）

| 分数 | 标准 | 示例 |
|------|------|------|
| 9-10 | 影响整个系统或多个核心模块 | 认证中间件缺陷影响所有API |
| 7-8 | 影响单个核心模块 | 订单处理模块 |
| 5-6 | 影响多个非核心模块 | 辅助功能模块 |
| 3-4 | 影响单个非核心模块 | 日志模块 |
| 0-2 | 仅影响局部代码段 | 单个工具函数 |

### Step 3: 计算加权得分

```
加权得分 = 证据充分性 × 0.30 +
          影响严重性 × 0.25 +
          可修复性   × 0.20 +
          发生概率   × 0.15 +
          范围广度   × 0.10
```

### Step 4: 确定Issue状态

根据加权得分确定最终状态：

| 加权得分范围 | 判定状态 | 说明 |
|-------------|----------|------|
| 8.0 - 10.0 | **Confirmed** | 证据充分，必须修复 |
| 6.0 - 7.9 | **Probable** | 有较大可能性，建议修复 |
| 4.0 - 5.9 | **Disputed** | 证据不足，存在争议 |
| 0.0 - 3.9 | **Rejected** | 证据不支持，不成立 |

## 评分操作检查清单

### 评分前检查
- [ ] 已完整阅读所有辩论轮次的记录
- [ ] 已识别每个Issue的原始Devil质疑
- [ ] 已整理Advocate的所有回应
- [ ] 已标记未回应的质疑点
- [ ] 已标记无证据的攻击

### 评分中检查
- [ ] 每个维度的评分都有具体理由
- [ ] 引用了具体的证据支持评分
- [ ] 加权计算正确无误
- [ ] 状态判定符合分数区间

### 评分后检查
- [ ] 所有Issue都有明确的判定状态
- [ ] Confirmed Issue有明确的修复建议
- [ ] Disputed Issue有补充证据的要求
- [ ] 评分报告已按优先级排序

## 典型评分案例

### 案例1：Confirmed（SQL注入漏洞）

**Devil质疑：**
"`UserDAO.findByName()`第23行直接拼接SQL字符串，未使用PreparedStatement。攻击者可通过构造`name=' OR '1'='1`绕过认证。参考OWASP Top 10 #1。"

**Advocate回应：**
"承认该问题。此代码为早期遗留实现，确实未做参数化查询。建议立即替换为PreparedStatement并添加输入校验。"

**Judge评分：**
| 维度 | 分数 | 理由 |
|------|------|------|
| 证据充分性 | 10/10 | 具体行号、攻击Payload、OWASP引用 |
| 影响严重性 | 10/10 | SQL注入可导致全库泄露 |
| 可修复性 | 9/10 | 替换为PreparedStatement即可 |
| 发生概率 | 10/10 | 可直接利用 |
| 范围广度 | 8/10 | 影响所有使用该DAO的接口 |

**加权得分：** 10×0.30 + 10×0.25 + 9×0.20 + 10×0.15 + 8×0.10 = 9.5

**判定：** Confirmed（必须立即修复）

---

### 案例2：Probable（缓存一致性问题）

**Devil质疑：**
"`CacheService.update()`更新缓存后未通知其他节点的本地缓存，在集群部署下可能导致数据不一致。"

**Advocate回应：**
"当前部署为单节点，缓存一致性不是问题。但未来若扩展为集群部署，确实需要考虑。建议添加分布式缓存支持（Redis）。"

**Judge评分：**
| 维度 | 分数 | 理由 |
|------|------|------|
| 证据充分性 | 7/10 | 指出了具体函数和场景，但未提供实际不一致的证据 |
| 影响严重性 | 7/10 | 数据不一致可能导致业务错误 |
| 可修复性 | 6/10 | 引入Redis有明确的方案但有一定工作量 |
| 发生概率 | 5/10 | 当前单节点部署不会发生，未来可能 |
| 范围广度 | 6/10 | 影响缓存相关的所有数据 |

**加权得分：** 7×0.30 + 7×0.25 + 6×0.20 + 5×0.15 + 6×0.10 = 6.4

**判定：** Probable（建议纳入技术债务计划）

---

### 案例3：Disputed（过度设计质疑）

**Devil质疑：**
"引入Event Sourcing模式对于当前仅需要CRUD操作的订单模块是过度设计，增加了不必要的复杂度。"

**Advocate回应：**
"Event Sourcing虽然增加了复杂度，但为后续审计和数据分析需求提供了基础。产品路线图Q3有数据分析需求。"

**Judge评分：**
| 维度 | 分数 | 理由 |
|------|------|------|
| 证据充分性 | 5/10 | Devil未证明复杂度不可接受，Advocate的Q3需求未提供文档引用 |
| 影响严重性 | 5/10 | 双方论点均有道理，取决于实际执行 |
| 可修复性 | 5/10 | 可简化但会失去某些能力 |
| 发生概率 | 4/10 | 过度设计的负面影响不一定发生 |
| 范围广度 | 5/10 | 影响订单模块的设计方向 |

**加权得分：** 5×0.30 + 5×0.25 + 5×0.20 + 4×0.15 + 5×0.10 = 4.85

**判定：** Disputed（需要补充证据：产品路线图文档、团队维护能力评估）

---

### 案例4：Rejected（无根据的质疑）

**Devil质疑：**
"使用Go语言编写该服务是不合适的，应该使用Java。"

**Advocate回应：**
"选择Go是因为该服务需要高并发处理，Go的goroutine模型在该场景下性能更优。团队也有Go开发经验。"

**Judge评分：**
| 维度 | 分数 | 理由 |
|------|------|------|
| 证据充分性 | 2/10 | Devil未提供任何具体证据，仅为语言偏好 |
| 影响严重性 | 3/10 | 即使有影响，也非技术问题 |
| 可修复性 | 3/10 | 重写服务的成本极高 |
| 发生概率 | 3/10 | 未证明Go不适合该场景 |
| 范围广度 | 4/10 | 涉及整个服务 |

**加权得分：** 2×0.30 + 3×0.25 + 3×0.20 + 3×0.15 + 4×0.10 = 2.9

**判定：** Rejected（Devil的质疑缺乏技术依据）

## 常见评分偏差及避免方法

| 偏差类型 | 表现 | 避免方法 |
|----------|------|----------|
| 权威偏差 | 因Devil引用了知名来源而高估证据分 | 评估证据与当前Issue的相关性而非来源知名度 |
| 近期偏差 | 因Advocate回应在最后而给予更高权重 | 按时间顺序整理双方论点后统一评估 |
| 确认偏差 | Judge倾向于支持自己预先认同的观点 | 明确列出每个维度的评分依据 |
| 数量偏差 | 因Devil提出更多质疑而倾向于Devil | 评估每个Issue的独立质量而非数量 |
| 语气偏差 | 因Advocate态度诚恳而降低评分标准 | 仅基于证据和事实评分，忽略语气因素 |
