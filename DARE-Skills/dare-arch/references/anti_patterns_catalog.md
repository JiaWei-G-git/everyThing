# 常见架构反模式目录

审查过程中重点识别以下反模式。每个反模式包含定义、症状、后果和修复策略。

## 服务划分反模式

### AP-001: 上帝服务 (God Service)

**定义**: 一个服务承担了过多的职责，包含了本应属于多个服务的业务逻辑。

**症状**:
- 服务代码量超过 5 万行
- 部署频率显著低于其他服务（因为改动影响面太大）
- 团队规模过大（超过 15 人）导致协作困难
- 数据库表数量超过 50 张

**后果**:
- 任何小改动都需要全量回归测试
- 无法独立扩展单个功能模块
- 技术栈升级困难
- 团队扩张瓶颈

**修复策略**:
- 识别领域边界，按 DDD 战略设计拆分
- 优先提取变更最频繁的功能模块
- 使用绞杀者模式逐步迁移
- 建立清晰的领域事件契约

### AP-002: 纳米服务 (Nanoprocess / Nanoservice)

**定义**: 服务拆分过度，每个服务只承担极小的职责，导致通信成本超过业务逻辑成本。

**症状**:
- 每个服务少于 500 行代码
- 单次用户请求跨越 10 个以上服务
- 服务间调用链深度超过 5 层
- 部署单元数量超过团队人数的 3 倍

**后果**:
- 网络延迟成为主要性能瓶颈
- 故障排查极其困难
- 事务一致性难以保证
- 运维复杂度爆炸

**修复策略**:
- 合并高度耦合的服务
- 按业务边界重新聚合
- 引入 BFF 层减少服务间调用
- 评估是否应回归单体或模块化单体

### AP-003: 循环依赖 (Circular Dependency)

**定义**: 两个或多个服务相互依赖，形成闭环调用链。

**症状**:
- 服务 A 调用服务 B，服务 B 又通过直接或间接路径调用服务 A
- 无法独立部署某个服务（必须先部署其依赖）
- 启动时出现超时或死锁
- 级联故障时无法定位根因

**后果**:
- 无法独立部署和扩展
- 级联故障风险极高
- 代码重构困难
- 团队间强耦合

**修复策略**:
- 引入异步消息解耦
- 提取共享库或公共服务
- 重构为单向依赖
- 使用事件溯源模式

## 数据管理反模式

### AP-004: 分布式单体数据库 (Distributed Monolith Database)

**定义**: 微服务架构中多个服务共享同一个数据库，服务间通过数据库直接耦合。

**症状**:
- 多个服务直接访问同一张表
- 数据库 schema 变更需要协调多个团队
- 无法独立对某个服务进行数据库优化
- 存在跨服务的事务（XA/2PC）

**后果**:
- 数据库成为扩展瓶颈
- schema 变更风险高
- 数据所有权不清晰
- 无法独立演进数据模型

**修复策略**:
- 每个服务独立数据库（Database-per-Service）
- 通过 API 暴露数据访问
- 使用 CQRS 分离读写
- 建立数据同步机制

### AP-005: API 组合地狱 (API Composition Hell)

**定义**: 前端或网关需要调用大量后端 API 来组装一个完整的页面或响应。

**症状**:
- 单个页面发起 20+ API 请求
- 网关层代码膨胀，充满聚合逻辑
- 移动端性能差（网络往返多）
- 重复的数据聚合逻辑散落在多个客户端

**后果**:
- 响应时间不可控
- 客户端代码复杂
- 网络资源浪费
- 一致性问题

**修复策略**:
- 引入 BFF（Backend for Frontend）模式
- 使用 GraphQL 或类似聚合查询方案
- 建立领域聚合服务
- 缓存频繁访问的组合数据

### AP-006: 缓存滥用 (Cache Abuse)

**定义**: 使用缓存掩盖架构或数据库设计缺陷，而非解决真正的性能问题。

**症状**:
- 缓存命中率低于 50%
- 大量缓存穿透和雪崩问题
- 数据一致性频繁出问题
- 数据库查询未经优化直接加缓存

**后果**:
- 缓存成为系统脆弱点
- 数据不一致引发业务问题
- 缓存故障时数据库直接被压垮
- 调试困难（缓存和数据库状态不一致）

**修复策略**:
- 优化数据库查询和索引
- 使用合适的缓存策略（Cache-Aside / Write-Through / Write-Behind）
- 实现缓存预热和降级机制
- 建立缓存一致性监控

## 通信反模式

### AP-007: 同步调用链过长 (Long Synchronous Chain)

**定义**: 服务间同步调用链过长，导致响应时间不可控和可用性降低。

**症状**:
- 同步调用链超过 4 个服务
- P99 响应时间超过 P50 的 10 倍
- 上游服务超时频繁
- 整体可用性等于各服务可用性的乘积

**后果**:
- 响应时间尾延迟严重
- 可用性随链路增长指数下降
- 线程池耗尽风险
- 用户体验不可控

**修复策略**:
- 异步化非关键路径
- 使用事件驱动架构
- 引入 Saga 模式管理长流程
- 并行化独立调用

### AP-008: 重试风暴 (Retry Storm)

**定义**: 多个客户端同时重试失败的请求，导致故障服务承受数倍流量冲击。

**症状**:
- 服务故障时流量不降反升
- 重试没有退避策略或退避时间相同
- 幂等性处理不完善导致数据重复
- 级联服务全部重试同一请求

**后果**:
- 小故障演变为全面雪崩
- 恢复时间延长
- 数据一致性风险
- 资源耗尽

**修复策略**:
- 实现指数退避 + 抖动
- 断路器模式限制重试
- 确保操作幂等性
- 分层重试（只在合适层级重试）

## 安全反模式

### AP-009: 隐式信任网络 (Implicit Trust Network)

**定义**: 架构假设内部网络和服务是可信的，未实施内部安全防护。

**症状**:
- 服务间通信无认证
- 内部 API 无权限控制
- 网络边界仅有简单防火墙
- 缺少内部流量加密

**后果**:
- 内部威胁风险极高
- 单点被攻破后横向移动容易
- 不满足零信任安全要求
- 合规审计不通过

**修复策略**:
- 实施 mTLS 服务间认证
- 每个服务独立认证授权
- 采用零信任架构
- 内部流量审计和监控

### AP-010: 秘密扩散 (Secret Sprawl)

**定义**: 密钥、密码、Token 等敏感信息散落在代码、配置和日志中。

**症状**:
- 配置文件包含明文密码
- 代码仓库中有 API Key
- 日志输出包含敏感信息
- 环境变量管理混乱

**后果**:
- 密钥泄露风险极高
- 轮换密钥极其困难
- 安全事件影响范围大
- 合规问题

**修复策略**:
- 使用密钥管理系统（Vault/AWS KMS 等）
- 密钥与代码分离
- 实施密钥自动轮换
- 日志脱敏

## 运维反模式

### AP-011: 雪花服务器 (Snowflake Server)

**定义**: 服务器配置无法通过代码重现，每次部署都可能产生差异。

**症状**:
- 手动修改服务器配置
- 同一角色服务器配置不一致
- 无法快速创建相同环境
- 灾难恢复时间极长

**后果**:
- 环境问题难以复现
- 部署风险高
- 扩展困难
- 灾难恢复不可靠

**修复策略**:
- 基础设施即代码（IaC）
- 不可变基础设施
- 配置管理工具（Ansible/Puppet/Chef）
- 容器化部署

### AP-012: 监控盲区 (Monitoring Blind Spot)

**定义**: 关键组件或指标未被监控，故障发生时无法及时发现和诊断。

**症状**:
- 监控覆盖率低于 80%
- 有服务未接入集中监控
- 缺少业务指标监控
- 只有基础资源监控，无应用层监控

**后果**:
- 故障发现延迟
- 根因分析困难
- 无法预测性维护
- SLA 无法保障

**修复策略**:
- 建立 RED 指标（Rate/Errors/Duration）监控
- 实施分布式链路追踪
- 统一日志收集和分析
- 设置关键业务指标告警

## 扩展性反模式

### AP-013: 共享瓶颈资源 (Shared Bottleneck Resource)

**定义**: 多个服务或组件竞争同一资源，该资源成为系统扩展的硬瓶颈。

**症状**:
- 所有服务共享同一个消息队列集群
- 全局锁或分布式锁使用广泛
- 单个数据库实例服务所有写请求
- 共享存储成为并发瓶颈

**后果**:
- 无法独立扩展竞争资源的服务
- 资源争用导致性能下降
- 故障影响范围大
- 扩展成本非线性增长

**修复策略**:
- 资源隔离和分区
- 消除全局锁，使用本地锁或乐观锁
- 数据分片和读写分离
- 引入背压机制

### AP-014: 状态污染 (State Pollution)

**定义**: 无状态服务中引入状态依赖，导致无法自由扩展和恢复。

**症状**:
- 服务将 session 存储在内存中
- 依赖本地文件系统存储数据
- 粘性会话（Sticky Session）强制路由
- 定时任务假设单实例运行

**后果**:
- 无法水平扩展
- 实例故障时状态丢失
- 负载均衡失效
- 弹性伸缩受限

**修复策略**:
- 使用外部缓存存储 session
- 对象存储替代本地文件
- JWT 等无状态认证
- 分布式调度框架

## 组织反模式

### AP-015: 康威定律逆行者 (Conway Violator)

**定义**: 组织架构与架构设计不匹配，导致沟通摩擦和决策低效。

**症状**:
- 一个服务由多个团队共同维护
- 团队间沟通成本高于开发成本
- 架构决策需要跨多个部门审批
- 代码所有权不清晰

**后果**:
- 决策缓慢
- 技术债务积累（无人负责偿还）
- 质量问题互相推诿
- 团队士气下降

**修复策略**:
- 按领域重新划分团队边界
- 明确服务所有权
- 建立内部开源模式（ InnerSource ）
- 调整组织架构匹配架构

## 审查检查清单

在架构审查中，对每个反模式检查以下问题：

- [ ] 是否存在服务职责不清晰的情况？（AP-001, AP-002）
- [ ] 服务间依赖是否存在闭环？（AP-003）
- [ ] 数据库访问是否通过服务 API 隔离？（AP-004）
- [ ] 前端是否需要过多后端调用？（AP-005）
- [ ] 缓存策略是否合理且有降级机制？（AP-006）
- [ ] 同步调用链长度是否可控？（AP-007）
- [ ] 重试策略是否包含退避和断路器？（AP-008）
- [ ] 内部通信是否有认证和加密？（AP-009）
- [ ] 密钥管理是否集中和安全？（AP-010）
- [ ] 基础设施是否可代码化和可重现？（AP-011）
- [ ] 监控是否覆盖所有关键组件和指标？（AP-012）
- [ ] 是否存在所有服务竞争的资源？（AP-013）
- [ ] 无状态服务是否真正无状态？（AP-014）
- [ ] 团队结构是否与架构匹配？（AP-015）
