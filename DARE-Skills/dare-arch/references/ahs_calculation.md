# 架构健康度评分（AHS）计算详细说明

## 核心公式

```
AHS = 0.25 * technical_selection + 0.30 * robustness + 0.25 * scalability + 0.20 * tech_debt
```

## 维度权重说明

| 维度 | 权重 | 理由 |
|------|------|------|
| 技术选型 | 25% | 技术选型影响深远但可在一定程度上修正 |
| 架构健壮性 | 30% | 最高权重，因为结构性风险可能导致系统性故障 |
| 扩展性 | 25% | 扩展性限制往往需要对架构进行根本性改造 |
| 技术债务 | 20% | 技术债务可渐进式偿还，权重相对较低 |

## 各维度评分标准

### 技术选型（0-100 分）

| 分数段 | 评级 | 标准 |
|-------|------|------|
| 90-100 | 优秀 | 技术选型有充分论证，已评估至少 3 个替代方案，选择理由清晰，团队技能匹配 |
| 70-89 | 良好 | 技术选型有合理理由，但替代方案评估不够充分或有轻微偏见 |
| 50-69 | 一般 | 技术选型理由薄弱，存在明显的选择偏见，未考虑关键替代方案 |
| 30-49 | 较差 | 技术选型存在严重问题，核心技术栈与需求不匹配 |
| 0-29 | 极差 | 技术选型完全不当，可能导致项目失败 |

### 架构健壮性（0-100 分）

| 分数段 | 评级 | 标准 |
|-------|------|------|
| 90-100 | 优秀 | 无单点故障，有完善的熔断降级策略，故障隔离机制健全，已做混沌工程验证 |
| 70-89 | 良好 | 主要组件有冗余，基本故障处理机制存在，但边界情况覆盖不足 |
| 50-69 | 一般 | 存在已知单点故障，故障处理机制不完整，无降级策略 |
| 30-49 | 较差 | 多处单点故障，无故障隔离，级联失败风险高 |
| 0-29 | 极差 | 架构脆弱，随时可能发生系统性故障 |

### 扩展性（0-100 分）

| 分数段 | 评级 | 标准 |
|-------|------|------|
| 90-100 | 优秀 | 架构支持水平扩展，已验证 10x 增长场景，扩展成本线性或次线性 |
| 70-89 | 良好 | 主要组件可扩展，但部分组件存在扩展上限 |
| 50-69 | 一般 | 扩展需要显著改造，存在明显的扩展瓶颈 |
| 30-49 | 较差 | 扩展极其困难，需要架构重构才能支持增长 |
| 0-29 | 极差 | 架构无法扩展，必须重新设计 |

### 技术债务（0-100 分）

注意：技术债务分数越高表示债务越少，架构越健康。

| 分数段 | 评级 | 标准 |
|-------|------|------|
| 90-100 | 优秀 | 无隐藏技术债务，所有临时方案已标记并有偿还计划，代码质量高 |
| 70-89 | 良好 | 少量技术债务，主要临时方案已标记，偿还计划可行 |
| 50-69 | 一般 | 存在未标记的技术债务，部分反模式已被使用 |
| 30-49 | 较差 | 大量技术债务积累，多个临时方案未标记，维护成本快速上升 |
| 0-29 | 极差 | 技术债务严重到影响开发效率和系统稳定性 |

## 扣分规则

每个维度从 100 分起，根据发现的问题进行扣分：

### Critical 问题
- 技术选型：-30 分
- 架构健壮性：-35 分
- 扩展性：-30 分
- 技术债务：-25 分

### High 问题
- 技术选型：-15 分
- 架构健壮性：-20 分
- 扩展性：-15 分
- 技术债务：-15 分

### Medium 问题
- 技术选型：-8 分
- 架构健壮性：-10 分
- 扩展性：-8 分
- 技术债务：-8 分

### Low 问题
- 技术选型：-3 分
- 架构健壮性：-5 分
- 扩展性：-3 分
- 技术债务：-3 分

### 扣分上限
单个维度的最低得分为 0 分，不出现负分。

## AHS 阈值与行动

| AHS 区间 | 评级 | 行动要求 |
|---------|------|---------|
| 90-100 | 卓越 | 架构优秀，记录最佳实践供团队参考 |
| 80-89 | 良好 | 架构合格，可选性采纳改进建议 |
| 70-79 | 一般 | 架构存在可修复的问题，建议在下一个迭代周期处理 |
| 60-69 | 及格 | 有条件通过，必须修复关键问题并设定明确时限 |
| 50-59 | 较差 | 不通过，必须解决阻塞性问题后方可继续，需要重新评审 |
| 0-49 | 极差 | 不通过，建议暂停开发并进行架构重新设计 |

## 计算示例

### 示例 1：良好架构

```
技术选型: 85/100
架构健壮性: 78/100
扩展性: 82/100
技术债务: 75/100

AHS = 0.25 * 85 + 0.30 * 78 + 0.25 * 82 + 0.20 * 75
    = 21.25 + 23.40 + 20.50 + 15.00
    = 80.15

评级: 通过
```

### 示例 2：有条件通过

```
技术选型: 65/100
架构健壮性: 58/100
扩展性: 72/100
技术债务: 55/100

AHS = 0.25 * 65 + 0.30 * 58 + 0.25 * 72 + 0.20 * 55
    = 16.25 + 17.40 + 18.00 + 11.00
    = 62.65

评级: 有条件通过
必须修复的问题清单（带时限）:
1. [ISSUE-ROB-001] 订单服务单点故障 - 2 周内完成冗余部署
2. [ISSUE-TS-003] 缓存策略未考虑一致性要求 - 1 周内补充方案
```

### 示例 3：不通过

```
技术选型: 45/100
架构健壮性: 35/100
扩展性: 40/100
技术债务: 50/100

AHS = 0.25 * 45 + 0.30 * 35 + 0.25 * 40 + 0.20 * 50
    = 11.25 + 10.50 + 10.00 + 10.00
    = 41.75

评级: 不通过
建议: 暂停当前开发，召开架构重新设计工作坊
```

## Judge-Arch 评分指引

1. **独立评分**: Judge-Arch 应基于 Devil 角色的审查报告独立评分，不受单个 Devil 观点的过度影响
2. **证据优先**: 评分必须有具体证据支撑，避免主观判断
3. **趋势考量**: 考虑架构的演进趋势，当前有债务但有清晰偿还计划的可以适度加分
4. **上下文敏感**: 考虑项目阶段（MVP vs 生产系统），对早期项目适当放宽标准
5. **一致性检查**: 确保四个维度的评分之间逻辑一致，避免出现矛盾
