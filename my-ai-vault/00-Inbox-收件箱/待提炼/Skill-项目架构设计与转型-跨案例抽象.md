# 项目架构设计与转型

> 基于 4 条会话记录跨案例抽象（需求设计 → 目录架构 → 脚手架搭建 → 技术栈转型）

## 触发条件
需要从零设计一个可长期维护的 AI 协作项目结构，或对现有项目的技术栈/架构进行重大调整时。

## 输入 / 输出

| 类型 | 说明 |
|------|------|
| **输入** | 需求描述（多工具兼容、流转流程、Git 管理要求）、现有技术债评估 |
| **输出** | 目录结构设计、命名规范、流转规则、关键决策文档、迁移/清理清单 |

## 通用步骤（基于 ≥3 条记录归纳）

### 1. 定义核心闭环流程
在动手设计目录之前，先用一条主线描述"数据/工作如何流动"：
```
使用 AI → 保存会话 → 提炼资产 → 人工确认 → 沉淀知识库
```
- 所有目录设计都必须服务于这条主线
- 主线中的每个阶段对应至少一个物理目录

### 2. 设计双轴目录结构

**纵向轴（处理流程）**：
```
00-Inbox（快速捕获）
  → 01-Work（按项目聚合）
    → 10~60（提炼为资产）
      → 70-Sharing（团队共享）
        → 99-Archive（历史归档）
```

**横向轴（资产类型）**：
```
10-Prompts      优质提问模板
20-Agents       角色定义与交互流程
30-Skills       原子技能（与 AI 工具共享）
40-MCP          MCP 服务定义
50-Workflows    多步骤工作流
60-Tutorials    工具使用经验与教程
```

### 3. 制定命名规范
- **工具链高频访问的目录**：使用英文命名（如 `10-Prompts`、`30-Skills`）
- **人工浏览为主的目录**：可保留中文注释（如 `00-Inbox-收件箱`）
- 统一使用 **编号前缀**（`00` `01` `10`），在文件管理器中天然排序
- 中文注释直接放在目录名中，比单独的 README 更直观

### 4. 预留中间状态区域
- 在原料层和资产层之间预留 **待提炼/ 待确认** 子目录
- LLM 自动提取的候选资产必须先落入中间区，经人工确认后才迁移到资产层
- 避免半成品资产污染正式的 Prompt/Skill 目录

### 5. 统一来源 + 符号链接
- 将需要被多个 AI 工具共享的目录（如 `30-Skills/`）设为**唯一来源**
- 各 AI 工具通过符号链接引用该目录，而非复制文件
- 原则：**修改一处，全局生效**；版本一致性由 Git 管理

### 6. 输出关键决策文档
将架构设计中的关键决策（及其理由）固化为文档，便于后续团队理解和架构评审：
- 为什么选择这种目录结构而非其他方案
- 命名规范的具体规则及反例
- 流转规则中的硬性约束（如"必须经过人工确认"）

## 特化选项（视情况而定，仅部分记录涉及）

| 选项 | 适用场景 | 来源记录 |
|------|----------|----------|
| **技术栈转型评估** | 现有技术栈出现兼容性问题时，评估替代方案的可行性 | 项目从Python到纯Skill模式转型 |
| **安装方式简化** | 跨操作系统/跨工具的安装脚本维护成本过高时，改为"对 AI 说一句话"的提示词式安装 | 项目从Python到纯Skill模式转型 |
| **子代理并行开发** | Hackathon / MVP 阶段需要快速搭建脚手架时，用子代理并行生成多个模块 | 项目脚手架搭建与Vault初始化 |
| **Git 版本控制设计** | 项目需要多人协作时，设计 `.gitignore` 规则和目录纳入策略 | Vault目录架构设计 |
| **模板独立管理** | 项目包含多种输出模板时，将 `templates/` 与源码分离，便于后续扩展 | 项目脚手架搭建与Vault初始化 |

## 常见误区（多条记录都踩过）

1. **技术栈过早固化，忽视跨工具兼容性**
   - 错误：假设所有用户都能跑 Python 环境、配 `.env`、装依赖
   - 正确：优先选择"纯文本/纯 Skill"方案，降低环境依赖；必须依赖脚本时提供 mock 回退

2. **安装脚本依赖操作系统细节**
   - 错误：提供 `ln -s`（Linux/Mac）或 `New-Item`（Windows）等具体命令
   - 正确：将安装逻辑抽象为"对 AI 说一句话"，由 AI 根据当前环境选择正确命令

3. **目录结构未预留中间状态**
   - 错误：原料直接流入资产层，或 LLM 提取结果无处暂存
   - 正确：强制预留 `待提炼/` 区域，作为"机器产出 → 人工确认"的缓冲区

4. **符号链接设计在转型中丢失**
   - 错误：重构时只关注删除旧代码，忘记保留"唯一来源 + 符号链接"的核心机制
   - 正确：将核心设计原则写入文档的"不可变约束"章节，任何转型都必须保留

5. **文档不同步导致新旧信息矛盾**
   - 错误：只更新代码和 Skill，历史文档（架构文档、PRD、PROJECT_GENERATOR.md）中的技术描述仍保留旧信息
   - 正确：技术栈变更后，全局搜索所有 `.md` 文件中的旧技术引用（如 `python -m`、`pip install`），逐一更新

6. **PROJECT_GENERATOR.md 只列文件名**
   - 错误：生成文档只包含文件列表，重建项目时 AI 无法获知文件内容
   - 正确：PROJECT_GENERATOR.md 必须包含所有关键文件的完整内容，确保任何 AI 拿到文档都能完整重建项目
