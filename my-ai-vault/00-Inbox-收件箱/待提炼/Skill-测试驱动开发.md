# 测试驱动开发

## 触发条件
在实现任何功能或修复错误时，**在编写实现代码之前**强制执行。

## 执行步骤

1. **编写失败测试**
   - 根据需求或 bug 描述，编写一个能暴露当前问题的测试用例
   - 运行测试，确认测试失败（红）
   - 预期结果：测试运行结果明确显示失败，且失败原因与预期一致

2. **编写最小实现**
   - 编写最简洁的代码使测试通过（绿）
   - 禁止在测试通过前编写超出当前测试范围的代码
   - 预期结果：重新运行测试，所有测试通过

3. **重构**
   - 在测试通过的前提下，优化代码结构、消除重复、改善命名
   - 每次重构后重新运行测试套件
   - 预期结果：测试持续通过，代码质量提升

4. **重复循环**
   - 如果功能未完全实现，返回步骤 1 编写下一个测试用例
   - 直到所有需求都被测试覆盖且通过
   - 预期结果：完整的测试套件覆盖所有功能点

## 铁律
```
没有失败测试，就没有生产代码
```
