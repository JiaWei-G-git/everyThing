# 会话记录保存技能 (Session Recorder)

**版本**: v1.1.0
**类型**: Skill · 工作记录归档器 + 反思评估器

支持两种调用模式：

| 触发词 | 模式 | 输出内容 |
|--------|------|---------|
| "保存会话" / "记录本次会话" | **标准模式** | 仅原料归档（v1.0.0 兼容） |
| **"结束会话"** / "总结会话" / "完整记录" | **增强模式** | 原料 + 反思维度 + 量化评分 + 资产候选种子 |

> 💡 **手动调用**：`/skill:session-recorder`

---

## 功能说明

将当前 AI 协作会话按标准格式归档到 Vault 的原料层。Skill 会自动记忆归档路径，首次询问、后续复用。

**v1.1.0 新增**：
- 双触发模式：保存会话（标准）/ 结束会话（增强）
- 增强模式自动识别提示词优化对话与潜在资产候选
- YAML Frontmatter 新增 4 个量化字段，供下游 skill 过滤消费
- 条件性生成"提示词优化对照表"小节
- 增强模式自动生成资产候选种子草稿到 `待归档/`

## 执行步骤

### 0. 判定调用模式

根据触发词识别模式：

| 触发词 | 模式 |
|--------|------|
| "结束会话"、"总结会话"、"完整记录" | **增强模式** |
| "保存会话"、"记录本次会话" 或其他 | **标准模式** |

**模式差异**：
- **标准模式**：执行步骤 1-6（与 v1.0.0 完全一致）
- **增强模式**：在步骤 4 额外执行 4.2 反思维度提取与 4.3 量化评估；步骤 5 写入增强字段并生成条件性对照表；步骤 6 末尾生成资产候选种子草稿

### 1. 读取项目级配置

检查当前工作目录下是否存在 `AGENTS.md` 或 `CLAUDE.md`，查找其中的 `session_archive_path` 配置。

**配置块格式**（精确格式，Skill 按此解析）：
```markdown
<!-- AI-Vault-Config -->
- session_archive_path: "01-Work-工作记录/XX项目/会话记录/"
```

**读取规则**：扫描文件内容，按行匹配 `session_archive_path:`，取冒号后的值（去除首尾引号和空格）。
**写入规则**：
- 若文件已存在且无 `<!-- AI-Vault-Config -->` 标记，在文件末尾追加标记和配置行
- 若标记已存在，替换已有的 `session_archive_path` 行
- 若两个候选文件都不存在，新建 `AGENTS.md`

### 2. 首次记录：交互式选择路径

如配置不存在，扫描 Vault 原料层目录，按以下顺序提供选项：
1. `00-Inbox-收件箱/未分类`（默认选项，序号 1）
2. `01-Work-工作记录/<现有项目>/会话记录`（每个现有项目一个）
3. `01-Work-工作记录/<现有项目>/日志`（每个现有项目一个）
4. 新建项目目录（最后一项）

用户直接告知选择（如"选第2个"），若未明确指定则默认选择第1项。选"新建项目"时询问项目名，自动拼接为 `01-Work-工作记录/{项目名}/会话记录`。

选择后，将路径写入项目级说明文件（`AGENTS.md` 或 `CLAUDE.md`），使用上述配置块格式。

### 3. 二次记录：直接复用路径

如配置存在，直接读取 `session_archive_path`，不再询问用户。若路径失效，退回步骤 2。

### 4. 从会话上下文中自动提取信息

#### 4.1 基础字段提取（两种模式都执行）

从当前对话历史中自动提取以下字段，**不向用户提问**：

| 字段 | 提取规则 |
|------|---------|
| **会话主题** | 提取会话的核心目标，用一句话概括（用于生成文件名） |
| **项目名** | 从 `session_archive_path` 的第二级目录名自动推断 |
| **来源** | 由 AI 工具自行识别（Kimi-Code / Claude-Code / Cursor） |
| **背景描述** | 提取用户最初提出任务的上下文和动机 |
| **有效提示词序列** | 按时间顺序提取，仅保留**实质性任务指令**（见下方判断标准） |
| **解决思路** | 提取 AI 提供的解决方案、关键决策和技术选型 |
| **优化经验** | 提取对话中产生的可复用经验、踩坑记录和最佳实践 |

**有效提示词判断标准**（简化为 2 类，模糊地带默认保留）：

| 类别 | 判断标准 | 示例 | 是否提取 |
|------|---------|------|---------|
| **任务指令** | 包含具体目标、交付物或工作要求（动词+名词结构） | "帮我设计一个分库分表方案" / "再加一个周报模板" | ✅ 保留 |
| **其他** | 触发词、追问、操作指令、纯确认、纯反馈、补充修正等一切非任务指令 | "保存会话" / "为什么" / "确认" / "删除刚才的" | ❌ 忽略 |

> **模糊处理规则**：判断不确定时，**默认保留**（宁可多留不错杀）。

**字段提取 checklist**（AI 必须逐项核对，禁止跳过任何字段）：

| 序号 | 字段 | 核对标准 | 若无法提取时的标注 |
|------|------|---------|-------------------|
| 1 | 会话主题 | 是否用一句话概括了本次会话的核心目标？ | `未明确` |
| 2 | 项目名 | 是否从 `session_archive_path` 的第二级目录名推断？ | `未配置` |
| 3 | 来源 | 是否识别了当前 AI 工具（Kimi-Code / Claude-Code / Cursor）？ | `未知工具` |
| 4 | 背景描述 | 是否提取了用户最初提出任务的上下文和动机？ | `无背景` |
| 5 | 有效提示词序列 | 是否按时间顺序列出了所有**实质性任务指令**（排除触发词/追问/确认）？ | `无有效指令` |
| 6 | 解决思路 | 是否提取了 AI 提供的解决方案、关键决策和技术选型？ | `未记录` |
| 7 | 优化经验 | 是否扫描到关键词（"注意"、"坑"、"教训"、"踩过"、"最佳实践"）并提取上下文？ | `无优化经验` |

**提取后流程**（全自动，不询问确认）：
1. 自动提取所有字段，逐项核对 checklist
2. 若 checklist 中存在标注为"无..."的字段，在提取摘要中**明确列出缺失字段及原因**
3. 自动渲染模板并保存
4. 向用户汇报保存路径和提取摘要（含缺失字段提醒）

#### 4.2 反思维度提取（仅增强模式执行）

从对话历史中额外识别以下 2 类内容：

**类别 A: 提示词优化对话**

判定标准（满足任一即可）：
- 用户对前一个 prompt 表达过不满或要求改进（出现"再加上"、"刚才不够"、"换种方式"、"重新生成"、"内容太少"等词）
- 用户在 AI 回答后给出更具体的约束条件
- AI 主动建议过更好的提问方式

提取格式（按时间顺序成对收集）：
```yaml
- original: "原始提问的完整内容"
  improved: "优化版本（用户实际修改后的提问 或 推荐表述）"
  reason: "优化原因（缺失什么维度、为何效果更好）"
```

> 若本次会话**没有任何提示词优化对话**，将 `has_optimization_advice` 标记为 `false`，跳过 5.3 对照表生成。

**类别 B: 潜在资产候选**

判定标准（必须同时满足 ≥2 项）：
- AI 给出了**结构化的多步骤流程**（≥3 步）
- 用户**明确表示有效或将复用**
- 流程具备**可迁移性**（替换业务参数后可在同类场景使用）
- 对话中讨论过"做成 skill / 写成 prompt / 沉淀下来"

提取格式：
```yaml
- type: prompt | skill
  name: "动词+宾语，不超过 15 字"
  scenario: "适用场景或触发条件"
  summary: "一句话概括内容（不要完整模板，避免与后续 asset-extractor 重复劳动）"
```

#### 4.3 量化评估（仅增强模式执行）

基于 4.1 + 4.2 的提取结果，自动评分：

| 字段 | 评分规则 |
|------|---------|
| `reusability_score` | 1-5 分。基础分 1；每满足一项 +1：①有效提示词 ≥ 3 条 / ②解决思路结构化（含编号步骤或表格）/ ③优化经验非空 / ④资产候选非空 |
| `pattern_candidates` | 从主题、tags、解决思路中提取 1-3 个可聚类的话题标签（用于 pattern-miner 后续聚类） |
| `has_optimization_advice` | 类别 A 是否非空（true / false） |
| `new_skill_candidates` | 类别 B 中 `type=skill` 的条目数量 |

**增强模式 checklist 扩展**：

| 序号 | 字段 | 核对标准 | 若无法提取时的标注 |
|------|------|---------|-------------------|
| 8 | 提示词优化对话 | 是否扫描了用户改进提问的痕迹？ | `has_optimization_advice: false` |
| 9 | 资产候选 | 是否识别了 ≥3 步的可复用流程或模板？ | `new_skill_candidates: 0` |
| 10 | 量化评分 | 4 个量化字段是否按规则自动评分？ | 缺失时不写入对应字段 |

### 5. 渲染并保存

#### 5.1 Frontmatter 必填校验

**基础字段（6 个，两种模式都校验，缺一不可）**：

| 字段 | 校验规则 | 缺失时的默认值 |
|------|---------|-------------|
| project | 非空字符串 | 从 `session_archive_path` 推断，失败则用 `未分类` |
| date | 符合 `YYYY-MM-DD HH:MM` 格式 | 当前时间 |
| topic | 非空字符串 | `未命名会话` |
| source | 非空字符串 | `Unknown` |
| tags | 非空数组 | `["uncategorized"]` |
| status | `active` 或 `archived` | `active` |

**增强字段（4 个，仅增强模式写入）**：

| 字段 | 类型 | 缺失时默认值 |
|------|------|------------|
| `reusability_score` | 1-5 整数 | 1 |
| `pattern_candidates` | 字符串数组 | `[]` |
| `has_optimization_advice` | 布尔 | `false` |
| `new_skill_candidates` | 整数 | 0 |

**校验不通过时**：向用户展示缺失字段及已填充的默认值，询问是否确认保存或手动修正。

#### 5.2 渲染并保存

使用 `90-Templates/session_record.md` 模板渲染，保存到：

```
{session_archive_path}/会话记录-YYYY-MM-DD-{简短主题}.md
```

**文件名生成规则**：
- 基础名：`会话记录-{YYYY-MM-DD}-{主题}`
- 主题自动清理（全程自动化，不询问确认）：
  1. 替换 Windows 非法字符 `\/:"*?<>|` 为 `-`
  2. 截断到 30 个字符（超出部分舍去）
  3. 去除首尾空格和连续 `-`
- 主题为空或长度 < 3 字时，自动使用 `未命名` 作为兜底
- 去重：若同名文件已存在，自动追加序号（如 `-1`、`-2`）
- `date` 字段使用 `YYYY-MM-DD HH:MM` 格式（精确到分钟）

**YAML Frontmatter 完整示例（增强模式）**：
```yaml
---
project: "XX项目"
date: "2026-05-16 21:30"
topic: "session-recorder skill 升级方案"
source: "Claude-Code"
tags: [skill-design, session-management]
status: active
# v1.1.0 增强字段（仅增强模式写入）
reusability_score: 4
pattern_candidates: ["skill 升级流程", "反思维度设计"]
has_optimization_advice: true
new_skill_candidates: 1
---
```

**正文模板**：
- 背景
- 用户提示词序列（仅有效提示词）
- 解决思路
- 优化经验
- **提示词优化对照表**（仅当 `has_optimization_advice: true` 时生成，见 5.3）
- 关联笔记

#### 5.3 条件性生成"提示词优化对照表"小节（仅增强模式）

**生成条件**：`has_optimization_advice: true`

**对照表格式**：
```markdown
## 提示词优化对照表

| 原始提问 | 优化版本 | 优化原因 |
|---------|---------|---------|
| {original_1} | {improved_1} | {reason_1} |
| {original_2} | {improved_2} | {reason_2} |
```

**生成规则**：
- 每对优化对话占一行
- 长文本（>80 字）折行显示，用 `<br>` 软换行保持表格结构
- 若类别 A 提取为空，**禁止生成空表**（已被 `has_optimization_advice: false` 拦截）

### 6. 资产候选种子草稿（仅增强模式）

若步骤 4.2 类别 B 非空（`new_skill_candidates > 0` 或检测到 prompt 候选），自动生成种子草稿到 `00-Inbox-收件箱/待归档/`：

**种子草稿文件名格式**：
```
00-Inbox-收件箱/待归档/[L]Prompt-{name}-seed.md
00-Inbox-收件箱/待归档/[L]Skill-{name}-seed.md
```

**说明**：
- 种子草稿统一标记 `confidence: low` + 文件名前缀 `[L]`
- 种子草稿仅包含步骤 4.2 提取的精简内容（type / name / scenario / summary）
- 后续可通过 `asset-extractor` 补全完整 content

**Skill 种子草稿模板**：
```markdown
---
type: skill_candidate
name: "{name}"
scenario: "{scenario}"
confidence: low
source: "{当前会话记录的完整路径}"
status: seed
---

# {name}

## 场景
{scenario}

## 摘要
{summary}

> ⚠️ 这是 session-recorder 增强模式自动识别的**种子草稿**，仅含粗粒度信息。
> 建议下一步使用 `asset-extractor` 调用完整 LLM 提取，补全可执行步骤。
```

**Prompt 种子草稿模板**：
```markdown
---
type: prompt
name: "{name}"
scenario: "{scenario}"
confidence: low
source: "{当前会话记录的完整路径}"
status: seed
---

# {name}

## 场景
{scenario}

## 摘要
{summary}

> ⚠️ 这是 session-recorder 增强模式自动识别的**种子草稿**，仅含粗粒度信息。
> 建议下一步使用 `asset-extractor` 调用完整 LLM 提取，补全完整 Prompt 模板。
```

**去重规则**：若 `待归档/` 目录已存在同名 seed 文件，追加序号（如 `-1`）。

### 7. 通知用户

**标准模式**汇报：
- 文件已保存的完整路径
- 字段缺失提醒（如有）

**增强模式**汇报：
- 文件已保存的完整路径
- 字段缺失提醒（如有）
- **量化评分**：
  - 复用价值评分（reusability_score）：X/5
  - 是否生成对照表：是 / 否
  - 资产候选数量：Skill X 条 / Prompt Y 条
- **种子草稿列表**（若有）：
  - 列出每个 seed 文件的完整路径
  - 提示「可使用 `asset-extractor` 进一步提炼」

## 核心规则（不可违反）

1. **必须包含完整 YAML Frontmatter** — 6 个基础字段缺一不可
2. **增强字段仅在增强模式写入** — 标准模式保持 v1.0.0 兼容
3. **文件名自动去重** — 如遇同名追加序号
4. **路径记忆持久化** — 配置必须写入项目级 `AGENTS.md / CLAUDE.md`，而非临时变量
5. **默认兜底** — 用户未选择路径且无配置时，默认写入 `00-Inbox-收件箱/未分类/`
6. **种子草稿仅在增强模式生成** — 标准模式不生成
7. **条件性对照表** — `has_optimization_advice: false` 时**禁止生成空表**
8. **资产索引维护提醒** — 若本次会话涉及新增或修改 Skill/Agent/Prompt，保存后提示用户更新对应目录的 `README.md`

## 使用示例

### 示例 A：标准模式（v1.0.0 兼容）

**用户**: "保存本次会话"

**执行**:
1. 读取项目级配置，检测 `session_archive_path`
2. 自动提取 7 个基础字段
3. 渲染 v1.0.0 兼容模板并保存（无增强字段、无对照表、无种子草稿）
4. 通知用户保存路径

### 示例 B：增强模式

**用户**: "结束会话"

**执行**:
1. 读取项目级配置，检测 `session_archive_path`
2. 自动提取 7 个基础字段
3. 提取反思维度（提示词优化对话 + 资产候选）
4. 自动量化评分（reusability_score / pattern_candidates / has_optimization_advice / new_skill_candidates）
5. 写入 6 + 4 = 10 个 frontmatter 字段
6. 条件性生成对照表小节
7. 若有资产候选，生成种子草稿到 `00-Inbox-收件箱/待归档/`
8. 向用户报告：
   ```
   ✅ 会话已归档到：01-Work-工作记录/XX项目/会话记录/会话记录-2026-05-16-XXX.md

   📊 量化评分
   - 复用价值：4/5
   - 提示词优化对照表：已生成（3 对）
   - 资产候选：1 条 Skill + 2 条 Prompt 种子

   🌱 种子草稿
   - 00-Inbox-收件箱/待归档/[L]Skill-skill升级双触发模式-seed.md
   - 00-Inbox-收件箱/待归档/[L]Prompt-反思维度提取-seed.md
   - 00-Inbox-收件箱/待归档/[L]Prompt-量化评分规则-seed.md

   💡 建议下一步：使用 asset-extractor 提炼这些种子草稿
   ```

## 下游消费方说明

v1.1.0 的增强字段为下游 skill 提供过滤条件：

| 下游 Skill | 消费方式 |
|----------|---------|
| `asset-extractor` | 优先处理 `new_skill_candidates > 0` 的会话；调用时直接消费 `待归档/[L]*-seed.md` 种子草稿 |
