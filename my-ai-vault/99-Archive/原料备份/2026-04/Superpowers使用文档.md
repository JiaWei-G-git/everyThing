## 项目概述

**Superpowers** 是一个为AI编程助手设计的完整软件开发工作流系统。它基于一组可组合的"技能"（Skills）和初始指令构建，确保您的AI代理在编写代码之前先理解需求、制定计划、遵循测试驱动开发（TDD）等最佳实践。

### 核心理念

- **测试驱动开发（TDD）** - 始终先写测试，再写代码
- **系统化而非随意** - 遵循流程而非猜测
- **降低复杂度** - 将简单性作为主要目标
- **证据胜于声明** - 在宣布成功之前先验证

## 核心工作流

Superpowers 定义了一个完整的开发工作流，从头脑风暴到完成分支：

```
头脑风暴 → Git工作树 → 编写计划 → 子代理开发/执行计划 → 代码审查 → 完成分支
```

### 工作流步骤

| 步骤 | 技能 | 触发时机 | 说明 |
|------|------|----------|------|
| 1 | **brainstorming** | 编写代码之前 | 通过提问完善想法，探索方案，展示设计并获得批准 |
| 2 | **using-git-worktrees** | 设计批准后 | 创建隔离的工作空间，运行项目设置，验证干净测试基线 |
| 3 | **writing-plans** | 设计批准时 | 将工作分解为2-5分钟的小任务，每个任务包含完整代码和验证步骤 |
| 4 | **subagent-driven-development** 或 **executing-plans** | 有计划时 | 分派子代理执行任务并审查，或在批量执行中带检查点 |
| 5 | **test-driven-development** | 实现期间 | 强制执行红-绿-重构循环 |
| 6 | **requesting-code-review** | 任务之间 | 对照计划审查，按严重性报告问题 |
| 7 | **finishing-a-development-branch** | 任务完成时 | 验证测试，提供选项（合并/PR/保留/丢弃），清理工作树 |

## 技能库

Superpowers 包含以下技能，按类别组织：

### 测试技能

#### 1. test-driven-development（测试驱动开发）

**触发时机：** 在实现任何功能或修复错误时使用，**在编写实现代码之前**

**自动触发：** ✅ 是 - 当用户要求实现功能或修复错误时自动触发

**核心原则：**
- 红-绿-重构循环
- 先写测试，看着它失败，编写最小代码使其通过
- 没有在测试之前写代码

**关键规则：**
```
没有失败测试，就没有生产代码
```

**使用场景：**
- 新功能开发
- 错误修复
- 重构
- 行为更改

---

### 调试技能

#### 2. systematic-debugging（系统化调试）

**触发时机：** 在遇到任何错误、测试失败或意外行为时使用，**在提出修复之前**

**自动触发：** ✅ 是 - 当检测到错误或测试失败时自动触发

**核心原则：** 在尝试修复之前始终找到根本原因。症状修复就是失败。

**四个阶段：**
1. **根本原因调查** - 阅读错误、重现、检查更改、收集证据
2. **模式分析** - 找到工作示例、比较、识别差异
3. **假设和测试** - 形成单一假设、最小化测试、验证
4. **实现** - 创建失败测试用例、实施单一修复、验证

**铁律：**
```
没有根本原因调查，就没有修复
```

**支持技术：**
- `root-cause-tracing.md` - 向后追踪堆栈中的错误
- `defense-in-depth.md` - 在多层添加验证
- `condition-based-waiting.md` - 用条件轮询替换任意超时

---

#### 3. verification-before-completion（完成前验证）

**触发时机：** 在即将声称工作完成、修复或通过时使用，**在提交或创建PR之前**

**自动触发：** ✅ 是 - 当代理准备声明成功时自动触发

**核心原则：** 始终证据先于声明。

**门函数：**
```
1. 识别：什么命令证明这个声明？
2. 运行：执行完整命令（新鲜、完整）
3. 读取：完整输出，检查退出代码，计数失败
4. 验证：输出是否确认声明？
5. 只有这样：做出声明
```

**铁律：**
```
没有新鲜验证证据，就没有完成声明
```

---

### 协作技能

#### 4. brainstorming（头脑风暴）

**触发时机：** 在任何创造性工作之前必须使用 - 创建功能、构建组件、添加功能或修改行为

**自动触发：** ✅ 是 - 当用户提出新功能或修改请求时自动触发

**硬门：** 在展示设计并获得用户批准之前，不要调用任何实现技能、编写任何代码。

**检查清单：**
1. 探索项目上下文
2. 提供可视化助手（如果涉及视觉问题）
3. 提出澄清问题（一次一个）
4. 提出2-3种方案（包含权衡和推荐）
5. 展示设计（按复杂度分节）
6. 编写设计文档（保存到 `docs/superpowers/specs/`）
7. 规范自检
8. 用户审查书面规范
9. 过渡到实现（调用 writing-plans）

**核心原则：**
- 一次一个问题
- 无情地YAGNI（你不会需要它）
- 探索替代方案
- 增量验证

---

#### 5. writing-plans（编写计划）

**触发时机：** 当你有规范或多步骤任务的需求时，在触及代码之前使用

**自动触发：** ✅ 是 - 在头脑风暴完成并获得用户批准后自动触发

**核心原则：** 假设工程师对我们的代码库一无所知且品味有问题。记录他们需要知道的一切。

**任务粒度：** 每个步骤是一个2-5分钟的动作

**计划文档头部模板：**
```markdown
# [功能名称] 实现计划

> **对于代理工作者：** 必需子技能：使用 superpowers:subagent-driven-development

**目标：** [一句话描述构建什么]

**架构：** [关于方法的2-3句话]

**技术栈：** [关键技术/库]

---
```

**任务结构：**
- 创建/修改的确切文件路径
- 每个步骤的完整代码
- 带有预期输出的确切命令
- 提交步骤

---

#### 6. subagent-driven-development（子代理驱动开发）

**触发时机：** 在当前会话中执行具有独立任务的实现计划时使用

**自动触发：** ✅ 是 - 当用户选择"子代理驱动（推荐）"执行选项时

**核心原则：** 每个任务新子代理 + 两阶段审查（规范合规性，然后代码质量）= 高质量、快速迭代

**过程：**
1. 阅读计划，提取所有带完整文本的任务
2. 为每个任务分派实现者子代理
3. 分派规范审查者子代理（确认代码与规范匹配）
4. 分派代码质量审查者子代理（代码质量审查）
5. 标记任务完成
6. 重复直到所有任务完成
7. 分派最终代码审查者
8. 使用 finishing-a-development-branch 完成

**模型选择策略：**
- 机械实现任务 → 快速、便宜的模型
- 集成和判断任务 → 标准模型
- 架构、设计和审查任务 → 最有能力的模型

---

#### 7. executing-plans（执行计划）

**触发时机：** 当你有一个书面的实现计划要在单独的会话中执行，并带有审查检查点时使用

**自动触发：** ✅ 是 - 当用户选择"内联执行"选项时

**注意：** 如果有子代理访问权限，使用 subagent-driven-development 会更好。

**过程：**
1. 加载和审查计划
2. 执行任务（每个任务：标记进行中 → 执行 → 标记完成）
3. 完成开发（使用 finishing-a-development-branch）

---

#### 8. using-git-worktrees（使用Git工作树）

**触发时机：** 在开始需要与当前工作区隔离的功能工作，或执行实现计划之前使用

**自动触发：** ✅ 是 - 在开始实现之前自动触发

**核心原则：** 系统化目录选择 + 安全验证 = 可靠隔离。

**目录选择优先级：**
1. 检查现有目录（`.worktrees/` 或 `worktrees/`）
2. 检查 CLAUDE.md 中的偏好
3. 询问用户

**安全验证：**
- 项目本地目录必须被 .gitignore 忽略
- 运行项目设置（自动检测 npm/cargo/pip 等）
- 验证干净测试基线

---

#### 9. requesting-code-review（请求代码审查）

**触发时机：** 在完成任务、实现主要功能或合并前验证工作是否符合要求时使用

**自动触发：** ✅ 是 - 在每个任务完成后自动触发

**强制审查时机：**
- 子代理驱动开发中的每个任务后
- 完成主要功能后
- 合并到 main 之前

**过程：**
1. 获取git SHAs（BASE_SHA 和 HEAD_SHA）
2. 分派 code-reviewer 子代理
3. 根据反馈行动（立即修复严重问题，继续之前修复重要问题）

---

#### 10. receiving-code-review（接收代码审查）

**触发时机：** 在收到代码审查反馈时使用，特别是如果反馈看起来不清楚或技术上可疑时

**自动触发：** ✅ 是 - 当接收到代码审查反馈时

**响应模式：**
1. 阅读：完整阅读反馈而不反应
2. 理解：用自己的话重述需求
3. 验证：对照代码库现实检查
4. 评估：对当前代码库技术上合理吗？
5. 回应：技术确认或有理由的反驳
6. 实施：一次一项，每项测试

**禁止的回应：**
- ❌ "你说得绝对正确！"
- ❌ "好观点！"
- ❌ "我现在就实现它"

---

#### 11. finishing-a-development-branch（完成开发分支）

**触发时机：** 当实现完成、所有测试通过，并且你需要决定如何集成工作时使用

**自动触发：** ✅ 是 - 在所有任务完成后自动触发

**过程：**
1. 验证测试（必须全部通过）
2. 确定基础分支
3. 呈现4个选项：
   - 本地合并回基础分支
   - 推送并创建拉取请求
   - 保持分支原样（稍后处理）
   - 放弃此工作
4. 执行选择
5. 清理工作树

---

#### 12. dispatching-parallel-agents（分派并行代理）

**触发时机：** 当面对2个或更多可以在没有共享状态或顺序依赖的情况下处理的独立任务时使用

**自动触发：** ✅ 是 - 当检测到多个独立的故障或任务时

**使用场景：**
- 3个或更多测试文件因不同根本原因失败
- 多个子系统独立损坏
- 每个问题可以在没有来自其他问题的上下文的情况下理解

**模式：**
1. 识别独立域
2. 创建聚焦的代理任务
3. 并行分派
4. 审查和集成

### 元技能

#### 13. using-superpowers（使用Superpowers）

**触发时机：** 在开始任何对话时使用 - 建立如何查找和使用技能

**自动触发：** ✅ 是 - **会话开始时自动加载**

**这是最重要的技能**，它告诉代理如何查找和使用其他技能。

**规则：**
- 在任何响应或行动之前调用相关或请求的技能
- 即使1%的可能性技能可能适用也意味着应该调用技能来检查
- 如果技能适用于任务，必须使用它

**红旗（合理化检查）：**
| 想法 | 现实 |
|------|------|
| "这只是一个简单的问题" | 问题是任务。检查技能。 |
| "我需要先更多上下文" | 技能检查在澄清问题之前。 |
| "让我先探索代码库" | 技能告诉你如何探索。先检查。 |
| "我可以快速检查git/文件" | 文件缺乏对话上下文。检查技能。 |

**指令优先级：**
1. 用户的显式指令（最高优先级）
2. Superpowers技能
3. 默认系统提示（最低优先级）

---

#### 14. writing-skills（编写技能）

**触发时机：** 在创建新技能、编辑现有技能或部署前验证技能有效时使用

**自动触发：** ❌ 否 - 需要显式调用

**核心原则：** 编写技能是将测试驱动开发应用于过程文档。

**TDD映射：**
| TDD概念 | 技能创建 |
|---------|----------|
| 测试用例 | 带有子代理的压力场景 |
| 生产代码 | 技能文档（SKILL.md） |
| 测试失败（红） | 没有技能的代理违反规则 |
| 测试通过（绿） | 有技能的代理合规 |

**铁律：**
```
没有失败测试，就没有技能
```

## 使用指南

### 基本使用

1. **启动对话** - `using-superpowers` 自动加载，建立技能查找机制

2. **提出需求** - 例如："帮我实现用户认证功能"
   - `brainstorming` 自动触发
   - 代理会询问澄清问题、探索方案、展示设计

3. **批准设计** - 审查并批准设计文档后
   - `writing-plans` 自动触发创建实现计划

4. **选择执行方式** - 计划完成后，代理会提供两种选项：
   - **子代理驱动（推荐）** - 每个任务分派新子代理，自动审查
   - **内联执行** - 在当前会话中批量执行任务

5. **实现过程** - 根据选择：
   - `subagent-driven-development` 或 `executing-plans` 执行计划
   - `test-driven-development` 强制执行红-绿-重构循环
   - `requesting-code-review` 每个任务后审查代码

6. **完成工作** - 所有任务完成后
   - `finishing-a-development-branch` 提供合并/PR/保留/丢弃选项

### 调试使用

当遇到错误时：

1. **报告错误** - 例如："测试失败了" 或 "遇到错误"
   - `systematic-debugging` 自动触发
   - 代理会引导完成4阶段调试过程

2. **遵循过程** - 代理会：
   - 阅读错误消息
   - 尝试一致重现
   - 检查最近更改
   - 追踪根本原因

3. **修复并验证** - 找到根本原因后：
   - `test-driven-development` 创建失败测试
   - 实施修复
   - `verification-before-completion` 验证修复有效

### 显式调用技能

如果需要显式调用特定技能：

```

```

```

```


## 总结

Superpowers 通过强制执行以下原则来提升AI辅助开发的质量：

1. **先设计，后编码** - brainstorming 确保需求清晰
2. **先测试，后实现** - TDD 确保代码可测试且正确
3. **找到根本原因** - 系统化调试避免症状修复
4. **持续审查** - 每个任务后代码审查保持质量
5. **验证一切** - 完成前验证确保证据先于声明
