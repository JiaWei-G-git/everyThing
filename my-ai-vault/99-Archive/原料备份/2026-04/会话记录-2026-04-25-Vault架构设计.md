# 会话记录 - Vault目录架构设计

## 背景

需要为 AI 协作知识库设计一个目录结构，要求：
1. 支持从原料到资产的完整流转
2. 兼容 Kimi Code / Claude Code / Cursor 等多工具
3. 便于 Git 版本控制

## Prompt

```
请帮我设计一个基于 Markdown Vault 的 AI 知识库目录结构。

需求：
- 区分"原料层"（临时捕获、按项目聚合）和"资产层"（沉淀后的可复用资产）
- 原料层包括：快速捕获收件箱、按项目的工作记录
- 资产层包括：Prompt、Agent定义、Skill、MCP服务、工作流、教程
- 还需要：团队共享区、模板区、归档区

输出要求：
1. 完整目录树（带中文注释说明每层用途）
2. 每个目录的命名规范
3. 文件流转规则（原料如何进入资产层）
```

## 解决思路

采用**双轴驱动**架构：

**纵向轴（处理流程）**：
```
00-Inbox（快速捕获）
  → 01-Work（按项目聚合）
    → 10~60（提炼为资产）
      → 70-Sharing（团队共享）
        → 99-Archive（历史归档）
```

**横向轴（资产类型）**：
```
10-Prompts      优质提问模板
20-Agents       角色定义与交互流程
30-Skills       原子技能（与AI工具共享）
40-MCP          MCP服务定义
50-Workflows    多步骤工作流
60-Tutorials    工具使用经验与教程
```

关键决策：
- `10/20/30/40` 使用英文命名（工具链兼容）
- `00/01/50/60/70/90/99` 保留中文（可读性）
- `30-Skills/` 是**唯一来源**，通过符号链接共享给各 AI 工具

## 优化经验

1. **目录编号前缀**非常有用，在文件管理器中天然排序，且一眼看出层级关系
2. **中文注释放在目录名中**（如 `00-Inbox-收件箱`）比单独的 README 更直观
3. `30-Skills/` 作为共享目录的设计很重要：修改一次，所有 AI 工具同步生效
4. 应该预留 `待提炼/` 子目录，让 LLM 提取的候选资产有临时落脚点
