# 模式挖掘技能 (Pattern Miner)

**版本**: v1.0.0
**类型**: Skill · 跨会话重复模式发现器

当用户说 **"盘点本周模式"**、**"发现重复主题"**、**"提炼高频模式"** 或表达类似意图时，AI 自动匹配本 Skill。

> 💡 **手动调用**：`/skill:pattern-miner`

---

## 功能说明

扫描 `01-Work-工作记录/` 下最近 N 天的会话记录，按主题聚类统计频次，发现重复出现的模式（≥3 次）。用户确认后，将相关的多条会话记录作为输入，进行**跨案例抽象**，提炼出通用 Skill / Prompt。

**与 `asset-extractor` 的核心区别**：
- `asset-extractor`：从**单条原料**中提取资产（可能提取出伪通用的单次经验）
- `pattern-miner`：从**多条同类会话**中抽象模式（确保提炼出的资产经过重复验证）

---

## 执行步骤

### 1. 确定扫描范围

**默认**：最近 30 天内的会话记录。
**用户可指定**：
- 周期：日 / 周 / 月（默认月）
- 范围：`01-Work/` 全量，或指定某个项目子目录

**增量扫描模式**（默认启用）：
1. 读取 `AGENTS.md` / `CLAUDE.md` 中的 `pattern_miner_last_scan` 时间戳
2. 若存在，只扫描 `date` 晚于该时间戳且未标记 `pattern_extracted: true` 的会话记录
3. 若不存在或用户明确要求「全量盘点」，则扫描全部符合条件的记录

### 2. 采集会话元数据

扫描目标目录下所有 `会话记录-YYYY-MM-DD-*.md` 文件，读取 YAML Frontmatter：
- `topic` — 用于文本相似度聚类
- `tags` — 用于主题分类
- `date` — 用于时间筛选
- `project` — 用于归属分析
- `pattern_extracted` — 若已存在且为 `true`，跳过该记录

**不读取正文**，此阶段只做元数据统计，减少 LLM 调用成本。

### 3. 主题聚类与频次统计

**聚类方式：一次性 LLM 语义分组（禁止关键词匹配）**

1. 收集所有会话的 `topic`、`tags`、`project`，按以下格式组成输入列表：
   ```
   [ID: 001] topic: SQL 慢查询排查 | tags: [database, performance] | project: 订单系统
   [ID: 002] topic: SQL 索引优化方案 | tags: [database, optimization] | project: 用户中心
   ...
   ```

2. 调用 LLM 进行**批量语义分组**（单次调用，O(1) 复杂度）

**禁止**：基于关键词匹配的聚类（如"SQL 慢查询"和"SQL 索引优化"仅因共享"SQL"就被视为同类）

**LLM 聚类提示词**：
```
你是一位文本聚类专家。请将以下会话主题按语义相似度分组。

输入（每行一个会话）：
{输入列表}

分组原则：
1. 只有真正解决同一类问题的主题才能分到同一组
2. 共享技术关键词但问题类型不同（如"SQL 慢查询排查"和"SQL 索引优化"）应分到不同组
3. 每组给一个 5-10 字的组名
4. 每条会话只能属于一个组，无法归类的放入 "其他"

输出格式（JSON）：
```json
{
  "groups": [
    {
      "name": "数据库性能排查",
      "members": ["001"],
      "reason": "都是排查 SQL 执行慢的问题"
    }
  ],
  "ungrouped": []
}
```
```

3. 根据 LLM 返回的分组结果统计每组频次，按频次降序排列

**阈值规则**：
- ≥ 3 次：标记为「建议提炼」，高亮展示
- 2 次：标记为「值得关注」，灰色展示
- 1 次：不展示（噪声过滤）

### 4. 输出「重复主题排行榜」

向用户汇报，格式如下：

```
📊 重复主题排行榜（最近 30 天）

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
🔥 建议提炼（出现 ≥3 次）

1. 【SQL 性能优化】出现 5 次
   涉及项目：订单系统重构(3) / 客户A系统对接(2)
   涉及文件：
   - 01-Work/订单系统重构/会话记录/会话记录-2026-04-21-数据库分库分表方案.md
   - 01-Work/订单系统重构/会话记录/会话记录-2026-04-22-Redis缓存策略设计.md
   - ...
   💡 建议：提炼通用排查 Skill

2. 【Docker 镜像优化】出现 3 次
   ...

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
👀 值得关注（出现 2 次）

3. 【Vue 组件设计】出现 2 次
   ...

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
请输入序号选择要提炼的主题，或回复「跳过」结束盘点。
```

### 5. 用户选择主题

**交互规则**：
- 用户输入序号（如 `1`）→ 进入跨案例抽象
- 用户回复「跳过」→ 结束本次盘点
- 用户输入多个序号（如 `1,2`）→ 依次处理

### 6. 跨案例抽象

读取该主题簇下的所有会话记录**正文**。

**LLM 系统提示词**：

```
你是一位知识库模式抽象专家。以下是从多个会话记录中收集的关于同一主题的工作记录。
这些记录来自不同时间、不同项目，但都解决同一类问题。

你的任务不是改写单条记录，而是从多条记录中归纳出"通用模式"：

1. 找出所有记录中**共同出现的核心步骤**（在 ≥3 条记录中出现的步骤才算通用）
2. 找出只出现在单条记录中的**特化操作**，标注为"可选/视情况而定"
3. 抽象出**触发条件** — 什么情况下应该使用这个模式
4. 抽象出**输入/输出** — 需要什么信息，产出什么结果
5. 总结**常见误区** — 多条记录中都踩过的坑

输出格式要求：
- 如果是 Skill：触发条件 / 通用步骤（编号）/ 特化选项 / 常见误区
- 如果是 Prompt：适用场景 / 通用模板 / 变量说明

严格约束：
- 不要包含特定项目的业务细节（如"客户A的订单表"、"XX项目的特殊配置"）
- 不要编造记录中没有提到的步骤或工具
- 如果记录之间矛盾，如实标注"记录中存在分歧"，不要强行统一
- 如果无法归纳出通用模式（记录差异太大），直接返回："暂不具备提炼条件"
```

**LLM 调用参数**：
- temperature: 0.3（低温度确保归纳一致性）
- 超时: 120 秒（多条记录内容较长）

### 7. 渲染并保存

将跨案例抽象的结果渲染为标准 Markdown 草稿。

**产出规则**：
- 若抽象出可执行流程 → 保存为 Skill 草稿
- 若抽象出可复用提问模板 → 保存为 Prompt 草稿
- 若两者都有 → 分别保存两个文件
- 若无法抽象 → 向用户说明原因，不保存空文件

**保存路径**：
```
00-Inbox-收件箱/待提炼/Skill-{主题名}-跨案例抽象.md
00-Inbox-收件箱/待提炼/Prompt-{主题名}-跨案例抽象.md
```

**文件名去重**：若同名文件已存在，追加序号（如 `-1`）。

### 8. 标记关联记录

对相关会话记录的 YAML Frontmatter 追加字段：
```yaml
pattern_extracted: true
pattern_id: "SQL性能优化-2026-04-25"
pattern_sessions: 5
```

**追加规则**：
- 若文件已有 Frontmatter，在末尾追加新字段
- 若文件无 Frontmatter，在文件开头插入完整 Frontmatter 块

### 9. 更新扫描时间戳

将当前时间写入 `AGENTS.md` / `CLAUDE.md`：
```markdown
<!-- AI-Vault-Config -->
- session_archive_path: "01-Work-工作记录/XX项目/会话记录"
- pattern_miner_last_scan: "2026-04-25T14:30:00"
```

**写入规则**：
- 若配置块已存在，追加或替换 `pattern_miner_last_scan` 行
- 若配置块不存在，在文件末尾追加完整配置块

### 10. 归档原料

跨案例抽象完成后，将该主题簇下的所有会话记录原料移动到归档区：

**归档路径规则**：
```
99-Archive/原料备份/{YYYY-MM}/{原文件名}
```

**执行动作**：
1. 读取每条原料 YAML Frontmatter 中的 `date` 字段，提取年月
2. 若 `date` 不存在，使用当前年月
3. 创建归档目录（如不存在）：`99-Archive/原料备份/2026-04/`
4. 在每条原料 Frontmatter 末尾追加归档字段：
   ```yaml
   archived: true
   archive_date: "2026-04-26"
   original_path: "01-Work-工作记录/XX项目/会话记录/会话记录-YYYY-MM-DD-xxx.md"
   ```
5. 移动原料文件到归档目录

**source 字段处理**：
- 草稿文件中的 `source` 字段保持原值不变
- 通过 `original_path` 可追溯原料最初来源位置

**边界情况**：
- 若归档目录已存在同名文件，追加序号去重（如 `-1`）

## 核心规则（不可违反）

1. **频次门槛 ≥3**：只有出现 3 次以上的主题才建议提炼，低于 3 次只展示不推荐
2. **跨案例不是改写**：必须基于多条记录归纳共性，不能只看单条记录生成通用 Skill
3. **不强求产出**：如果记录之间缺乏共性，允许返回"暂不具备提炼条件"
4. **来源可追溯**：每条抽象出的规则必须标注来源会话数量（如"基于 5 条会话归纳"）
5. **排除伪通用**：剔除只出现过 1 次的特化操作，不把偶然操作包装成通用步骤

## 使用示例

**用户**: "盘点本周模式"

**执行**: 
1. 扫描 `01-Work/` 下最近 7 天的会话记录
2. 聚类统计：发现"SQL 性能优化"出现 5 次，"Docker 构建"出现 2 次
3. 输出排行榜，高亮"SQL 性能优化"
4. 用户输入 `1`
5. 读取 5 条 SQL 相关会话记录的正文
6. 调用 LLM 进行跨案例抽象：
   - 共同步骤：查执行计划 → 分析索引 → 改写 SQL → 验证效果（4 条记录都有）
   - 特化选项：添加复合索引（仅 2 条涉及，标注为可选）
   - 常见误区：LEFT JOIN 在 WHERE 中未使用仍执行（3 条都踩过）
7. 渲染为 `Skill-SQL性能优化-跨案例抽象.md`，保存到 `待提炼/`
8. 标记 5 条会话记录 `pattern_extracted: true`
9. 向用户汇报产出结果和文件路径

> 📌 **维护提醒**：如抽象出新 Skill/Prompt，别忘了同步更新 `30-Skills/README.md` 的资产索引
