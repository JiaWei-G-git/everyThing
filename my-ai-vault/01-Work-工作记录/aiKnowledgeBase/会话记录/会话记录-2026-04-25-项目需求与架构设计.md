# 会话记录 - 项目需求与架构设计

## 背景

日常开发中与 AI（Claude Code、Kimi Code、Cursor）协作产生大量高价值信息：Prompt、调试经验、角色定义、工作流等。这些信息散落在 IDE 会话和聊天记录中，无法沉淀为可复用资产。需要构建一个基于 Markdown Vault 的个人 AI 知识库管理系统。

## 用户提示词序列

```
[1] 请帮我设计一个基于 Markdown Vault 的 AI 知识库管理系统

[2] 需要支持从原料到资产的完整流转，兼容 Kimi Code / Claude Code / Cursor 等多工具

[3] 目录结构要区分原料层和资产层，还要有团队共享区、模板区、归档区
```

## 解决思路

**核心闭环设计**：
```
使用 AI -> 保存会话 -> 提炼资产 -> 人工确认 -> 沉淀知识库
```

**双轴驱动架构**：

纵向轴（处理流程）：
```
00-Inbox（快速捕获）
  -> 01-Work（按项目聚合）
    -> 10~60（提炼为资产）
      -> 70-Sharing（团队共享）
        -> 99-Archive（历史归档）
```

横向轴（资产类型）：
```
10-Prompts      优质提问模板
20-Agents       角色定义与交互流程
30-Skills       原子技能（与 AI 工具共享）
40-MCP          MCP 服务定义
50-Workflows    多步骤工作流
60-Tutorials    工具使用经验与教程
```

**关键决策**：
- `10/20/30/40` 使用英文命名（工具链兼容）
- `00/01/50/60/70/90/99` 保留中文（可读性）
- `30-Skills/` 是唯一来源，通过符号链接共享给各 AI 工具

## 优化经验

1. 目录编号前缀非常有用，在文件管理器中天然排序，且一眼看出层级关系
2. 中文注释放在目录名中（如 `00-Inbox-收件箱`）比单独的 README 更直观
3. `30-Skills/` 作为共享目录的设计很重要：修改一次，所有 AI 工具同步生效
4. 应该预留 `待提炼/` 子目录，让 LLM 提取的候选资产有临时落脚点
5. 双轴驱动的架构比线性分类更清晰，原料层->资产层->产出层的流转一目了然
