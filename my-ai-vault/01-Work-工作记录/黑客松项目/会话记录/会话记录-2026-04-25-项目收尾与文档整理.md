# 会话记录 - 项目收尾与文档整理

## 背景

Hackathon MVP 最后一天，需要完成：
1. 清理代码，去掉 Python 依赖，改为纯 Skill 模式
2. 更新 README、PROJECT.md、PROJECT_GENERATOR.md
3. 验证所有文件无 Python / mock 残留
4. Git 提交整理

## Prompt

```
请帮我整理项目文档，确保：
1. README.md 清晰说明"这是什么、如何创建、如何使用"
2. 所有安装方式统一为"提示词提交给 AI"，不出现具体脚本命令
3. 项目文件结构与实际一致
4. 文档间无矛盾
```

## 解决思路

**纯 Skill 模式的转型**：

原计划是 Python CLI + Skill 双轨，经过讨论后决定完全去掉 Python：
- 原因1：不同 AI 工具的 Python 环境差异大
- 原因2：API Key 配置麻烦，容易泄露
- 原因3：纯 Skill 更轻量，跨工具统一

转型后的架构：
```
PROJECT_GENERATOR.md  ->  AI 一键生成整个项目
         |
    my-ai-vault/       ->  知识库目录
    templates/         ->  Markdown 模板
         |
  提示词触发 Skill     ->  AI 自动执行归档/提炼/生成
```

**文档一致性策略**：

1. PROJECT_GENERATOR.md 是唯一蓝图，包含所有文件的完整内容
2. README.md 面向用户，回答"是什么、怎么用"
3. PROJECT.md 面向技术，包含架构和里程碑
4. 三者通过提示词生成保持同步

## 优化经验

1. **统一安装方式**的决策很重要。之前用了 ln -s / New-Item 等脚本，不同操作系统和 AI 工具差异太大。改为"对 AI 说一句话"后，跨工具体验统一了
2. **符号链接的核心设计不能丢**。虽然不让用户手动执行脚本，但"30-Skills 是唯一来源"这个原则要明确告诉 AI，确保修改一处、全局生效
3. **PROJECT_GENERATOR.md 需要包含所有文件内容**，不能只是文件列表。这样任何 AI 拿到这份文档都能完整重建项目
4. 清理时容易遗漏：docs/ 目录下的历史文档、PROJECT_GENERATOR.md 里的文件副本、Git 忽略规则，都要逐一检查
