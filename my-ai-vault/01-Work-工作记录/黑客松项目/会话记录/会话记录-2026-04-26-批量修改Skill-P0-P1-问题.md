# 批量修改 Skill P0+P1 问题

## 背景

基于前期 Brainstorming 分析文档（`docs/superpowers/Skill-Brainstorming-分析.md`），用户要求按顺序修改 4 个 Skill 的 P0（高优先级）和 P1（中优先级）问题。本次会话共修改了 8 个 Skill 文件 + 8 个 bootstrap 文件，最后分析了 P2 问题并决定暂缓。

## 用户提示词序列

1. **"修改 4 个 P0 问题"** — 触发 P0 批量修改
2. **"按顺序修改 P1 问题"** — 触发 P1 按序修改（asset-extractor → session-recorder → pattern-miner → weekly-generator）
3. **"先列出中优先级问题"** — 要求展示 P1 问题列表
4. **"P2 的问题 列出来"** — 要求展示 P2 问题列表
5. **"暂时不改这几个"** — 确认暂缓 P2，并抛出三个合理质疑（配置中心可迁移性、反馈闭环必要性、token 成本）
6. **"保存会话"** / **"保存"** — 触发 session-recorder 保存本次会话

## 解决思路

### P0 修改（4 项）

| # | Skill | 问题 | 修复方案 |
|---|-------|------|---------|
| 1 | **asset-extractor** | JSON 解析不稳定 | LLM 直接输出 Markdown 草稿（YAML frontmatter + 正文），取消 JSON 解析环节，按 `---` 分隔拆分保存 |
| 2 | **session-recorder** | AI 提取字段易遗漏 | 新增 7 项字段提取 checklist，AI 必须逐项核对，缺失字段需明确标注原因 |
| 3 | **weekly-generator** | 时间基准不明确 | 明确规定筛选基准为 YAML Frontmatter 的 `date` 字段，**绝不使用文件修改时间** |
| 4 | **pattern-miner** | topic 关键词误聚类 | 关键词匹配 → **一次性 LLM 语义分组**（单次调用 O(1)），LLM 按"真正解决同一类问题"分组 |

### P1 修改（4 项）

| # | Skill | 问题 | 修复方案 |
|---|-------|------|---------|
| 1 | **asset-extractor** | LLM 调用成本高 | 步骤 2.1 新增预筛选层：正文 < 200 字或无代码块/步骤描述的原料直接跳过 |
| 2 | **session-recorder** | 模板字段无校验 | 步骤 5.1 新增 Frontmatter 必填校验：6 个字段逐项检查，缺失时自动填充默认值 |
| 3 | **pattern-miner** | 每次全量扫描 | 步骤 1 新增增量扫描：读取 `pattern_miner_last_scan` 时间戳，跳过已标记 `pattern_extracted: true` 的记录 |
| 4 | **weekly-generator** | 内容聚合无去重 | 步骤 3.1 新增智能去重：同一任务不同阶段合并为一条并标注进度演变 |

### P2 分析（暂缓）

用户提出三个合理质疑后，决定暂缓 P2：
- **S1 统一配置中心**：可能降低 Skill 可迁移性（折中方案：Skill 保留硬编码默认值，AGENTS.md 只做覆盖）
- **S3 反馈闭环**：目前产出量太少，追踪效果无意义，等积累 20+ 次使用后再考虑
- **Token 成本**：P2 整体增加 2~15% token 消耗，在产出量上来之前不急着做

## 优化经验

1. **PowerShell UTF-8 编码陷阱**：`Get-Content` 默认 ANSI/GBK，读取含中文的文件必须加 `-Encoding UTF8`，否则重写时内容会乱码
2. **StrReplaceFile 优于 WriteFile 做局部修改**：批量替换时精确匹配旧字符串，避免整文件重写带来的风险
3. **修改顺序：先 SKILL.md → 再 bootstrap**：确保两个源文件保持一致，避免遗漏同步
4. **P2 修改需要前置评估**：不能只看功能价值，还要评估 token 成本、可迁移性、维护复杂度
5. **语义分组比关键词匹配更准**："SQL 慢查询"和"SQL 索引优化"共享关键词"SQL"但问题类型完全不同，LLM 语义判断能正确分开
