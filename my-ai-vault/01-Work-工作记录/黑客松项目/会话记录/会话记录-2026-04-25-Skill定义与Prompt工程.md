# 会话记录 - Skill定义编写与Prompt工程

## 背景

需要为知识库系统编写 3 个核心 Skill：
1. 会话记录保存（Session Recorder）
2. 资产提炼（Asset Extractor）
3. 周报生成（Weekly Generator）

要求 Skill 能被不同 AI 工具（Kimi / Claude）统一理解和执行。

## Prompt

```
请帮我设计一个"会话记录保存"的 Skill 定义文档。

背景：
用户在与 AI 完成一次有价值的协作后，希望一键保存会话内容到知识库 Vault 中。

Skill 需要包含：
1. 触发条件（用户说什么话时激活）
2. 执行步骤（详细到每一步操作）
3. 输出格式（文件模板、YAML frontmatter 要求）
4. 核心规则（不可违反的约束）

额外要求：
- 首次使用询问归档路径，后续自动复用（记忆到 AGENTS.md）
- 支持交互式采集会话信息
- 文件名自动生成且去重
```

## 解决思路

**Skill 文档结构标准化**：

每个 SKILL.md 统一包含以下章节：
```
# Skill名称
版本 + 类型
触发条件
---
功能说明
---
执行步骤（1. 2. 3. ...）
---
核心规则（不可违反）
---
使用示例
---
版本信息
```

**Prompt 工程技巧**：

1. **system prompt 分层**：
   - 第一层：角色定义（"你是一位知识库管理员"）
   - 第二层：输入说明（"下面会提供一段工作记录"）
   - 第三层：提取目标（具体定义每类资产）
   - 第四层：输出格式（严格的 JSON schema）
   - 第五层：边界规则（空产出、质量标准）

2. **confidence 机制**：
   - high：有明确成功证据
   - medium：出现模式但缺少验证
   - low：推断可能有价值但未直接体现
   这让 LLM 对提取结果的质量有自我评估

3. **变量标注规范**：
   要求 Prompt 模板中用 `{{变量名}}` 标注可替换参数，便于后续复用

## 优化经验

1. **Skill 中的"执行步骤"越具体越好**。AI 执行时不会"脑补"，每一步都要明确告诉它做什么
2. **"核心规则"章节很重要**。用"不可违反"的措辞能显著提升 AI 的遵循率
3. **LLM 系统提示词中要求"不添加额外解释文字"**，可以显著减少 JSON 解析失败的概率
4. **name 字段要求"动词+宾语"格式**（如"分析慢查询"而非"SQL优化"），命名更规范

一个意外收获：在写 Asset Extractor 的 system prompt 时，要求 LLM "宁可少提取也不降低质量标准"后，提取出的资产质量明显更高，噪声大大减少。
