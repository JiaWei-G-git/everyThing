## 三、`<script setup>` 内部组织顺序

**必须严格按以下顺序排列代码块**，这是存量代码最一致的习惯：

```
1. import 语句（外部库 → 内部组件 → hooks → utils）
2. defineOptions({ name: 'xxx' })
3. defineProps({ ... })
4. 常量 / 配置数据（formList、formRules、formData → tableColumn、tableData）
5. ref / reactive 状态声明
6. computed
7. 工具函数（纯函数、辅助函数）
8. 业务操作函数（searchFunc、resetFunc、getTableData 等）
9. watch
10. defineEmits(['eventName'])  ← 注意：emit 声明放在最后
11. 生命周期钩子（onBeforeMount / onMounted）
```

> **AI 指令**：配置数据按页面展示顺序排列——先表单（`formList`、`formRules`、`formData`），后表格（`tableColumn`、`tableData`）。同一组件内多个区块时，按从上到下顺序排列。

| 规则 | 级别 | 置信度 | 备注 |
|------|------|--------|------|
| emit 声明在生命周期之后 | 【观察】 | MEDIUM | `~ElectMeterAcceptance.vue` 中 emit 在函数后定义，非顶部 |
| import 分组：外部 → 内部 | 【强制】 | HIGH | `vue` → 组件库 → `@/hooks/*` → `@/utils/*` → 相对路径 |
| hooks 解构紧跟 import | 【强制】 | HIGH | `const { pager, pagerReset } = useTablePagerHook()` |
| 常量声明在状态之前 | 【强制】 | HIGH | `const dictFieldMap = { ... }` 在 `ref` 之前 |

## 五、响应式状态管理

### 5.1 ref vs reactive 决策表

| 场景 | 使用 | 级别 | 置信度 |
|------|------|------|--------|
| 基本类型（string, number, boolean, array） | `ref()` | 【强制】 | HIGH |
| 对象类型（formData, pager） | `ref()` | 【强制】 | HIGH |
| 导出配置对象 | `reactive()` | 【观察】 | MEDIUM | 历史代码中 `exportOption` 用 `reactive` |

### 5.2 状态声明模板

```vue
<script setup>
// 表单状态
const formData = ref({
  supplyOrgNo: '',
  businessType: '1',
})

// 表格状态
const tableData = ref([])
const tableLoading = ref(false)

// 分页状态（来自 hook）
const { pager, pagerReset, changeSize, changeCurrent } = useTablePagerHook()

// 导出配置（唯一 reactive 例外）
let exportOption = reactive({
  url: '/member/V1/xxx/export',
  filter: {},
})
</script>
```

> **AI 指令**：除非明确需要重新赋值整个引用，否则 `ref` 声明统一使用 `const`。

### 5.4 formData 默认值设置

**默认值必须在 `ref({...})` 初始化时写入**，禁止在声明后再逐个字段赋值。

```vue
<script setup>
// 正确：默认值在 ref 初始化时设置
const formData = ref({
  supplyOrgNo: $macro.$store.getters.account.roleOrgNo,
  dataTime: [getPreMonthDateByN(30), getPreMonthDateByN(0)],
  userCategory: [],
})

// 错误：声明后再逐个赋值
const formData = ref({})
formData.value.supplyOrgNo = $macro.$store.getters.account.roleOrgNo
formData.value.dataTime = [getPreMonthDateByN(30), getPreMonthDateByN(0)]
</script>
```

| 规则 | 级别 | 置信度 |
|------|------|--------|
| `formData` 默认值在 `ref({...})` 中设置 | 【强制】 | HIGH |

### 5.3 const vs let 决策（矛盾点解决方案）

| 情况 | 选择 | 原因 |
|------|------|------|
| 默认 | `const` | 与 Vue 的响应式系统配合，`const ref` 可修改 `.value` |
| 需要重新赋值引用本身 | `let` | 极少数场景（如 exportOption 用 reactive 时可能 let） |

> **AI 指令**：遇到 `ref` 声明时，优先使用 `const`。只有明确需要 `xxx = newValue` 重新赋值整个引用时才用 `let`。

## 十、矛盾决策矩阵（AI 必须遵循）

当以下矛盾同时出现时，**按右侧优先级执行**：

| 矛盾项 | 存量分布 | AI 决策 |
|--------|----------|---------|
| `const` vs `let` 声明 ref | 各占 50% | **优先 `const`**，除非需重新赋值引用 |
| Template 双引号 vs 单引号 | 双引号多 | **统一双引号** |
| 行尾分号 vs 无分号 | 无分号多 | **无分号** |
| `::v-deep` vs `/deep/` | 混用 | **新代码用 `::v-deep`** |
| emit 顶部声明 vs 底部声明 | 底部多 | **放底部（函数之后）** |
| `ref` vs `reactive` | ref 占 95%+ | **统一 `ref`**，仅 exportOption 例外 |

## 十一、反模式清单（AI 禁止生成）

| 反模式 | 原因 | 正确做法 |
|--------|------|----------|
| `lang="ts"` | 纯 JS 项目 | 不写 `lang` 属性 |
| `reactive()` 声明表单/表格状态 | 与团队习惯冲突 | 用 `ref()` |
| `defineEmits` 放在 import 之后 | 与存量顺序冲突 | 放在生命周期之后 |
| `??` 空值合并 | 存量用 `\|\|` | 用 `\|\|` |
| `async` 箭头函数直接做事件处理器 | 不符合习惯 | 单独声明函数 |
| `scoped` 用于大屏页 | 需要覆盖子组件 | 大屏页不加 `scoped` |
| 独立 `constants.ts` | 常量就近声明 | 常量放组件顶部 |
| URL query 同步筛选条件 | 与存量设计冲突 | 通过 emit + props 通信 |
| `useFormConfig` hook | 项目中仅 commModMon 使用，其他页面均直接定义 | 直接声明 `formList` / `formRules` / `formData` |
| `formData.value.xxx = ...` 逐个设默认值 | 与规范默认值初始化方式冲突 | 在 `ref({...})` 中统一设置 |

## 十三、快速复制模板
